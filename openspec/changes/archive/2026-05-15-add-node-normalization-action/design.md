## Context

Нужно встроить в существующую страницу узла (Action Bar) управляемый запуск нормализации конкретного markdown-узла с использованием Cursor Agent. Система локальная и git-first, поэтому после успешной нормализации изменения должны попадать в стандартный sync-поток, а не оставаться только на файловой системе без синхронизации.

Функция затрагивает несколько модулей: web UI, REST API, runtime-конфигурацию и контейнеризацию. Также есть внешняя зависимость: установленный Cursor Agent.

## Goals / Non-Goals

**Goals:**
- Добавить в UI явное действие «Нормализация» для текущего узла.
- Обеспечить серверный запуск Cursor Agent с безопасно сформированным промтом по выбранному узлу.
- Сделать выполнение наблюдаемым для UI (running/success/error).
- После успеха автоматически выполнить `sync`.
- Сделать поведение воспроизводимым в Docker через предустановленный Cursor Agent.

**Non-Goals:**
- Массовая нормализация всей базы знаний.
- Полноценный job-оркестратор с очередями и распределённым исполнением.
- Изменение логики нормализации вне конкретного узла (например, глобальные авто-рефакторинги базы).

## Decisions

1. Серверная оркестрация нормализации
- Решение: запуск Cursor Agent инициируется через REST API и исполняется на сервере с привязкой к одному узлу.
- Почему: позволяет централизованно контролировать окружение и последующий `sync`.
- Альтернатива: запускать Agent напрямую из браузера. Отклонено из-за отсутствия контроля над post-step sync.

2. Асинхронная модель выполнения
- Решение: запуск нормализации асинхронный (start + polling статуса) с отражением состояния в UI.
- Почему: нормализация может занимать время и включать внешние вызовы.
- Альтернатива: синхронный HTTP-запрос. Отклонено из-за риска таймаутов и плохого UX.

3. Триггер sync только после успешной нормализации
- Решение: `sync` выполняется только при статусе success у операции нормализации.
- Почему: не нужно синхронизировать частичные/ошибочные изменения.
- Альтернатива: всегда выполнять sync. Отклонено, так как это может зафиксировать нежелательные промежуточные правки.

4. Runtime-зависимость через Docker
- Решение: добавить установку Cursor Agent в Dockerfile.
- Почему: self-hosted сценарий должен быть воспроизводим без ручной донастройки контейнера.
- Альтернатива: документировать ручную установку. Отклонено как ненадёжно для деплоя.

## Risks / Trade-offs

- [Риск] Cursor Agent недоступен в runtime (не установлен/не найден в PATH) → Митигация: явная ошибка конфигурации и диагностируемый ответ API.
- [Риск] Конкурентные запуски нормализации одного узла → Митигация: блокировка повторного старта для того же path до завершения активной задачи.
- [Риск] `sync` завершится ошибкой после успешной нормализации → Митигация: отдельный статус шага sync и возврат ошибки с сохранением факта, что нормализация уже применилась локально.
- [Trade-off] Асинхронная модель сложнее синхронной → получает устойчивый UX и предсказуемость для длительных операций.
