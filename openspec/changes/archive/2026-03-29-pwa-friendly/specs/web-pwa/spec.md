## Purpose

Спецификация описывает поведение веб-интерфейса knowledge-db как **Progressive Web App**: наличие манифеста, регистрация service worker, кэширование статики фронтенда и ограничения на кэширование API. Данные базы знаний по-прежнему доступны только через работающий kb-server; PWA улучшает установку и повторный запуск оболочки UI.

### Requirement: Web App Manifest

Система ДОЛЖНА (SHALL) поставлять вместе с production-сборкой веб-UI **Web App Manifest**, пригодный для установки приложения в поддерживаемых браузерах.

#### Scenario: Манифест доступен после сборки

- **WHEN** выполнена production-сборка фронтенда (`web`)
- **THEN** в артефактах сборки MUST присутствовать манифест (JSON) с полями, достаточными для установки: как минимум `name` или `short_name`, `start_url`, `display`, иконки не менее двух размеров (в т.ч. подходящие для маски/лайнчера)

#### Scenario: Согласованность start_url с SPA

- **WHEN** пользователь устанавливает PWA и открывает приложение с домашнего экрана
- **THEN** `start_url` MUST указывать на корень приложения (или согласованный путь), совместимый с клиентским роутингом React, без поломки редиректов авторизации

### Requirement: Service Worker и precache статики

Система ДОЛЖНА (SHALL) регистрировать **Service Worker** в production-режиме, обеспечивающий предкэширование (precache) статических ресурсов собранного UI (HTML, JS, CSS, ассеты сборки).

#### Scenario: Регистрация только в production

- **WHEN** приложение запущено в режиме production (собранный бандл)
- **THEN** браузер MUST иметь возможность зарегистрировать Service Worker для этого origin согласно политике безопасности (HTTPS или localhost)

#### Scenario: Повторный визит без сети для оболочки

- **WHEN** пользователь уже открывал UI при работающем сервере, Service Worker активирован, затем сеть для статики недоступна
- **THEN** клиент MUST иметь возможность отобразить ранее закэшированную статическую оболочку приложения (в пределах возможностей precache), не гарантируя доступ к данным API

### Requirement: Политика кэширования для API

Система ДОЛЖНА (SHALL) настраивать Service Worker так, чтобы запросы к **REST API** (`/api` или эквивалентный префикс приложения) не подменялись долгоживущим кэшем «как статика», искажающим актуальность ответов или сессию.

#### Scenario: Запросы к API не обслуживаются из precache JSON

- **WHEN** клиент выполняет `fetch` к эндпоинтам под `/api/*` (включая сессию и CRUD)
- **THEN** стратегия SW MUST NOT отдавать ответы API исключительно из precache сборки без сети так, как для файлов фронтенда

### Requirement: Совместимость со встраиванием в kb-server

Система ДОЛЖНА (SHALL) сохранять работоспособность раздачи статики через встроенный в бинарник kb-server сервер: манифест, service worker и связанные workbox-файлы MUST быть доступны по тем же правилам, что и остальная статика SPA (корректные пути, отсутствие 404 для обязательных entrypoints PWA после деплоя).

#### Scenario: Сборка для embed

- **WHEN** выполнены сборка `web` и сборка kb-server со встроенной статикой
- **THEN** запрос к корню UI и к путям PWA (manifest, SW) MUST завершаться успешно так же, как для текущей SPA без PWA
