## Context

Веб-UI knowledge-db собирается Vite (`web/`), статика встраивается в бинарник kb-server. Авторизация — сессионная cookie (`credentials: 'include'`), API на том же origin при относительных путях или на отдельном порту при `VITE_API_URL`. Сейчас нет Web App Manifest и нет Service Worker — установка «как приложение» и фоновый кэш статики недоступны.

## Goals / Non-Goals

**Goals:**

- Производственная сборка `web/` SHALL содержать валидный **Web App Manifest** и регистрируемый **Service Worker**, чтобы поддерживались установка PWA (где это позволяет браузер/OS) и **precache** фронтенд-статики.
- Запросы к **`/api/*`** SHALL не обслуживаться из долгоживущего кэша SW как статические JSON (сеть или короткий/отсутствующий кэш для API).
- Сборка и отдача через kb-server SHALL оставаться рабочими (корректные пути к `manifest`, `sw`, workbox-чанкам при `base` Vite по умолчанию).

**Non-Goals:**

- Оффлайн-репликация данных базы знаний в IndexedDB или полноценная работа UI без kb-server.
- Push-уведомления и фоновая синхронизация данных.
- Изменение модели аутентификации (куки, эндпоинты `/api/auth/*`).

## Decisions

1. **Плагин `vite-plugin-pwa` (Workbox)**  
   **Решение:** использовать официально распространённый плагин для Vite с генерацией precache по манифесту сборки.  
   **Альтернативы:** ручной SW без Workbox (больше кода и ошибок); только manifest без SW (нет precache, слабее PWA).  
   **Почему:** баланс между поддержкой и контролем стратегий кэша.

2. **Стратегия для API**  
   **Решение:** явно исключить `/api` из aggressive caching или использовать **NetworkOnly** / сеть по умолчанию для префикса API.  
   **Почему:** избежать устаревших ответов и путаницы с 401/сессией.

3. **Режим разработки**  
   **Решение:** по умолчанию **отключать** или минимизировать SW в `vite dev`, чтобы не мешать HMR (через опции плагина или `devOptions`).  
   **Почему:** типичная боль при PWA.

4. **Иконки**  
   **Решение:** добавить набор **закоммиченных** PNG в `public/` (192/512 и при необходимости maskable); при необходимости вручную экспортировать из `favicon.svg`. Генерация иконок из SVG в CI (`@vite-pwa/assets-generator` и аналоги) **не используется** на данном этапе.

5. **Обновление SW**  
   **Решение:** `registerType: 'autoUpdate'` или prompt по умолчанию проекта; после деплоя новой версии пользователь получает обновлённый precache.  
   **Почему:** предсказуемое поведение при выкладках.

6. **Цвета manifest и splash**  
   **Решение:** `theme_color` и `background_color` в манифесте задавать **под светлую** тему UI (как в дефолтном светлом режиме приложения). Отдельный набор под тёмную тему или динамическая смена в manifest **не делаем** на данном этапе.  
   **Почему:** проще реализация; пользователь в тёмной теме внутри приложения по-прежнему видит тёмный интерфейс — расхождение возможно только у системных полос/splash ОС.

## Risks / Trade-offs

- [Устаревший кэш статики после релиза] → Mitigation: автообновление SW + корректные `revision` в precache (даёт Workbox).  
- [SW усложняет отладку локально] → Mitigation: отключение в dev, документирование `skipWaiting` при необходимости.  
- [Разные origin для UI и API] → Mitigation: не кэшировать API; для кук при кросс-origin уже действуют правила CORS/SameSite — вне scope данного change, но не ломать существующий сценарий same-origin при embed.

## Migration Plan

1. Добавить зависимости и конфигурацию PWA в `web/`.  
2. Выполнить `npm run build`, проверить `dist/` (manifest, sw).  
3. Собрать kb-server, убедиться, что новые файлы отдаются с теми же правилами, что и остальная статика.  
4. Откат: удалить плагин и сгенерированные ссылки из `index.html`/конфига, пересобрать — поведение как до change.
