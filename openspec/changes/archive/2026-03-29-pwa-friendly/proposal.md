## Why

Пользователи kb-server часто открывают веб-UI с одной машины (localhost или локальная сеть) и хотели бы закрепить интерфейс как «приложение» (иконка, отдельное окно, быстрый повторный запуск без лишних шагов). Сейчас UI — обычный SPA без Web App Manifest и без Service Worker, поэтому браузер не предлагает установку и не кэширует статику на уровне PWA. Настоящая офлайн-работа с данными по-прежнему требует запущенного API; цель — сделать **оболочку** PWA-friendly без подмены модели хранения данных.

## What Changes

- Добавить **Web App Manifest** (имя, иконки, `start_url`, `display`, цвета темы) для веб-фронтенда в `web/`.
- Подключить генерацию **Service Worker** (например через `vite-plugin-pwa` / Workbox) с **precache** собранной статики и явной политикой для **`/api`** (не отдавать JSON из долгого кэша как статику).
- Обеспечить корректные пути при **встраивании статики в kb-server** (тот же базовый URL, что и сейчас для SPA).
- Добавить **иконки** нужных размеров (или генерацию из исходника) и при необходимости мета-теги для iOS/Chrome.
- Документировать ограничение: **сессия и данные** по-прежнему через API; PWA не заменяет kb-server.

## Capabilities

### New Capabilities

- `web-pwa`: установка веб-UI как PWA (manifest + service worker), кэширование статики фронтенда, политика сети для API, совместимость со сборкой Vite и отдачей статики из Go.

### Modified Capabilities

- (нет) — поведение экранов и API не меняется на уровне существующих спеков; добавляется отдельная способность «оболочка PWA».

## Impact

- `web/`: `package.json`, `vite.config.ts`, `index.html`, новые/сгенерированные ассеты (иконки, при необходимости `public/`).
- `internal/ui` или путь embed статики: убедиться, что manifest и SW попадают в артефакты сборки и отдаются с корректными MIME/путями.
- Новая зависимость dev/build: плагин PWA для Vite (или эквивалент).
- Тесты: при наличии — `npm run build` / smoke; при необходимости — проверка отсутствия регрессии маршрутизации SPA.
