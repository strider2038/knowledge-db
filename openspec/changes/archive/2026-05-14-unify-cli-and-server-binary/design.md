## Context

Сейчас проект использует два отдельных бинарника: `kb-server` для запуска сервера и `kb-cli` для технических команд. В сценариях self-hosted и особенно при использовании Cursor Agent это приводит к дублированию шагов сборки/доставки и усложняет runtime-окружение контейнера. Решение должно упростить эксплуатацию, не изменяя доменное поведение API, ingestion и работы с базой знаний.

## Goals / Non-Goals

**Goals:**

- Ввести единый бинарник `kb` как единственную точку входа.
- Поддержать запуск серверной части через `kb serve`.
- Сохранить функциональность существующих CLI-команд внутри единого командного дерева.
- Упростить Docker/CI и документацию за счёт единого интерфейса запуска.

**Non-Goals:**

- Сохранение обратной совместимости со старыми бинарниками `kb-server` и `kb-cli`.
- Редизайн внутренних пакетов `internal/*` без необходимости.
- Изменение контрактов REST API, MCP, Telegram bot или формата данных базы.

## Decisions

1. Единый root command на базе `urfave/cli` с именем `kb`.
- Почему: `urfave/cli` проще по surface area для небольшого набора команд, снижает связность между командами и делает код entrypoint более прямолинейным для поддержки агентами.
- Альтернатива: оставить `cobra`. Отклонено, потому что добавляется избыточная инфраструктура для текущего масштаба CLI и ухудшается читаемость bootstrap-кода команд.

2. Серверный запуск становится подкомандой `serve`.
- Почему: явное разделение режимов работы (`kb serve` vs `kb validate`) и предсказуемый UX.
- Альтернатива: запускать сервер по умолчанию без подкоманды. Отклонено, потому что ухудшает discoverability CLI-функций и делает поведение менее очевидным.

3. Миграция без compatibility-layer.
- Почему: это снижает объём временного кода и технический долг, соответствует целевому упрощению.
- Альтернатива: алиасы/shim-бинарники. Отклонено по требованию change scope.

4. Обновление всех операционных точек входа в рамках change.
- Почему: при breaking-change критично синхронно обновить Dockerfile, task-цели и docs, чтобы не оставить нерабочие инструкции.
- Альтернатива: частичное обновление с последующим догоном. Отклонено из-за повышенного риска регрессий при эксплуатации.

## Risks / Trade-offs

- [Риск] Поломка существующих скриптов/контейнеров, где зашиты `kb-server` или `kb-cli`.
  → Mitigation: атомарно обновить Docker/CI/docs и явно отметить breaking-change в changelog/коммите.

- [Риск] Регрессия в инициализации сервера и флагов из-за одновременного переноса в `urfave/cli`.
  → Mitigation: добавить/обновить smoke-тест запуска `kb serve` и проверить существующие server integration tests.

- [Trade-off] Более крупный бинарник и единая поверхность команд.
  → Mitigation: это приемлемо для local/self-hosted модели; выигрыши в DX и эксплуатации перевешивают.

- [Trade-off] Стоимость миграции между CLI-фреймворками.
  → Mitigation: мигрировать поэтапно по командам, добавить проверку help/flags и удалить `cobra` только после прохождения тестов.
