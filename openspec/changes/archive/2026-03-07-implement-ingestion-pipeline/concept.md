# Концепция: Ingestion Pipeline

Результаты explore-сессии по проектированию наполнения базы знаний.

## Виды материалов

Три основных типа контента:

| Тип | Описание | Объём | Частота |
|-----|----------|-------|---------|
| **article** | Копии статей с Habr и других источников | Большой (полный текст) | Редко |
| **link** | Ссылки на крупные сервисы/ресурсы с аннотацией | Маленький (URL + описание) | Средне |
| **note** | Личные заметки (от строчки до эссе) | Разный | Часто |

Тип хранится как поле `type` во frontmatter, а не как отдельная папка. Все ноды располагаются в тематическом дереве — тип это атрибут, а не место хранения.

## Каналы добавления

**Основной канал — Telegram.** Пользователь пересылает боту сообщения с инструкциями. Второй по важности — браузер (через API/Web UI).

Сценарии использования Telegram:

1. Переслать ссылку → скачать статью, сохранить копию + metadata
2. Переслать ссылку + инструкция ("сохрани в go/concurrency") → скачать + положить куда указано
3. Переслать сообщение из канала (текст) → сохранить как заметку
4. Написать текст-заметку → личная заметка
5. Переслать ссылку на ресурс/сервис с комментарием → закладка с аннотацией

## Автоматизация через LLM

Максимальная автоматизация. LLM определяет:
- **keywords** — ключевые слова для поиска
- **annotation** — краткое описание контента
- **theme/subtheme** — куда положить в дерево тем
- **type** — article/link/note
- **source_date** — дата оригинала (если статья)

Пользователь правит постфактум, если нужно. Бот отвечает подтверждением (куда положил, какие keywords), но без интерактивного цикла правок — это отдельная задача на будущее.

## Расширение Frontmatter

Текущий формат:
```yaml
---
keywords: [example]
created: "2024-01-01T00:00:00Z"
updated: "2024-01-01T00:00:00Z"
annotation: "..."
---
```

Расширенный формат:
```yaml
---
keywords: [goroutines, memory-leak, profiling]
created: "2026-03-06T12:00:00Z"     # когда добавлено в KB
updated: "2026-03-06T12:00:00Z"     # когда правилось в KB
annotation: "Статья о типичных утечках горутин и способах их обнаружения"
type: article                        # article | link | note
source_url: "https://habr.com/..."   # откуда взято (опционально)
source_date: "2026-02-20"            # дата оригинала (опционально)
---
```

Важно хранить обе даты: дату добавления в базу (`created`) и дату создания оригинала (`source_date`).

## Парсинг URL: стратегия

### Основной: Jina Reader API

```
URL → GET https://r.jina.ai/{URL} → готовый Markdown
```

- Один HTTP-запрос, возвращает markdown
- Отличное качество извлечения
- Обрабатывает JS-rendered страницы
- Возвращает title, date, author
- Бесплатно с лимитами

### Fallback: Go-native (go-readability + html-to-markdown)

```
URL → HTTP GET → HTML → go-readability → чистый HTML → html-to-markdown → Markdown
```

- Всё внутри Go-бинарника, zero dependencies
- Качество ниже, но достаточно для Habr и типичных блогов
- Не справляется с JS-rendered страницами

### Стратегия:

```
URL ──► Jina Reader API
          │
          ├── ✓ → markdown + metadata
          │
          └── ✗ (таймаут / rate limit / недоступен)
               │
               ▼
        Go-native (readability + html-to-md)
```

Оффлайн-first — принцип для чтения базы. Добавление по URL по определению требует интернета, поэтому зависимость от Jina при добавлении не нарушает принцип.

### Альтернативы (сохранены для справки)

- **Trafilatura (Python)** — топ в бенчмарках по качеству извлечения. Требует Python-рантайм, subprocess-вызовы из Go. Рассмотреть, если качество go-readability окажется недостаточным.
- **Headless browser (rod/chromedp)** — для JS-rendered SPA. Overkill для текущих источников (Habr, блоги — серверный рендеринг).
- **LLM-extraction** — дорого (HTML = много токенов), неидемпотентно. Не подходит для массовой обработки.

## Рекаталогизация (будущее)

Периодическая задача на агента (Cursor skill или CLI):
- Ревизия keywords — актуальность, дубли, точность
- Placement — правильная тема/подтема, выделение разросшихся тем
- Связи между записями (territory knowledge graph)
- Поиск дубликатов для объединения

Это отдельная задача, не входит в текущий change.

## Решения, принятые в ходе обсуждения

1. **Inbox не нужен** — LLM сразу определяет тему, пользователь правит постфактум
2. **Тип — атрибут, не структура** — все ноды в тематическом дереве, `type` во frontmatter
3. **Ответ бота — простой** — подтверждение без интерактивных правок (усложнение — отдельная задача)
4. **Картинки из статей** — вопрос открыт, пока не решён
5. **Ссылки** — на крупные сервисы (документация, туториалы), хранятся с аннотацией для поиска
