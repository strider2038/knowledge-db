# Design: Обзор базы знаний — технические решения

## Context

Текущая страница «Поиск» (SearchPage) использует `GET /api/nodes?path=X`, который возвращает только узлы в одной папке. `TreeNode` содержит только `name` и `path`. Дерево тем строится через `GET /api/tree`. Переводы (article.ru.md) отображаются как отдельные узлы. Нет фильтрации по типу, поиска, сортировки и пагинации.

Proposal требует: рекурсивный список поддерева, фильтры по типам (article/link/note), схлопывание переводов, сортируемую таблицу с метаданными, поиск по тексту, breadcrumbs, «Вся база», состояние в URL, пагинацию.

## Goals / Non-Goals

**Goals:**
- Единый endpoint `GET /api/nodes` с параметрами path, recursive, q, type; расширенный ответ с метаданными
- Страница «Обзор» с деревом-фильтром, toggle по типам, таблицей, поиском, пагинацией
- Схлопывание переводов: в списке только оригиналы, переводы в поле `translations`
- Состояние (path, type, q, sort, page) в URL
- Breadcrumbs, вариант «Вся база»

**Non-Goals:**
- Компактный/развёрнутый режим таблицы
- Экспорт в CSV/JSON
- Полнотекстовый поиск по content (только title, keywords, annotation)

## Decisions

### 1. API: расширение GET /api/nodes

**Решение:** Добавить query-параметры к существующему `GET /api/nodes`:

| Параметр | Тип | Описание |
|----------|-----|----------|
| `path` | string | Путь темы (как сейчас). Пустой или отсутствует = вся база |
| `recursive` | bool | true = узлы всего поддерева, false = только в path (по умолчанию false для обратной совместимости) |
| `q` | string | Подстрока для поиска в title, keywords, annotation (case-insensitive) |
| `type` | string | Фильтр по типу на бэкенде: `article`, `link`, `note`. Несколько через запятую: `type=article,link` |
| `limit` | int | Лимит строк (пагинация), по умолчанию 50 |
| `offset` | int | Смещение (пагинация), по умолчанию 0 |

**Альтернатива:** Отдельный endpoint `GET /api/nodes/browse?path=...` — отклонена: один endpoint проще, обратная совместимость через `recursive=false`.

### 2. Формат ответа: NodeListItem

**Решение:** Новый тип ответа для списка (отличается от полного Node):

```json
{
  "nodes": [
    {
      "path": "programming/scaling/load-balancing",
      "title": "Load balancing",
      "type": "article",
      "created": "2026-03-08T13:02:48Z",
      "source_url": "https://...",
      "translations": ["load-balancing.ru"]
    }
  ],
  "total": 42
}
```

Поле `total` — общее количество до пагинации (для UI пагинации). `title` из metadata, fallback на slug из path.

**Альтернатива:** Возвращать полный Node — избыточно (content, annotation не нужны в таблице).

### 3. Схлопывание переводов

**Решение:** При обходе файлов считать переводом файл с общим паттерном `*.[a-z]{2}.md` (двухбуквенный код языка: .ru, .en и т.д.). Такие файлы не включать в список как отдельные узлы. Для оригинала (slug.md) читать frontmatter и извлекать `translations` — массив slug'ов переводов (например `["article.ru"]`).

**Альтернатива:** Только `.ru.md` — отклонена: общий паттерн обеспечивает расширяемость.

### 4. Фильтрация дерева по типам

**Решение:** Фильтрация на фронтенде. Бэкенд возвращает полное дерево (GET /api/tree) и список узлов с фильтрами. Фронт: (1) получает узлы с `type`; (2) строит множество path'ов узлов, прошедших фильтр; (3) рекурсивно скрывает ветки дерева, у которых нет ни одного узла в этом множестве.

**Альтернатива:** Бэкенд возвращает отфильтрованное дерево — требует нового endpoint или изменения GetTree, усложняет API.

### 5. Пагинация

**Решение:** offset/limit на бэкенде. Сортировка выполняется до пагинации. Поля `limit` (default 50, max 200) и `offset` в query.

**Альтернатива:** Cursor-based — избыточно для табличного списка без real-time обновлений.

### 6. Состояние в URL

**Решение:** Query-параметры: `?path=programming&type=article,link&q=go&sort=title&order=asc&page=2`. `page` маппится на offset (page*limit). При загрузке страницы — чтение из `useSearchParams`, при изменении фильтров — `navigate` с обновлённым URL.

**Альтернатива:** localStorage — не shareable, хуже для закладок.

### 7. Выбор языка на NodePage

**Решение:** Если у узла есть `translations`, показывать переключатель (tabs или dropdown): «Оригинал» | «Русский» | … Переход по `/node/{path}` — оригинал, `/node/{path.ru}` — перевод. Роут уже поддерживает path с точкой.

**Альтернатива:** Отдельный query `?lang=ru` — дублирует информацию в path, усложняет кэширование.

### 8. Сортировка

**Решение:** Сортировка на бэкенде. Параметры `sort` (title, type, created, source_url) и `order` (asc, desc). Default: `sort=title`, `order=asc`.

**Альтернатива:** Сортировка на фронте — при пагинации не работает (данные неполные).

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| Рекурсивный обход + чтение frontmatter для каждого файла — медленно на больших базах | Ограничить limit (max 200), при необходимости добавить кэш или индексацию позже |
| Поиск `q` — линейный перебор всех узлов поддерева | Приемлемо для персональной базы (сотни/тысячи узлов); при росте — полнотекстовый индекс |
| Дерево с фильтрацией на фронте — лишние данные | Дерево мало (~десятки веток), узлов может быть много — пагинация снижает нагрузку |
| Обратная совместимость ListNodes(path) без recursive | Сохраняем: при отсутствии `recursive` или `recursive=false` поведение как сейчас (только узлы в path) |

## Migration Plan

1. Backend: расширить internal/kb (ListNodesRecursive или опции в ListNodes), internal/api (query params)
2. Frontend: рефакторинг SearchPage → OverviewPage, новые компоненты
3. Navbar: переименование «Поиск» → «Обзор»
4. NodePage: добавление выбора языка
5. API-тесты для GET /api/nodes с новыми параметрами

Роллбек: revert коммитов; старый клиент продолжит работать (recursive=false по умолчанию).
