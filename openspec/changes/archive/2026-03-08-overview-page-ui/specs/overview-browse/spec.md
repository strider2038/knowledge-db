## Purpose

Страница «Обзор» — основной интерфейс для просмотра и навигации по базе знаний. Дерево тем служит фильтром, таблица показывает узлы поддерева с метаданными, фильтрами по типам и поиском по тексту.

## Requirements


### Requirement: Дерево и таблица

UI ДОЛЖЕН (SHALL) отображать слева иерархическое дерево тем, справа — сортируемую таблицу узлов выбранного поддерева. Дерево служит фильтром по path.

#### Сценарий: Выбор темы

- **WHEN** пользователь выбирает тему в дереве
- **THEN** таблица справа показывает узлы всего поддерева (рекурсивно)

#### Сценарий: Вся база

- **WHEN** пользователь выбирает вариант «Вся база» в дереве
- **THEN** таблица показывает узлы из всей базы

#### Сценарий: Визуал дерева

- **WHEN** отображается дерево тем
- **THEN** «Вся база» явно выделена (сдвинута влево), подузлы имеют серые вертикальные черточки (border-l), узлы отсортированы по алфавиту

### Requirement: Фильтры по типам

UI ДОЛЖЕН (SHALL) предоставлять toggle-кнопки для выбора типов узлов (article, link, note). Таблица и дерево MUST фильтроваться по выбранным типам. При выборе конкретного узла дерева не должно быть дополнительной фильтрации соседних веток — дерево фильтруется только по типу (по всей базе), не по текущему path.

#### Сценарий: Включение типа

- **WHEN** пользователь включает toggle «статья»
- **THEN** в таблице отображаются только узлы типа article, дерево скрывает ветки без статей

#### Сценарий: Выбор узла дерева

- **WHEN** пользователь выбирает тему в дереве при включённом фильтре по типу
- **THEN** таблица показывает узлы выбранного поддерева; дерево не скрывает соседние ветки (фильтрация дерева — только по типу)

### Requirement: Сортировка по умолчанию

По умолчанию таблица MUST сортироваться по названию (title) по возрастанию.

### Requirement: Таблица

Таблица MUST содержать колонки: Название (из title), Тип (цветом), Дата, Ссылка (для link/article — кликабельная, target="_blank"). Сортировка по клику на заголовок. Кнопки фильтров по типам MUST использовать те же цвета, что и badge типа в таблице (article=синий, link=зелёный, note=серый).

#### Сценарий: Сортировка

- **WHEN** пользователь кликает на заголовок колонки
- **THEN** данные сортируются по этой колонке; направление (asc/desc) переключается

#### Сценарий: Ссылка

- **WHEN** узел типа link или article имеет source_url
- **THEN** в колонке отображается кликабельная ссылка, открывающаяся в новой вкладке

### Requirement: Схлопывание переводов

В таблице MUST отображаться только оригиналы узлов; переводы (article.ru.md) не показываются отдельно. При наличии переводов — в поле translations или на странице узла выбор языка.

#### Сценарий: Один ряд на оригинал

- **WHEN** существуют article.md и article.ru.md
- **THEN** в таблице одна строка для article (оригинал)

### Requirement: Поиск

UI ДОЛЖЕН (SHALL) содержать инпут поиска вверху. При вводе подстроки выполняется серверная фильтрация по title, keywords, annotation.

#### Сценарий: Поиск по тексту

- **WHEN** пользователь вводит текст в инпут
- **THEN** таблица обновляется и показывает только узлы, где подстрока входит в title, keywords или annotation

### Requirement: Выделение текущей темы

При выборе темы в дереве MUST явно подсвечиваться текущий пункт (цветом/фоном). Breadcrumbs не обязательны — достаточно визуального выделения выбранной темы.

### Requirement: Состояние в URL

Фильтры (path, type, q, sort, order, page) MUST сохраняться в query-параметрах URL. При загрузке страницы с URL состояние восстанавливается.

#### Сценарий: Восстановление

- **WHEN** пользователь открывает страницу с ?path=ai&type=article&q=go
- **THEN** отображается поддерево ai, фильтр article, поиск «go»

### Requirement: Пагинация

Таблица MUST поддерживать пагинацию. Страница отображается в URL (page).

#### Сценарий: Переключение страницы

- **WHEN** пользователь выбирает страницу 2
- **THEN** URL обновляется, отображаются следующие записи
