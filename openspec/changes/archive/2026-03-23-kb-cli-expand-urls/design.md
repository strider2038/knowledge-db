## Context

В сервере и ingestion уже используется `internal/pkg/urlutil.NormalizeURL`: HTTP HEAD с следованием редиректам, снятие `utm_*`, исключения для `t.me` / `telegram.org` (не раскрывать как «целевой ресурс»). Пользователи хотят запускать ту же нормализацию из `kb-cli` по готовым markdown-файлам, без сервера.

## Goals / Non-Goals

**Goals:**

- Подкоманда `expand-urls` с UX, согласованным с `dump-images`: `--path`, `--file`, `--dry-run`.
- Переиспользование `urlutil.NormalizeURL` для единообразия с ingestion/API/Telegram.
- Замена URL в markdown там, где они явно выражены как ссылки `(...)` и при необходимости в frontmatter YAML (поля со значением `http(s)://...`).
- Идемпотентность в смысле: повторный запуск не должен ломать файлы; если сеть недоступна, оставлять исходный URL (как `NormalizeURL`).

**Non-Goals:**

- Раскрытие «умных» ссылок без HTTP-схемы или нестандартных схем.
- Гарантированная обработка каждого возможного варианта markdown (таблицы, HTML внутри md) — достаточно предсказуемого покрытия + документированные ограничения.
- Изменение поведения `t.me`/`telegram.org` без согласования (оставляем как в `urlutil`).

## Decisions

1. **Имя команды: `expand-urls`** — понятно пользователю; альтернатива `normalize-urls` ближе к коду, но «expand» отражает сокращения.

2. **Логика нормализации: только `NormalizeURL`** — один источник правды; при необходимости расширить снятие трекинг-параметров (например `fbclid`) в `urlutil` один раз для CLI и сервера.

3. **Область замены в файле:**
   - Markdown-ссылки `[...](url)` и автоссылки `<url>`.
   - Строки в frontmatter, где значение — один URL в кавычках (типично `source_url: "https://..."`), чтобы править метаданные заметок.
   - Опционально: сырой текст с изолированными URL — только если простой regex без ложных срабатываний; иначе ограничиться ссылками + frontmatter.

4. **Режимы:**
   - Один файл: `--file` (обязателен для MVP).
   - Рекурсия по базе (`--all` или glob) — второй этап, если нужно в tasks.

5. **Вывод:** при `--dry-run` печатать строки `old -> new` в stderr или stdout; при записи — краткое резюме (сколько файлов, сколько замен).

## Risks / Trade-offs

- **[Risk]** HEAD не поддерживается сервером → редирект не раскрывается полностью. **Mitigation:** оставить исходный URL; при необходимости в будущем fallback на GET с ограничением размера тела.
- **[Risk]** Двойная замена или частичное пересечение regex. **Mitigation:** собирать замены по позициям или заменять с конца файла; тесты на типичные статьи.
- **[Trade-off]** Не трогаем `t.me` по правилам `urlutil` — согласованность с импортом, но пользователь может ожидать раскрытия; задокументировать.

## Migration Plan

Не требуется: опциональная CLI-команда, данные не мигрируются автоматически.
