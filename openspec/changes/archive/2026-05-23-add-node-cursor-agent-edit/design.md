## Context

В проекте уже реализована нормализация узла через `cursor-agent` (`internal/api/node_normalization.go`): асинхронная job, парсинг `stream-json`, инкрементальные логи, UI-панель на `NodeActionBar`. Новая функция — тот же операционный паттерн, но промт формируется из **пользовательской инструкции**, а не из фиксированного шаблона нормализации.

Ограничения: локальный/self-hosted сервер, `KB_DATA_PATH` как рабочая директория агента, git-first sync после успеха.

## Goals / Non-Goals

**Goals:**

- Кнопка «Редактировать с агентом» и модалка с текстовой инструкцией перед запуском.
- Серверный запуск `cursor-agent` с контекстом узла (path, metadata, annotation, content) и инструкцией пользователя.
- Наблюдаемость: статус операции + нижняя панель логов с polling (`after` offset).
- Блокировка повторного запуска для того же path, пока операция running.
- Автоматический `sync` после успешного шага edit (если git не отключён).

**Non-Goals:**

- Редактирование нескольких узлов за одну операцию.
- Выбор модели/профиля агента из UI.
- Запуск агента из браузера без сервера.
- Замена или удаление существующей «Нормализации».
- Пресеты/шаблоны инструкции в модалке (только свободный ввод).

## Decisions

1. **Отдельная операция, не расширение normalize**
   - Решение: новый job type `node_agent_edit` и отдельные endpoint-ы `/api/nodes/{path}/agent-edit`, `/api/node-agent-edit/{id}`, `/api/node-agent-edit/{id}/logs`.
   - Почему: разные контракты (тело запроса с `instruction`), метаданные job и UX; проще тестировать и эволюционировать независимо.
   - Альтернатива: параметр `mode` у normalize. Отклонено — смешивает фиксированный и произвольный сценарии.

2. **Переиспользование cursor runner и парсера логов**
   - Решение: вынести общий запуск `cursor-agent` и `parseCursorLogEvents` в shared helper (или общий тип runner с `Run(ctx, workdir, prompt, onLog)`); agent-edit и normalize только собирают промт.
   - Почему: дублирование stdout/stderr и JSON-парсинга уже есть в normalization — DRY снижает риск расхождения логов в UI.

3. **Формирование промта**
   - Решение: сервер оборачивает `instruction` пользователя системными ограничениями: редактировать **только** файл узла по `node.Path`, не менять path/имя файла, сохранять смысл, плюс блок контекста узла (как в normalize).
   - Почему: снижает риск случайного переименования/удаления соседних файлов.
   - Альтернатива: передавать instruction как есть. Отклонено из соображений безопасности git-tree.

4. **Валидация instruction**
   - Решение: обязательное непустое поле после trim, максимальная длина (например 8–16 KiB) на API.
   - Почему: защита от случайного пустого запуска и чрезмерных промтов.

5. **UI: модалка + та же панель логов**
   - Решение: модалка (`Dialog`) с `textarea`, кнопки «Отмена» / «Запустить»; после старта — нижняя fixed-панель по образцу нормализации (compact/raw, auto-scroll).
   - Почему: пользователи уже знакомы с паттерном; минимум новых компонентов.

6. **Совместимость с unified job API**
   - Решение: при необходимости зарегистрировать тип `node_agent_edit` в общем job-слое (как fallback для `startJob`), но primary path — dedicated endpoints.
   - Почему: единообразие с normalize/dump-images в UI fallback.

7. **Сохранение последней инструкции в sessionStorage**
   - Решение: при успешном запуске agent-edit сохранять `instruction` в `sessionStorage` под ключом, привязанным к `node_path` (например `kb:agent-edit-instruction:{path}`); при открытии модалки подставлять сохранённое значение, если оно есть.
   - Почему: удобно повторять похожие правки на том же узле без копирования из истории.
   - Ограничение: только текущая вкладка браузера; не синхронизируется между устройствами (достаточно для локального UI).

## Risks / Trade-offs

- [Риск] Агент выполнит нежелательные правки по свободной инструкции → Митигация: жёсткие ограничения в промте, git history, опциональный ручной review до sync (sync только после success агента).
- [Риск] `cursor-agent` недоступен → Митигация: 503/явная ошибка при старте (как normalize).
- [Риск] Две нижние панели (normalize + agent-edit) конфликтуют визуально → Митигация: показывать только активную операцию; при новом запуске закрывать/заменять панель предыдущей операции того же типа.
- [Trade-off] Дублирование endpoint-ов относительно normalize → проще контракты и независимая эволюция.

## Migration Plan

- Деплой: обновление server + web; без миграции данных.
- Откат: откат бинарника/образа; незавершённые операции теряются из in-memory job store (как у normalize).

## Resolved Questions

- **Пресеты инструкции:** нет — только свободное поле ввода.
- **sessionStorage:** да — последняя успешно отправленная инструкция сохраняется per `node_path` и подставляется при следующем открытии модалки.
