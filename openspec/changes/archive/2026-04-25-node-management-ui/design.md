## Context

Страница узла (NodePage) отображает содержимое записи с метаданными и двумя действиями: отметка «Проверено» и перевод. Управляющие элементы визуально разрозненны — кнопка «Проверено» висит рядом с заголовком, кнопка перевода — отдельным блоком выше. Нет доступа к деструктивным операциям (удаление, перемещение) и к git-операциям из UI.

В бэкенде уже есть git-инфраструктура: `ExecGitCommitter` с методами `CommitNode` и `Sync`, сериализованная очередь, периодический sync-runner. Но нет методов для удаления/перемещения узлов и нет API для ручного коммита всех изменений.

LLM-интеграция уже есть в проекте: OpenAI Responses API используется в `internal/ingestion/llm` для оркестратора. Конфигурация LLM — через переменные окружения.

## Goals / Non-Goals

**Goals:**

- Объединить управляющие действия на странице узла в единую панель (action bar)
- Реализовать удаление узла с обязательным подтверждением через модалку
- Реализовать перемещение узла между темами с удобным UX (дерево тем, предпросмотр пути)
- Добавить в Navbar кнопку **«Сохранить»** (git commit + push) с автогенерацией commit message через LLM и toast-уведомлениями снизу
- Минимальные изменения: новый функционал строится на существующих паттернах (Go handler + React hook)

**Non-Goals:**

- Полноценный редактор содержимого узла (in-place editing)
- История изменений / undo для деструктивных операций
- Пакетные операции (массовое удаление/перемещение)
- Механизм soft-delete / корзина
- Поддержка перемещения узлов с переводами (первая итерация — перемещение только оригинала)
- Конфликты при параллельном редактировании

## Decisions

### 1. Панель действий (Action Bar) — sticky toolbar под заголовком

**Решение**: Горизонтальная полоса с кнопками действий, расположенная сразу под breadcrumbs и заголовком узла. Кнопки слева направо: «Проверено» (toggle), «Перевести» (для статей), «Переместить», «Удалить».

**Альтернативы**:

- *Dropdown menu (⋮)*: компактнее, но требует лишнего клика и скрывает доступные действия
- *Боковая панель*: занимает место, неудобна на мобильных

**Обоснование**: Action bar виден сразу, не скрывает действия. Sticky-позиционирование сохраняет доступ при прокрутке длинного контента. Разделение через spacing и divider между основными (toggle, перевод) и деструктивными (перемещение, удаление) действиями.

### 2. Модалка подтверждения удаления — AlertDialog (Radix)

**Решение**: Использовать Radix AlertDialog (компонент `ui/alert-dialog.tsx` из shadcn/ui) — стилизованную модалку с двумя кнопками: «Отмена» и «Удалить» (красная). Показывает заголовок и путь узла. Поддерживает закрытие по Escape.

**Обоснование**: Radix AlertDialog — стандартный паттерн для необратимых действий. Устойчива к случайному клику (фокус на «Отмену», Escape закрывает).

### 3. Модалка перемещения — Dialog с полным путём и деревом тем

**Решение**: Полноценная модалка (Radix Dialog) с двумя взаимодополняющими способами задания пути:

1. **Текстовое поле «Новый путь»** — единое поле, предзаполненное текущим путём узла. Пользователь MAY редактировать его произвольно, задавая любой полный путь (включая несуществующие темы и новый slug). Промежуточные директории создаются рекурсивно при перемещении. Это основной способ — даёт максимальную свободу.
2. **Дерево тем** (компонент TreeView, переиспользуемый из OverviewPage) — вспомогательный способ для навигации по существующим темам. Клик по узлу дерева обновляет текстовое поле пути (только тему, сохраняя текущий slug).

**Элементы модалки:**

- Текущий путь (нередактируемый, для справки)
- Текстовое поле «Новый путь» (редактируемое, полный путь включая slug)
- Дерево существующих тем (клик заполняет тему в текстовом поле)
- Кнопки «Отмена» и «Переместить»

**UX-flow**:

1. Открытие модалки — текстовое поле содержит текущий путь, дерево раскрыто на текущей теме
2. Пользователь может: (а) напрямую отредактировать полный путь в текстовом поле, (б) выбрать тему в дереве → путь обновляется с новым корнём + текущий slug
3. Путь может содержать несуществующие сегменты — при подтверждении создаются все промежуточные директории
4. Кнопка «Переместить» активна только если путь изменился и нет конфликта
5. После перемещения — переход на новую страницу узла

**Альтернативы**:

- *Только дерево + отдельный slug*: ограничивает пользователя существующими темами, не позволяет создать новую тему напрямую
- *Простое текстовое поле без дерева*: не даёт навигации по существующей структуре, легко ошибиться в пути
- *Drag & drop*: сложно реализовать, не работает на мобильных

### 4. Кнопка «Сохранить» в Navbar

**Решение**: Справа в Navbar (между навигацией и logout) после получения статуса git: при **включённом** git и `has_changes: true` — активная кнопка **«Сохранить (N)»**; при включённом git и чистом дереве — неактивная **«Сохранить»** с подсказкой по наведению; при **503** (git отключён, например KB_GIT_DISABLED) — всегда неактивная **«⚠️ Сохранить»** с подсказкой о причине. Polling каждые 30 секунд через `GET /api/git/status`. Успех и ошибки — **Radix Toast снизу экрана**, не текстом рядом с кнопкой.

Поток:

1. Frontend периодически проверяет `GET /api/git/status`
2. Если `has_changes: true` и git доступен → активная «Сохранить (N)»; иначе — неактивная кнопка (с эмодзи предупреждения при отключённом git)
3. По клику на активной → `POST /api/git/commit` — сервер вызывает `git diff --stat`, отправляет в LLM для генерации сообщения, выполняет `git add -A && git commit && git push`
4. Во время запроса подпись «Сохранение...», кнопка disabled
5. После успеха — toast с сообщением коммита, обновление статуса; при отсутствии изменений кнопка снова неактивна

**LLM-генерация commit message**: Используем OpenAI Responses API (как в orchestrator). Входные данные: `git diff --stat --cached` (или `--name-status`). Instructions — сгенерировать conventional commit message. Важно: промпт на русском (соглашение проекта), commit message на английском (conventional commits).

### 5. API-дизайн

**DELETE /api/nodes/{path}**:

- Удаляет файл `.md` и директорию вложений (если есть)
- Возвращает 200 с `{ path, deleted: true }`
- 404 если не найден

**POST /api/nodes/{path}/move**:

- Тело: `{ target_path: "topic/subtopic/slug" }` — полный целевой путь (тема + slug), как в delta specs
- Перемещает файл `.md` и вложения
- Обновляет frontmatter: `updated`
- Возвращает 200 с новым узлом
- 404 если не найден, 409 если целевой путь занят

**GET /api/git/status**:

- Возвращает `{ has_changes: bool, changed_files?: number }`
- Вызывает `git status --porcelain`
- 503 если git отключён

**POST /api/git/commit**:

- Без тела (или опционально `{ message?: string }`)
- Выполняет `git diff --stat`, отправляет в LLM, коммитит, пушит
- Возвращает `{ message: "сгенерированное сообщение", committed: true }`
- 503 если git отключён; при недоступности LLM — fallback `chore: manual commit via UI`, коммит выполняется

### 6. Backend-реализация — Store методы

**DeleteNode**: `os.Remove` для `.md` файла + `os.RemoveAll` для директории вложений (если имя совпадает со slug). Все операции через `afero.Fs` (как существующие методы Store).

**MoveNode**: `os.Rename` для `.md` файла + директории вложений. Создаёт промежуточные директории при необходимости. Проверяет отсутствие конфликта в целевом пути.

## Risks / Trade-offs

- **[Удаление необратимо]** → Нет soft-delete, но git даёт возможность восстановления. В модалке показываем предупреждение. В будущем можно добавить корзину.
- **[Перемещение не переносит переводы]** → Первая итерация: перевод остаётся на старом месте. Для корректной работы нужна связь `translation_of` → оставлено на следующую итерацию.
- **[LLM-зависимость для коммита]** → Если LLM недоступна — показывать ошибку через toast, не блокировать UI без обратной связи. При недоступности LLM — fallback на шаблонное сообщение `chore: manual commit via UI`.
- **[Polling git-статуса]** → Нагрузка от polling каждые 30 сек минимальна (git status --porcelain быстр). При необходимости можно переключить на SSE.
- **[Сериализация git-операций]** → Существующий `SerializedGitCommitter` уже обеспечивает очередь. Коммит через UI должен использовать тот же экземпляр.
