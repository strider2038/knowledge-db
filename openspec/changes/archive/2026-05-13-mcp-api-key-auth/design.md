## Context

В проекте уже существует маршрут `/api/mcp`, но handler возвращает `501` и не предоставляет MCP-протокол. Текущая спецификация `mcp-server` ориентирована на cookie-сессию при включённой web-auth, что неудобно для machine-to-machine интеграций (OpenWebUI, кодинговые агенты), где ожидается статический токен.

Проект имеет явный local-first и personal-first фокус. Поэтому полноценный lifecycle ключей (CRUD, хранение в БД, UI-админка, ротация без рестарта) на данном этапе является избыточным по сложности и стоимости поддержки.

## Goals / Non-Goals

**Goals:**

- Реализовать production-ready MCP endpoint для интеграции внешних MCP-клиентов.
- Обеспечить простой и надёжный доступ по `KB_MCP_API_KEY` через Bearer auth.
- Использовать `github.com/modelcontextprotocol/go-sdk` как базовую библиотеку протокола.
- Предоставить минимум полезные инструменты: поиск по заметкам, семантический поиск и чтение заметки по path (`get_note`).
- Переиспользовать текущую логику retrieval/index без дублирования поисковых алгоритмов.

**Non-Goals:**

- Управление множеством API-ключей (создание, отзыв, аудит, last_used) и UI для этого.
- Отдельная БД/таблица ключей.
- RBAC/scopes для MCP tools.
- Изменение публичных REST API для обычного web UI (кроме внутреннего wiring).

## Decisions

### 1) Аутентификация MCP через env key (`KB_MCP_API_KEY`)

- Решение: `/api/mcp` принимает только Bearer-токен, равный значению `KB_MCP_API_KEY`.
- Почему: минимальная сложность для персонального деплоя и понятная операционная модель.
- Альтернативы:
  - Хранение ключей в БД + CRUD: гибче, но заметно сложнее и преждевременно для MVP.
  - Cookie-сессии: плохо подходят для внешних MCP-клиентов и headless интеграций.

### 2) Реализация MCP протокола через `github.com/modelcontextprotocol/go-sdk`

- Решение: использовать go-sdk для объявления сервера и tools, вместо самописного протокольного слоя.
- Почему: снижает риск несовместимости с MCP-клиентами, упрощает поддержку эволюции протокола.
- Альтернативы:
  - Самостоятельная реализация протокола: дороже и рискованнее по совместимости.

### 3) Разделение поиска на два MCP инструмента

- Решение:
  - `search_notes` — keyword/hybrid поиск с поведением, близким к текущему обзорному/поисковому UX.
  - `semantic_search` — поиск с упором на векторную релевантность.
- Почему: явный API для клиентов, простой prompt/tool selection, предсказуемый контракт.
- Альтернатива: один tool с `mode` параметром; более компактно, но менее прозрачно для клиентов на старте.

### 4) Поведение при выключенных эмбеддингах

- Решение: `search_notes` остаётся доступным; `semantic_search` возвращает протокольно корректную ошибку о недоступности semantic-режима.
- Почему: сохраняем offline-first полезность и не ломаем базовый доступ к базе.

### 5) Чтение узла через отдельный tool `get_note`

- Решение: добавить отдельный MCP tool `get_note` с аргументами `path` (обязательный) и `max_chars` (опциональный), который возвращает содержимое и метаданные узла.
- Почему: разделяем этапы поиска и чтения первоисточника; агент получает предсказуемый способ дочитать релевантную запись после `search_notes`.
- Альтернатива: возвращать полный контент прямо в `search_notes`; отклонено из-за перегрузки ответа и ухудшения релевантности ранжирования.

### 6) Конфигурация и валидация

- Решение: `KB_MCP_API_KEY` — обязательный параметр для работы MCP. Если переменная пустая или не задана, MCP endpoint не регистрируется и MCP-функциональность считается отключённой.
- Решение: поддерживается только один ключ из `KB_MCP_API_KEY`; поддержка `KB_MCP_API_KEYS` (мульти-ключи) не входит в этот change.

## Risks / Trade-offs

- [Риск] Утечка ключа из env/логов → Mitigation: не логировать полное значение ключа, маскировать при диагностике.
- [Риск] Один общий ключ для всех клиентов → Mitigation: документировать как MVP-ограничение, оставить путь эволюции к multi-key без смены схемы `Authorization: Bearer`.
- [Риск] Клиенты ожидают иные MCP transport-поведения → Mitigation: использовать go-sdk и покрыть совместимость API-тестами на handshake/tool invocation.
- [Риск] Непредсказуемые результаты semantic при слабой embedding-конфигурации → Mitigation: явный error-path и fallback на `search_notes`.

## Migration Plan

1. Обновить OpenSpec требования (`mcp-server`, новый `mcp-search-tools`).
2. Подключить go-sdk dependency.
3. Реализовать key-auth middleware для `/api/mcp` с `KB_MCP_API_KEY`.
4. Реализовать MCP server/tools и wiring с retrieval/index.
5. Добавить/обновить тесты на auth, tools и semantic unavailable path.
6. Rollback: вернуть заглушку MCP и предыдущую регистрацию маршрута, удалив новую dependency.
