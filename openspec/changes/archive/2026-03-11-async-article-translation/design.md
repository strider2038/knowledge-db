## Context

Перевод статей сейчас выполняется синхронно в рамках одного HTTP‑запроса, что делает систему уязвимой к долгим операциям во внешнем сервисе перевода и таймаутам на уровне HTTP, reverse‑proxy и клиента. Цель изменений — вынести перевод в асинхронный процесс, сохранив при этом простой для пользователя флоу и прозрачную модель состояния.

В систему уже входит `kb-server` (Go, net/http), web‑UI (React, Vite) и инфраструктура фоновых процессов на базе `pior/runnable`. В проекте важны оффлайн‑first, git‑first и простота деплоя (один бинарник + `data/`). Хранение статей и переведённых версий уже реализовано через markdown‑файлы (`*.md` / `*.ru.md`), менять это не планируется.

## Goals / Non-Goals

**Goals:**

- Обеспечить, чтобы длительность вызова внешнего сервиса перевода не приводила к таймауту пользовательского HTTP‑запроса.
- Ввести явную, но лёгкую модель состояния фонового перевода (в памяти сервера).
- Сохранить простой и предсказуемый UX в web‑UI: пользователь может запустить перевод и видеть его статус.
- Минимизировать изменения в существующем API и формате данных на диске.

**Non-Goals:**

- Не вводить персистентную очередь или универсальный job‑scheduler для всего проекта.
- Не проектировать сложный UI для мониторинга всех фоновых задач системы.
- Не менять формат хранения статей и переведённых версий в `data/`.

## Decisions

### In‑memory очередь переводов

- В `kb-server` добавляется компонент `TranslationQueue`:
  - ключом служит идентификатор статьи (например, пара `themePath` + `slug`);
  - для каждой статьи хранится структура состояния: `status` (`pending`, `in_progress`, `done`, `failed`), временные метки, необязательное сообщение об ошибке;
  - данные живут только в памяти процесса, при рестарте очереди и статусы обнуляются.
- Очередь реализуется как комбинация:
  - мапы для хранения состояния по ключу статьи;
  - канала/очереди ключей для фоновой обработки;
  - примитивов синхронизации (mutex) для безопасного доступа из нескольких горутин.

### Интеграция с ingestion‑пайплайном

- В `PipelineIngester.maybeTranslateAndSave`:
  - больше нет прямого вызова `Translator.Translate` в контексте HTTP‑запроса;
  - вместо этого вызывается `queue.Enqueue(themePath, slug)`:
    - если перевода ещё нет и статья не в очереди:
      - создаётся запись со `status=pending`,
      - ключ статьи добавляется в очередь для воркера;
    - если для этой статьи статус уже `pending` или `in_progress` — повторная постановка в очередь не выполняется;
    - дальнейшая логика перевода полностью живёт в фоне.
- Таким образом, ingestion‑запрос завершает работу сразу после сохранения исходной статьи; перевод выполняется независимо.

### Минималистичный article‑based API для UI

- Добавляется два endpoint'а:

  1. `POST /api/articles/{id}/translate`:
     - находит статью по `id`;
     - проверяет:
       - если переведённый файл (`*.ru.md`) уже существует — возвращает `status="done"`;
       - если статья уже в очереди (`pending` / `in_progress`) — возвращает текущий статус, не создавая новую задачу;
       - если перевода нет и статья не в очереди — ставит её в очередь (`queue.Enqueue`) и возвращает `status="pending"`.
  2. `GET /api/articles/{id}/translate`:
     - возвращает текущее состояние:
       - если переведённый файл есть — `status="done"`;
       - если статья в очереди — её статус из `TranslationQueue`;
       - если ни перевода, ни записи в очереди — `status="none"` или эквивалент.

- UI:
  - кнопка «Перевести» вызывает `POST` и получает начальный статус;
  - далее фронт опрашивает `GET` до получения `done` или `failed`, после чего перечитывает статью.

### Запись файлов и защита от гонок

- При записи `*.ru.md` и обновлении оригинального файла воркер использует **атомарную запись**:
  - сначала пишет содержимое во временный файл в той же директории (например, `*.ru.md.tmp`);
  - затем выполняет `os.Rename` в целевой путь — на POSIX это атомарная операция;
  - при падении воркера во время записи во временный файл целевой файл не повреждается.
- Это защищает от сценария, когда пользователь или `git add` захватывает частично записанный файл.

### Синхронизация git-операций

- Все вызовы `CommitNode` (из pipeline при сохранении ноды и из воркера при сохранении перевода) проходят через **общую очередь**:
  - реализация — обёртка над `GitCommitter`, которая сериализует вызовы (например, через канал или mutex + очередь задач);
  - pipeline и воркер не вызывают `git add` / `git commit` / `git push` параллельно в одном репозитории.
- `GitSyncRunner` (периодический fetch + merge) продолжает работать отдельно; при необходимости его можно включить в ту же очередь или оставить независимым (merge читает индекс, commit пишет — при сериализации commit'ов конфликтов меньше).
- Цель: избежать гонок при одновременном `git commit` из разных горутин и снизить риск non-fast-forward при push.

### Фоновый воркер перевода

- Через `pior/runnable` в `kb-server` регистрируется фоновой воркер `TranslationWorker` (при включённом LLM). При отключённом git используется `NoopGitCommitter` — переводы создаются на диске, но без коммитов. Воркер:
  - читает ключи статей из in‑memory очереди;
  - для каждой статьи:
    - переводит статус в `in_progress`;
    - создаёт отдельный `context.WithTimeout` для вызова LLM‑переводчика (не завязанного на HTTP‑таймаут исходного запроса);
    - вызывает существующий `translation.Translator`, который уже умеет делать чанкинг и вызывать LLM;
    - по результату:
      - при успехе создаёт `*.ru.md` и обновляет оригинальный файл (ссылки на перевод) так же, как сейчас делает `maybeTranslateAndSave`;
      - обновляет статус в очереди на `done`;
      - при ошибке — на `failed` + сохраняет диагностическое сообщение.
- Воркера можно реализовать одиночным или как небольшой пул (фиксированное число горутин), в зависимости от допустимой параллельности вызовов к внешнему сервису перевода.

### Наблюдаемость и логирование

- Логируются:
  - постановка статьи в очередь (`pending`);
  - старт обработки (`in_progress`);
  - успешное создание перевода (`done`);
  - ошибки при переводе (`failed`), включая тип ошибки (таймаут, ошибка сервиса и т.п.).
- При наличии подсистемы метрик можно добавить:
  - гистограмму длительности перевода;
  - счётчики ошибок по типам.

## Risks / Trade-offs

- **Риск:** Потеря очереди и статусов при рестарте `kb-server`.
  **Митигирование:** Для текущих сценариев это приемлемо — статьи и их переводы живут на диске; максимум, что теряется, — незавершённые переводы, которые можно перезапустить из UI.

- **Риск:** Повторная обработка одной и той же статьи после сбоев воркера.
  **Митигирование:** Идемпотентность на уровне файлов (перезапись `*.ru.md` допустима) и аккуратная проверка перед добавлением в очередь.

- **Риск:** Нагрузка на внешний сервис перевода при большом количестве параллельных задач.
  **Митигирование:** Ограничение размера пула воркера и конфигурируемые таймауты/ретраи.

- **Риск:** Гонки при одновременном `git commit` / `git push` (воркер, pipeline, пользователь вручную).
  **Митигирование:** Сериализация всех `CommitNode` через общую очередь; атомарная запись файлов (temp + rename). Ручной `git push` пользователя может конфликтовать с push'ем сервера (non-fast-forward) — пользователю нужно делать `git pull` перед push. Рекомендуется документировать в README или AGENTS.md.

- **Риск:** Конфликт при merge, если пользователь и воркер изменили один и тот же файл статьи.
  **Митигирование:** Существующая стратегия: при конфликте merge логировать ошибку и оставить разрешение пользователю. Не пытаться автоматически резолвить — потеря данных хуже задержки.
