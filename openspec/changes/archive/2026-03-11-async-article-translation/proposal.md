## Why

Перевод статей сейчас выполняется синхронно в рамках одного HTTP‑запроса к серверу. При длинных текстах и/или медленном ответе внешнего сервиса перевода обработка упирается в таймауты (HTTP, reverse‑proxy, клиент), что делает функциональность ненадёжной и плохо масштабируемой. Нужно вынести перевод в асинхронный процесс с чёткой моделью состояния и наблюдаемостью.

## What Changes

- Добавить асинхронный пайплайн перевода статей вместо текущего синхронного вызова:
  - перевод выполняется фоновой джобой в отдельной очереди, хранящейся в оперативной памяти `kb-server`;
  - ingestion‑пайплайн при авто‑переводе кладёт запросы на перевод в очередь и не ждёт результата.
- Добавить модель состояний фоновой джобы перевода (например, `pending`, `in_progress`, `done`, `failed`) и базовые поля для отладки (ошибка, тайминги).
- Добавить минималистичный REST API для UI:
  - endpoint запуска перевода статьи, если перевода ещё нет (при уже запущенной джобе повторный запуск запрещён до завершения);
  - endpoint получения текущего статуса перевода статьи и, при готовности, доступа к переведённому содержимому через уже существующие файлы/структуры.
- Добавить фонового воркера в `kb-server`, обрабатывающего in‑memory очередь задач перевода:
  - реализация через `pior/runnable` или другой принятый в проекте механизм фоновых процессов;
  - поддержка ограничения параллелизма и повторов при временных ошибках внешнего сервиса.
- Зафиксировать стратегию хранения переведённого контента:
  - переводы по‑прежнему хранятся в markdown‑файлах рядом с оригиналом, как сейчас;
  - очередь и статусы задач не персистятся на диск и теряются при перезапуске сервера.
- Добавить пользовательский флоу в web‑UI:
  - возможность запускать перевод статьи из UI, если он ещё не создан;
  - индикация того, что перевод в процессе, и опрос статуса без долгих висящих запросов;
  - отображение ошибок перевода.
- Усилить наблюдаемость:
  - логирование ключевых событий жизненного цикла фоновых переводов;
  - при необходимости — метрики по длительности и ошибкам.

## Capabilities

### New Capabilities
- `async-article-translation`: Асинхронная обработка переводов статей через in‑memory очередь и фоновые воркеры с минимальным REST API для запуска/проверки статуса перевода статьи.

### Modified Capabilities
- `article-management`: Требования к модели статей и связанным операциям дополняются поддержкой многоязычного контента и интеграцией с асинхронными переводами (отражение наличия/отсутствия перевода, наличие переведённых версий в CRUD/поиске там, где это необходимо).

## Impact

- Backend (`cmd/kb-server`, `internal/api`, `internal/kb`, `internal/ingestion`):
  - добавление in‑memory очереди переводов и фонового воркера для её обработки;
  - минимальные HTTP‑обработчики для запуска перевода статьи и чтения статуса перевода;
  - адаптация работы с внешним сервисом перевода под асинхронный сценарий, обработка таймаутов и повторов.
- Хранение данных (`data/` и связанные пакеты):
  - формат хранения статей и переведённых версий остаётся прежним;
  - очередь и статусы задач не сохраняются на диск.
- Web UI (`web/`):
  - обновление экранов, связанных с просмотром/редактированием статей, чтобы поддержать запуск перевода и отображение статуса;
  - индикация процессов перевода и ошибок.
- Интеграции и окружение:
  - требования к конфигурации внешнего сервиса перевода (таймауты, лимиты, ключи);
  - возможные изменения в настройках reverse‑proxy, если ранее таймауты подстраивались под синхронный перевод.
