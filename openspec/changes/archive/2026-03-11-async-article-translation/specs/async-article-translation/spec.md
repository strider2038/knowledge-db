## Purpose

Спек описывает поведение системы при асинхронном переводе статей: постановку статей в in‑memory очередь перевода, обработку их в фоне и получение статуса/результата через минимальный article‑based API без риска упереться в HTTP‑таймаут.

## ADDED Requirements

### Requirement: Постановка статьи в очередь перевода

Система ДОЛЖНА (SHALL) ставить статью в очередь перевода при запуске перевода, не дожидаясь завершения внешнего сервиса перевода в рамках исходного HTTP‑запроса.

#### Scenario: Успешная постановка статьи в очередь

- **WHEN** клиент либо ingestion‑пайплайн запускает перевод существующей статьи
- **THEN** система MUST добавить эту статью в in‑memory очередь перевода со статусом `pending` и немедленно вернуть управление вызывающей стороне

#### Scenario: Попытка постановки несуществующей статьи

- **WHEN** клиент отправляет запрос на перевод статьи с несуществующим идентификатором
- **THEN** система MUST вернуть ошибку клиенту и НЕ добавлять запись в очередь перевода

### Requirement: Статусы перевода статьи

Система ДОЛЖНА (SHALL) поддерживать явную модель статусов перевода статьи, отражающую текущий прогресс обработки и результат.

#### Scenario: Переход статуса при начале обработки

- **WHEN** фоновой воркер начинает обрабатывать статью, находящуюся в очереди со статусом `pending`
- **THEN** система MUST перевести её статус в `in_progress` и зафиксировать время начала обработки

#### Scenario: Успешное завершение перевода статьи

- **WHEN** внешний сервис перевода успешно возвращает переведённый текст для статьи
- **THEN** система MUST сохранить переведённый текст в виде `*.ru.md` рядом с исходной статьёй, обновить ссылки в оригинале и перевести статус статьи в очереди на `done`

#### Scenario: Ошибка при выполнении перевода статьи

- **WHEN** при выполнении перевода статьи возникает ошибка (сетевой сбой, таймаут или ошибка сервиса перевода)
- **THEN** система MUST перевести её статус в очереди на `failed` и сохранить диагностическое сообщение об ошибке, доступное через API статуса перевода

### Requirement: Article‑based API для запуска перевода и получения статуса

Система ДОЛЖНА (SHALL) предоставлять REST API, позволяющее запускать перевод статьи и опрашивать статус перевода по идентификатору статьи без использования отдельных job‑идентификаторов.

**API endpoints:**

- `POST /api/articles/translate/{path...}` — запуск перевода статьи; `path` — путь статьи (themePath/slug), например `go/test-article` или `topic/subtopic/article`
- `GET /api/articles/translate/{path...}` — получение статуса перевода по пути статьи

#### Scenario: Запуск перевода статьи через API

- **WHEN** клиент отправляет запрос `POST` на запуск перевода статьи через публичный REST endpoint
- **THEN** система MUST либо поставить статью в очередь перевода (если перевода ещё нет и задача не запущена), либо вернуть текущий статус перевода, не создавая дубликат, и в любом случае НЕ дожидаться завершения операции перевода в рамках этого запроса

#### Scenario: Получение статуса перевода статьи

- **WHEN** клиент отправляет запрос `GET` для получения статуса перевода по идентификатору статьи
- **THEN** система MUST вернуть одно из состояний (`none`, `pending`, `in_progress`, `done`, `failed`) и, в случае `done`, указать, что переведённый контент доступен через соответствующий markdown‑файл

#### Scenario: Повторный запуск при уже идущем переводе

- **WHEN** клиент повторно отправляет запрос на запуск перевода статьи, для которой статус уже `pending` или `in_progress`
- **THEN** система MUST вернуть текущий статус перевода и НЕ создавать новую задачу в очереди

### Requirement: Асинхронный воркер перевода

Система ДОЛЖНА (SHALL) обрабатывать накопившиеся в in‑memory очереди статьи для перевода в фоне с помощью отдельного воркера, не блокируя основной HTTP‑поток обработки запросов.

#### Scenario: Регулярная обработка очереди статей на перевод

- **WHEN** фоновой воркер периодически запускается или пробуждается для обработки очереди статей на перевод
- **THEN** система MUST выбирать статьи в статусе `pending`, переводить их в статус `in_progress`, выполнять операцию перевода через внешний сервис и по результату переводить статус в `done` или `failed`

#### Scenario: Идемпотентная обработка статьи

- **WHEN** одна и та же статья по ошибке обрабатывается воркером повторно (например, после сбоя)
- **THEN** система MUST гарантировать, что повторная обработка не приводит к некорректному дублированию переведённого контента и не нарушает целостность файлов статьи и её перевода

### Requirement: Атомарная запись файлов перевода

Система ДОЛЖНА (SHALL) записывать файлы перевода (`*.ru.md`) и обновления оригинального файла атомарно, чтобы исключить захват частично записанного содержимого при параллельных операциях (git add, чтение).

#### Scenario: Запись файла перевода

- **WHEN** воркер сохраняет переведённый контент в файл `*.ru.md`
- **THEN** система MUST сначала записать содержимое во временный файл в той же директории, затем выполнить атомарный rename в целевой путь

#### Scenario: Защита при падении во время записи

- **WHEN** воркер прерывается во время записи файла перевода
- **THEN** целевой файл `*.ru.md` НЕ должен содержать частично записанные данные (остаётся прежним или отсутствует)

### Requirement: Синхронизация git-операций

Система ДОЛЖНА (SHALL) сериализовать все вызовы `CommitNode` (git add, commit, push), выполняемые из pipeline и воркера перевода, чтобы исключить гонки при одновременном доступе к репозиторию.

#### Scenario: Последовательное выполнение commit

- **WHEN** pipeline и воркер перевода одновременно завершают операции, требующие git commit
- **THEN** система MUST выполнять `CommitNode` последовательно, не допуская параллельного выполнения git add/commit/push в одном репозитории
