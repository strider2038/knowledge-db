## Context

Проект уже использует `KB_DATA_PATH/.kb` для служебных данных (`index.db`, `chat.db`), а пользовательская база остается в markdown/git. Для сопровождения по кейсам требуется:
- сохранять сырые входы Telegram для восстановления первичного контекста;
- сохранять пользовательские багрепорты из UI вместе с полным диагностическим контекстом;
- предоставлять агентам удобный программный доступ к этим артефактам через MCP.

Ключевые ограничения:
- оффлайн-first и локальное владение данными;
- не смешивать служебные debug-инструменты с основным MCP поиска/чтения заметок;
- не вводить обязательную БД-миграцию для debug данных.

## Goals / Non-Goals

**Goals:**
- Добавить файловое хранилище debug-артефактов в `KB_DATA_PATH/.kb`.
- Добавить UI-поток создания багрепорта на страницах узла, поиска и чата.
- Добавить отдельный debug MCP endpoint с отдельным API-ключом.
- Сохранить полные данные для чата в issue-репорте (без усечения и без санитизации на этом этапе).
- Добавить статусный пайплайн issues (`new`, `investigating`, `fixed`) с API обновления.
- Добавить автоматическую ротацию Telegram raw логов с TTL 14 дней.

**Non-Goals:**
- Санитизация/редакция чувствительных данных.
- Переход на отдельную БД для issues/raw-логов.
- Интеграция debug-инструментов в основной MCP endpoint.
- Автоматическая triage/классификация багрепортов.

## Decisions

1. **Source of truth для debug-данных — файловая система (`.kb/issues`, `.kb/telegram-raw`)**
- Почему: соответствует локально-прозрачной модели проекта, легко читать человеком и агентом, нет миграций схемы.
- Альтернатива: SQLite-таблицы для issues/raw.
- Почему не выбрано сейчас: дополнительная сложность, а текущий сценарий отладки лучше покрывается файловыми артефактами.

2. **Формат issues — Markdown с frontmatter**
- Почему: удобно для ручного чтения и последующей передачи агенту; сохраняет структуру для машинного парсинга.
- Альтернатива: JSON-only файлы.
- Почему не выбрано: хуже читаемость и review UX.

3. **Формат Telegram raw — NDJSON по дням (`YYYY-MM-DD.ndjson`)**
- Почему: append-friendly, устойчив к частичным сбоям записи, удобен для tail/grep.
- Альтернатива: один JSON-массив или отдельный файл на сообщение.
- Почему не выбрано: JSON-массив сложнее безопасно апдейтить, per-message дает слишком много файлов.

4. **Отдельный debug MCP endpoint и отдельный ключ**
- Почему: разделяет продуктовые и служебные инструменты, снижает шум и риски случайного использования.
- Альтернатива: добавить инструменты в существующий `/api/mcp`.
- Почему не выбрано: смешение доменов ответственности и более сложный контроль доступа.

5. **Issue создается через backend API, а не прямой записью из браузера**
- Почему: единая валидация, гарантированная запись в `KB_DATA_PATH`, контроль формата и файловых имен.
- Альтернатива: local-only сохранение в браузере.
- Почему не выбрано: недоступно для серверного пути `.kb` и непереносимо между устройствами.

6. **Status pipeline для issues через frontmatter + API update**
- Почему: позволяет формализовать сопровождение кейсов без перехода на БД и без потери читаемости файлов.
- Альтернатива: статус только вручную в файле или отдельная БД таблица.
- Почему не выбрано: ручной режим плохо автоматизируется; БД добавляет лишнюю сложность на текущем этапе.

7. **TTL raw Telegram логов фиксирован 14 дней**
- Почему: ограничивает рост служебных файлов и сохраняет практически полезное окно диагностики недавних проблем.
- Альтернатива: бессрочное хранение или только ручная очистка.
- Почему не выбрано: бессрочное хранение ведет к неограниченному росту объема и шуму.

## Risks / Trade-offs

- **[Риск] Рост объема файлов в `.kb/issues` и `.kb/telegram-raw`** → Митигация: иерархия по дате, лимиты выборки в MCP, автоматическая очистка raw-логов старше 14 дней.
- **[Риск] Полные чат-данные могут содержать чувствительную информацию** → Митигация: явно документировать режим “без санитизации”, позже добавить opt-in маскирование.
- **[Риск] Debug MCP может раскрыть служебные данные при утечке ключа** → Митигация: отдельный `KB_MCP_DEBUG_API_KEY`, отключенный по умолчанию endpoint, приватный контур доступа.
- **[Риск] Дублирование контекста между issue и raw-логом** → Осознанный trade-off: для UI-issues по целям change сохраняется полный диагностический контекст (в т.ч. чат), а сырые Telegram update остаются только в `.kb/telegram-raw` и не подмешиваются в issue автоматически. Дальнейшая митигация объёма — ссылки на сущности (`chat_id`, `message_id`, `session_id`) и opt-in маскирование (вне текущего scope).

5. Добавить API/интеграционные тесты и обновить smoke-проверки.
6. Rollback: отключить новые флаги (`KB_TELEGRAM_RAW_LOG_ENABLED=false`, пустой `KB_MCP_DEBUG_API_KEY`), endpoint и запись raw/issues перестанут использоваться без влияния на основную базу.
