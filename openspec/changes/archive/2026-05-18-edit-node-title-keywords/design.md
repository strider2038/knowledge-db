## Context

Страница узла уже отображает `title` и `keywords`, но редактирование доступно только через ручную правку markdown-файла. При этом API PATCH узла ограничен полем `manual_processed`, а web-клиент не имеет контракта для редактирования `title`/`keywords`.

Изменение затрагивает backend (контракт PATCH + нормализация frontmatter) и frontend (модалки, UX тегов, подсказки).

## Goals / Non-Goals

**Goals:**
- Дать пользователю быстрый ручной edit `title` и `keywords` на странице узла.
- Сохранить единый PATCH endpoint для частичного обновления метаданных.
- Нормализовать и санитизировать ввод (`trim`, dedupe, исключение пустых значений).
- Повысить скорость заполнения тегов за счёт подсказок из существующего словаря keywords.

**Non-Goals:**
- Полный WYSIWYG-редактор frontmatter.
- Введение отдельной таблицы/индекса тегов в backend-хранилище.
- Массовое переименование или bulk-редактирование keywords по всей базе.

## Decisions

1. Расширяем существующий PATCH `/api/nodes/{path...}`, а не вводим новый endpoint
- Решение: использовать тот же endpoint с whitelist полей `manual_processed | title | keywords`.
- Почему: единая точка изменения frontmatter и меньше API-поверхности.
- Альтернатива: отдельные endpoints `PATCH /title` и `PATCH /keywords`. Отклонено как избыточное дробление контракта.

2. Санитизация на уровне Store
- Решение: нормализовать значения перед записью в файл:
  - `title`: trim, пустое значение удаляет поле.
  - `keywords`: trim, remove empty, dedupe с сохранением порядка.
- Почему: защита данных независимо от клиента.
- Альтернатива: санитизировать только во frontend. Отклонено — сервер должен быть источником корректности.

3. Подсказки keywords формируем из существующих узлов без нового backend endpoint
- Решение: web-клиент собирает keywords через paginated `GET /api/nodes` и строит частотный список.
- Почему: достаточно для текущего масштаба, без дополнительного server API.
- Альтернатива: новый endpoint словаря keywords. Отложено до появления подтверждённых проблем производительности.

4. UX тегов по паттерну token input
- Решение: chips + quick add (`Enter`, `,`) + remove (`x`, `Backspace`) + кликабельные suggestions.
- Почему: знакомый и быстрый паттерн для тегов, минимальный порог ошибок.
- Альтернатива: простое textarea с comma-separated строкой. Отклонено как менее надёжное и менее удобное.

## Risks / Trade-offs

- [Риск] Большая база может увеличить число запросов для построения suggestions.
  - Митигация: загрузка только при открытии модалки тегов, ограничение выдачи подсказок в UI.
- [Риск] Изменение PATCH-контракта может повлиять на старые клиенты с ожиданием ровно одного поля.
  - Митигация: backward-совместимость сохранена — `manual_processed` остаётся поддерживаемым.
- [Trade-off] Дополнительные состояния в NodePage увеличивают сложность компонента.
  - Компенсация: выделены предсказуемые обработчики сохранения и локальные функции нормализации.
