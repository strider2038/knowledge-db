## Context

Текущий чат в knowledge-db ориентирован на single-turn или ограниченный контекст запроса и не предоставляет модель persistent-сессий. Пользовательский запрос требует UX и поведения, привычных для ChatGPT/OpenWebUI: список чатов, возврат к прошлому диалогу, продолжение сессии, автоматическое управление длиной контекста, но в формате временного вспомогательного инструмента.

Ограничения проекта: оффлайн-first, git-first для базы знаний, но чат-подсистема допускает отдельное временное хранилище (SQLite), т.к. это вспомогательная функция.

## Goals / Non-Goals

**Goals:**
- Ввести сущность chat session с хранением в SQLite и возможностью продолжения.
- Гарантировать, что ответы LLM учитывают историю текущего чата.
- Ограничить передаваемый в модель контекст и автоматически суммаризировать старые сообщения.
- Добавить UI-паттерн «история чатов + активный чат», включая удаление и ручное переименование чата.

**Non-Goals:**
- Не внедрять облачную синхронизацию чатов между устройствами.
- Не решать в этом change продвинутую коллаборацию (shared chats, ACL).
- Не превращать чат в долговременное хранилище знаний пользователя.

## Decisions

1. Ввести новый доменный модуль `chat-session-memory` с хранением в SQLite.
- Почему: для временных чатов важнее простота lifecycle-операций (delete, rename, list, TTL), чем git-diff-прозрачность.
- Альтернатива: файловое хранение в data/. Отклонена как избыточная для вспомогательного временного функционала.

2. Модель данных чата: `session` + ограниченный буфер сообщений + `summary windows`.
- Почему: позволяет держать только актуальный рабочий контекст и снижать объем хранимых данных.
- Альтернатива: хранить полную историю. Отклонена, потому что здесь нужен временный, а не архивный характер хранения.

3. Алгоритм лимита контекста: комбинированный порог (оценка по рунам + сообщения). При превышении порога сворачивать oldest-range в служебный summary-message.
- Почему: в первой версии используется лёгкая оффлайн-оценка размера контекста по рунам (приближение token budget) вместе с лимитом сообщений для ограничения worst-case.
- Альтернатива: только число сообщений или только токены. Отклонена как менее устойчивая стратегия.

4. Заголовок чата генерировать автоматически (из первых сообщений), но разрешить ручное переименование.
- Почему: автогенерация ускоряет старт, ручное имя повышает управляемость списка чатов.
- Альтернатива: только автогенерация. Отклонена по решению пользователя.

5. API слой: полностью заменить старый chat endpoint новым набором endpoint'ов списка/создания/получения/продолжения/удаления/переименования чатов.
- Почему: функциональность в отдельной ветке, поэтому упрощаем архитектуру и исключаем двойную поддержку старого/нового контракта.
- Альтернатива: сохранять обратную совместимость и постепенную миграцию. Отклонена как лишняя сложность на данном этапе.

6. Summary-сообщения оставить служебными и не показывать в UI на первой версии.
- Почему: это внутренняя оптимизация контекста, а не пользовательский контент.
- Альтернатива: показывать summary в ленте. Отклонена на первую версию.

7. Разделить исходное сообщение пользователя и retrieval-запрос к базе знаний.
- Почему: разговорные формулировки вроде "Что есть про RAG в базе знаний?" не должны превращаться в FTS-запрос `RAG AND знаний`, иначе чат теряет тематические источники, которые обзор находит по `RAG`.
- Реализация: перед retrieval строится компактный поисковый запрос с удалением служебных слов и сохранением доменных терминов, акронимов и названий.
- Альтернатива: использовать исходное сообщение без нормализации. Отклонена из-за слишком строгого keyword AND-поиска.

8. Ввести routing сообщений чата по режимам `chat_memory`, `rag_kb`, `hybrid`.
- Почему: вопросы о резюме/контексте диалога не должны искать по базе знаний, а явные вопросы "в базе" должны получать более строгий RAG-fallback.
- Реализация: эвристический детектор режима по фразам запроса; `chat_memory` строит ответ из истории сессии и возвращает пустой список источников.
- Альтернатива: всегда выполнять RAG. Отклонена из-за шума и неверных источников в memory-only сценариях.

9. Фильтровать источники чата с разными правилами для lexical/exact и vector-only результатов.
- Почему: общий hybrid score не является абсолютной мерой релевантности; keyword/exact результаты с overlap по теме могут иметь score ниже длинного chunk-heavy документа.
- Реализация: lexical/exact/keyword источники сохраняются при overlap доменных терминов, weak vector-only источники отбрасываются по score/overlap, а обрезка хвоста по gap применяется только после минимального набора релевантных источников.
- Альтернатива: единый высокий порог score для всех источников. Отклонена, потому что он срезал релевантные RAG-ноды.

10. Безопасный fallback при отсутствии релевантного RAG-контекста.
- Почему: для явного вопроса по базе знаний лучше честно сообщить, что данных нет, чем просить LLM отвечать по пустому контексту.
- Реализация: если после retrieval и фильтрации контекст пуст в `rag_kb`, backend возвращает фиксированное сообщение "В базе знаний не найдено релевантной информации по запросу." и сохраняет его в истории.
- Альтернатива: всегда вызывать LLM. Отклонена из-за риска неподтверждённых ответов.

## Risks / Trade-offs

- [Рост SQLite и фрагментация временных данных] → Ввести TTL/cleanup политику с on-access очисткой (при запросе списка чатов) и vacuum в этой же процедуре.
- [Потеря важных деталей при суммаризации] → В первой версии использовать локальную extractive-summary эвристику: сохранять роли и усечённое содержание oldest-range без вызова LLM. Для более качественной абстрактивной summary выделить отдельное последующее изменение с консервативным prompt и тестами на сохранение ключевых фактов.
- [Регрессия при полном переключении на новый chat API] → Покрыть критические пользовательские сценарии API-тестами и smoke-тестом UI.
- [Случайное удаление чата пользователем] → Добавить подтверждение удаления и мягкий guard в UI.
- [Чат теряет релевантные источники из-за слишком строгого retrieval/filtering] → Нормализовать retrieval-запрос, сохранять lexical/exact совпадения при overlap и покрыть сценарий "Что есть про RAG в базе знаний?" тестами.

## Configuration Decisions

- Дефолт для `openai/gpt-4o-mini` через OpenRouter: комбинированный порог `max_context_runes=24000`, `max_messages=40`, `max_summary_runes=1200`. Эти значения держат рабочий prompt в безопасном приближённом бюджете для первой версии.

## Migration Plan

1. Добавить новый модуль хранения chat sessions на SQLite и сервис контекстного окна.
2. Полностью переключить chat API на новые endpoint'ы и подключить их к текущему RAG-chat pipeline.
3. Настроить routing режимов чата, compact retrieval query, фильтрацию источников и fallback для пустого RAG-контекста.
4. Обновить web UI: список чатов, выбор активного, продолжение диалога.
5. Удалить/деактивировать старый chat endpoint и обновить роутинг клиентов на новый API.
6. Rollback: восстановить предыдущую версию ветки; SQLite-данные чатов можно очистить без влияния на базу знаний.
