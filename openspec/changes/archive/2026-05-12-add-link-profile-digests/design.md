## Context

В текущей базе `type=link` часто означает просто "URL не был скопирован как статья". На практике в этом классе оказываются разные сущности: GitHub-репозитории, документация, SaaS-продукты, онлайн-инструменты, каталоги, длинные статьи без полного копирования и новости. Все найденные link-узлы в пользовательской базе имеют пустое markdown-тело, поэтому RAG получает только `title`, `annotation`, `keywords` и `source_url`.

Это смешивает две оси:

- локальная форма знания: полная статья, заметка, закладка;
- природа внешнего источника: репозиторий, статья, новость, документация, продукт, каталог и т.п.

Решение должно сохранить offline-first/git-first модель: итоговое знание хранится в markdown-файле и остаётся читаемым/редактируемым без сервера и интернета.

## Goals / Non-Goals

**Goals:**

- Разделить `type` как локальную форму хранения и профиль внешнего источника.
- Генерировать концептуальные digest-тела для внешних источников, когда полная копия не нужна или вредна.
- Улучшить RAG-контекст для link/note узлов без обязательного копирования исходных статей целиком.
- Дать пользователю точечное UI-действие для обновления описания существующего узла из `source_url`.
- Сохранить совместимость существующих узлов без новых обязательных frontmatter-полей.
- Поддержать основной набор профилей: `repository`, `documentation`, `product_service`, `online_tool`, `directory_catalog`, `learning_resource`, `article`, `news`, `social_post`.

**Non-Goals:**

- Не реализовывать массовую миграцию существующей базы в рамках обязательного runtime-пути.
- Не вводить новый обязательный тип контента вместо `article|link|note`.
- Не сохранять полный README или полный текст длинной статьи в `type=link`, если нужен только digest.
- Не решать задачу цитирования/inline citations в ответах RAG-чата.

## Decisions

### Decision 1: `type` остаётся локальной формой знания

`type` продолжает принимать существующие значения:

- `article` — полная локальная копия источника;
- `note` — локальная заметка, выжимка или интерпретация, опционально с `source_url`;
- `link` — закладка на внешний ресурс, где локально хранится описание/профиль ресурса.

Для природы источника добавляются опциональные поля:

- `source_kind`: `repository`, `documentation`, `product_service`, `online_tool`, `directory_catalog`, `learning_resource`, `article`, `news`, `social_post`, `unknown`;
- `content_profile`: `repository_profile`, `product_profile`, `documentation_profile`, `online_tool_profile`, `directory_profile`, `learning_resource_profile`, `conceptual_digest`, `brief_digest`, `link_bookmark`.

Альтернатива: добавить новые `type`, например `repository` или `digest`. Это хуже для совместимости и UI, потому что существующие сценарии уже завязаны на `article|link|note`.

### Decision 2: длинные статьи без полного копирования становятся `note`

Если пользователь сохраняет длинную статью, но система не должна копировать её целиком, ingestion создаёт `type=note`, `source_kind=article`, `content_profile=conceptual_digest`. Примеры: "Designing Go Libraries" и "Harness Engineering".

Такой узел содержит авторскую/машинную концептуальную выжимку: главная идея, ключевые принципы, эвристики, ограничения и связанные понятия. Это ближе к локальной заметке, чем к закладке.

Альтернатива: оставить `type=link` и добавить body. Это хуже семантически: ссылка-закладка и digest по статье имеют разное поведение при поиске и чтении.

### Decision 3: репозитории и продуктовые страницы остаются `link`, но получают профильное тело

Для репозиториев, сервисов, инструментов, каталогов и документации ценность узла в том, что пользователь может открыть внешний ресурс. Поэтому они остаются `type=link`, но тело становится не пустым, а профильным digest:

- `repository_profile`: назначение, ключевая идея, основные абстракции, архитектурные свойства, применимость, ограничения;
- `product_profile`: какую работу закрывает продукт/сервис, для кого, основной workflow, данные/интеграции, ограничения;
- `documentation_profile`: ментальная модель документации, основные сущности, типовые сценарии, границы применимости;
- `directory_profile`: область каталога, принцип организации, какие классы ресурсов помогает находить;
- `online_tool_profile`: вход/выход, решаемая задача, ограничения, приватность/локальность, когда полезнее локальной альтернативы;
- `learning_resource_profile`: чему учит ресурс, структура пути, prerequisites, практический результат.

Альтернатива: хранить только расширенную `annotation`. Это не даёт RAG достаточно плотного контекста и плохо чанкуется.

### Decision 4: digest генерируется из источника с фильтрацией технического шума

Для репозиториев основным источником является README, дополненный repo metadata. Для обычных страниц используется fetch content preview или полный извлечённый markdown как вход LLM, но в узел записывается только digest.

Промпт должен запрещать переносить в digest:

- инструкции установки;
- quick start;
- команды запуска;
- длинные примеры кода/API;
- badge/changelog/license/sponsor/contributing секции;
- benchmark-таблицы без концептуального вывода.

Промпт должен сохранять:

- мотивацию и решаемую проблему;
- ключевую модель мышления;
- основные абстракции;
- архитектурные свойства;
- сценарии применимости и ограничения;
- доменные термины для поиска.

### Decision 5: индексирование digest body обязательно

Тело `type=link` с профильным digest и тело `type=note` с conceptual/brief digest должны участвовать в:

- node embedding text;
- searchable text для keyword/FTS;
- chunk indexing, если body достаточно длинное.

Без этого digest будет виден человеку в markdown, но не станет полноценным RAG-контекстом.

### Decision 6: миграция существующих узлов выполняется точечно из UI

Существующие узлы без `source_kind`, `content_profile` и body остаются валидными. Автоматический массовый backfill не выполняется. Вместо этого страница узла получает действие **«Обновить описание из источника»**, которое запускает новый алгоритм для конкретного узла.

Действие доступно, если у узла есть `source_url`. Backend заново получает материал источника, классифицирует `source_kind`, выбирает `content_profile`, генерирует `annotation`, `keywords` и markdown-тело digest. Endpoint может изменить `type`, если текущая классификация была ошибочной: например, новость, сохранённая как `link`, может стать `note` с `content_profile=brief_digest`.

При обновлении нужно сохранять стабильные пользовательские поля: `created`, `source_url`, существующий `source_author` и `source_date`, если новый источник не даёт более точного значения. `manual_processed` не должен неожиданно сбрасываться.

Альтернатива: сделать только CLI backfill. Это хуже для пользовательского контроля: часть узлов требует индивидуального решения, а массовое обновление может перетереть вручную выверенные заметки.

### Decision 7: действие в UI называется «Обновить описание из источника»

Короткое «Обновить» слишком неоднозначно: оно может означать refresh страницы, переиндексацию, git pull или перезагрузку источника. «Обновить описание из источника» явно говорит, что будет пересоздано локальное описание/digest на основе `source_url`.

В UI это действие должно быть недеструктивным по визуальному стилю, но с явным loading-состоянием и toast/alert результатом. Для узлов без `source_url` действие скрывается или отключается с подсказкой.

## Risks / Trade-offs

- [Риск] LLM может добавить неподтверждённые свойства источника. → [Mitigation] Промпт требует опираться только на README/meta/content и формулировать сомнительное как "источник позиционирует".
- [Риск] Digest станет слишком длинным и начнёт конкурировать с полными статьями. → [Mitigation] Ограничить шаблоны профилей и требовать концептуальную выжимку без usage-разделов.
- [Риск] Неправильная классификация `type` ухудшит навигацию. → [Mitigation] Явно разделить правила: длинная статья без копии — `note`; репозиторий/сервис/каталог — `link`; полная копия — `article`; новость — `note` с `brief_digest`.
- [Риск] Индекс станет больше из-за chunk indexing link/note. → [Mitigation] Чанковать только digest body выше минимального размера и переиспользовать существующий chunker.
- [Риск] Старые узлы останутся бедными для RAG. → [Mitigation] Runtime совместимость в этом change, точечное действие «Обновить описание из источника» для ручного улучшения существующих узлов.
- [Риск] Пользователь случайно перетрёт вручную написанное тело заметки. → [Mitigation] Действие доступно как явная команда, а backend должен обновлять только поля описания/digest по правилам endpoint; для узлов без `source_url` действие недоступно.

## Migration Plan

Существующая база не мигрируется автоматически. Пользователь открывает конкретный узел и запускает «Обновить описание из источника». После успешного обновления узел сохраняется в markdown, индекс получает событие переиндексации, а незакоммиченные изменения видны через существующий git-status/save flow.

Rollback выполняется штатно через git: пользователь может посмотреть diff и откатить изменения в базе знаний.
