## ADDED Requirements

### Requirement: Классификация профиля внешнего источника

Система ДОЛЖНА (SHALL) классифицировать внешний URL по природе источника перед созданием узла с digest. Классификация MUST поддерживать значения `repository`, `documentation`, `product_service`, `online_tool`, `directory_catalog`, `learning_resource`, `article`, `news`, `social_post`, `unknown`. Классификация MUST использовать URL, домен, метаданные страницы, README или preview извлечённого контента.

#### Scenario: Репозиторий классифицируется как repository

- **WHEN** пользователь сохраняет URL вида `https://github.com/{owner}/{repo}`
- **THEN** система классифицирует источник как `source_kind=repository`

#### Scenario: Длинная статья без полного копирования классифицируется как article

- **WHEN** пользователь сохраняет URL на длинную статью и не запрашивает полную локальную копию
- **THEN** система классифицирует источник как `source_kind=article` и выбирает digest-профиль вместо полного копирования

#### Scenario: Новость классифицируется как news

- **WHEN** пользователь сохраняет URL новостной публикации о релизе продукта или модели
- **THEN** система классифицирует источник как `source_kind=news`

### Requirement: Выбор локальной формы знания для digest

Система ДОЛЖНА (SHALL) выбирать `type` узла на основе локальной формы знания, а не только природы внешнего URL. Репозитории, сервисы, инструменты, каталоги и документация MUST сохраняться как `type=link` с профильным digest-телом. Длинные статьи без полного копирования, новости и социальные публикации MUST сохраняться как `type=note` с digest-телом. Полная локальная копия источника MUST сохраняться как `type=article`.

#### Scenario: Репозиторий сохраняется как link с профильным digest

- **WHEN** пользователь сохраняет GitHub-репозиторий библиотеки
- **THEN** создаётся узел `type=link`, `source_kind=repository`, `content_profile=repository_profile` с markdown-телом digest

#### Scenario: Длинная статья сохраняется как conceptual note

- **WHEN** пользователь сохраняет длинную статью в режиме концептуального описания
- **THEN** создаётся узел `type=note`, `source_kind=article`, `content_profile=conceptual_digest` с markdown-телом digest

#### Scenario: Новость сохраняется как brief note

- **WHEN** пользователь сохраняет новость о релизе модели или продукта
- **THEN** создаётся узел `type=note`, `source_kind=news`, `content_profile=brief_digest` с кратким markdown-телом digest

#### Scenario: Полная копия статьи остаётся article

- **WHEN** пользователь явно запрашивает сохранить полный текст статьи
- **THEN** создаётся узел `type=article` с полным markdown-контентом источника

### Requirement: Структура концептуального digest

Система ДОЛЖНА (SHALL) генерировать markdown-тело digest, пригодное для чтения человеком и RAG retrieval. Digest MUST быть структурирован заголовками второго уровня и MUST описывать концептуальные особенности источника, а не копировать технические инструкции.

Для `repository_profile` digest MUST включать назначение, ключевую идею, основные абстракции, архитектурные свойства, сценарии применимости, ограничения и связанные понятия.

Для `product_profile` digest MUST включать какую работу закрывает продукт или сервис, целевую аудиторию, основной workflow, данные или интеграции, ограничения и связанные понятия.

Для `documentation_profile` digest MUST включать ментальную модель документации, основные сущности, типовые сценарии, границы применимости и связанные понятия.

Для `online_tool_profile` digest MUST включать входные и выходные данные инструмента, решаемую задачу, ограничения, свойства приватности/локальности и когда инструмент полезнее локальной альтернативы.

Для `directory_profile` digest MUST включать область каталога, принцип организации, классы ресурсов, которые он помогает находить, и связанные понятия.

Для `learning_resource_profile` digest MUST включать чему учит ресурс, структуру пути, prerequisites, практический результат и связанные понятия.

Для `conceptual_digest` digest MUST включать главную идею, ключевые принципы, практические эвристики, антипаттерны или ограничения и связанные понятия.

Для `brief_digest` digest MUST включать суть новости, технически важные изменения, возможную значимость, ограничения информации и связанные понятия.

#### Scenario: README превращается в repository profile

- **WHEN** система генерирует digest для репозитория по README
- **THEN** markdown-тело содержит разделы про назначение, ключевую идею, абстракции, архитектурные свойства, применимость, ограничения и связанные понятия

#### Scenario: Длинная статья превращается в conceptual digest

- **WHEN** система генерирует digest для длинной статьи без полного копирования
- **THEN** markdown-тело содержит концептуальную выжимку без полного текста статьи

#### Scenario: Сервис превращается в product profile

- **WHEN** система генерирует digest для продуктовой страницы или SaaS-сервиса
- **THEN** markdown-тело описывает работу сервиса, целевую аудиторию, workflow, интеграции, ограничения и связанные понятия

#### Scenario: Каталог превращается в directory profile

- **WHEN** система генерирует digest для каталога ресурсов
- **THEN** markdown-тело описывает область каталога, принцип организации и классы ресурсов, которые он помогает находить

#### Scenario: Новость превращается в brief digest

- **WHEN** система генерирует digest для новости
- **THEN** markdown-тело кратко фиксирует суть, значимость и ограничения информации

### Requirement: Фильтрация технического шума

Система ДОЛЖНА (SHALL) исключать из digest технический шум, если он не нужен для концептуального понимания источника. Digest MUST NOT переносить инструкции установки, quick start, команды запуска, длинные примеры кода, API reference, badge/changelog/license/sponsor/contributing секции и benchmark-таблицы без концептуального вывода.

#### Scenario: README с installation и quick start

- **WHEN** README содержит разделы installation, quick start и длинные usage-примеры
- **THEN** digest сохраняет концептуальные свойства библиотеки и не копирует команды установки или usage-примеры

#### Scenario: Benchmark содержит только численные таблицы

- **WHEN** источник содержит benchmark-таблицы без объяснения архитектурного компромисса
- **THEN** digest не включает таблицы и MAY упомянуть только подтверждённый концептуальный вывод

### Requirement: Основанность digest на источнике

Система ДОЛЖНА (SHALL) генерировать digest только на основе доступных фактов из README, метаданных, preview или извлечённого контента. Digest MUST NOT добавлять неподтверждённые свойства. При неуверенности формулировка MUST указывать, что источник позиционирует проект определённым образом.

#### Scenario: Источник не подтверждает архитектурное свойство

- **WHEN** README не описывает concurrency model библиотеки
- **THEN** digest не утверждает наличие конкретной concurrency model

#### Scenario: Маркетинговое описание источника

- **WHEN** источник содержит маркетинговые заявления без технических деталей
- **THEN** digest формулирует их как позиционирование источника, а не как проверенный факт
