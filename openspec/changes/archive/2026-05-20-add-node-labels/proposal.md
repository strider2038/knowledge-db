## Why

Пользователю нужна личная разметка узлов (избранное, «перечитать», «важно» и произвольные метки) без смешения с семантическими `keywords` и без влияния на RAG/эмбеддинги. Сейчас есть только boolean `manual_processed`; этого недостаточно для фильтрации иерархии по нескольким произвольным меткам с визуальным различием.

## What Changes

- Добавить опциональное поле `labels` (массив строк) во frontmatter узла — свободные теги, только на узлах (не на темах).
- API: возвращать `labels` в списке и деталях узла; `PATCH` для добавления/замены меток; фильтр `GET /api/nodes?labels=...` с режимом **AND** (узел должен содержать все указанные метки).
- Индексатор: **не** включать `labels` в текст эмбеддингов и в полнотекстовый поиск по смыслу; допустимо хранить в SQLite только для UI-фильтра (как `manual_processed`).
- Web UI: редактирование меток на странице узла; фильтр и цветные чипы на «Обзоре»; сужение дерева тем при активном фильтре по меткам (по всей базе, как для type/manual_processed).
- Цвет метки в UI: стабильный выбор из общей палитры по hash(id метки), без отдельного конфиг-файла в базе.
- Область: только веб-UI пользователя; Telegram-бот и MCP в этой итерации не расширяются.

### Modified Capabilities

- `knowledge-storage`: формат frontmatter, валидация и нормализация `labels`.
- `rest-api`: список/фильтр/PATCH меток узла.
- `overview-browse`: фильтр по меткам, отображение в таблице, фильтрация дерева.
- `webapp`: редактирование меток на странице узла, стили чипов по палитре.
- `embedding-index`: явное исключение `labels` из индексируемого контента.

## Impact

- `internal/kb` — чтение/запись/фильтрация `labels`, нормализация.
- `internal/api` — query `labels`, расширение PATCH.
- `internal/index` — колонка/фильтр опционально; исключение из embedding text.
- `web/src` — OverviewPage, NodePage, api client, `lib/label-styles.ts` (или аналог).
