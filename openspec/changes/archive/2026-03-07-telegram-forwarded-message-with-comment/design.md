## Context

Текущий бот (`internal/telegram/bot.go`) обрабатывает каждый update независимо: извлекает `message.text` или `message.caption` и передаёт в `IngestText`. Структура `update` не содержит `reply_to_message` — при пересылке с комментарием Telegram присылает два update: сначала пересланное сообщение, затем комментарий (reply). Оба обрабатываются по отдельности, контекст теряется.

Telegram Bot API: при reply сообщение содержит `reply_to_message` с полным объектом исходного сообщения (включая `forward_origin`, `text`, `caption`). То есть во втором update уже есть оба фрагмента — комментарий в `message.text`, пересланный контент в `message.reply_to_message`.

## Goals / Non-Goals

**Goals:**
- Обрабатывать пару (пересланное сообщение + комментарий) как единый input
- Поддерживать оба порядка прихода: forward→reply и reply (с reply_to_message)
- Один цикл подтверждения и итогового ответа на пару

**Non-Goals:**
- Объединение произвольных последовательных сообщений без reply (только forward+reply)
- Изменение IngestText или ingestion pipeline

## Decisions

### 1. Использовать reply_to_message при наличии

**Решение:** Если `message.reply_to_message` не nil, извлекать текст из обоих сообщений и объединять.

**Альтернатива:** Буферизация всех сообщений по времени. Отклонена — сложнее, выше риск ложных срабатываний, reply_to_message даёт явную связь.

### 2. Буферизация пересланных сообщений без reply

**Проблема:** Первый update — пересланное сообщение без reply. Второй — комментарий с reply_to_message. Если обработать первый сразу, получим два ingestion.

**Решение:** При получении пересланного сообщения (forward_origin — текущий формат Telegram Bot API) без reply_to_message — откладывать обработку на 3 с. Если в течение окна приходит reply с reply_to_message, указывающим на это сообщение, объединять и обрабатывать. Иначе — обрабатывать пересланное как одиночное. Окно — константа, не конфиг.

**Альтернатива:** Не буферизовать, всегда обрабатывать forward сразу. Отклонена — тогда комментарий придёт вторым update и будет обработан отдельно (дублирование/потеря контекста).

### 3. Формат объединённого текста

**Решение:** Явные метки для LLM:
```
Инструкции пользователя: <comment>
Пересланное сообщение: <forwarded_content>
```
Комментарий (инструкция, теги) идёт первым — LLM в pipeline лучше интерпретирует его как контекст для контента.

**Альтернатива:** `forwarded_content + "\n\n" + comment`. Отклонена — инструкция должна быть в начале.

### 4. Расширение структуры update

**Решение:** Добавить в `update.message` поля `ReplyToMessage` и `ForwardOrigin` (json.RawMessage — текущий формат Telegram Bot API), чтобы различать forward и reply.

### 5. Логирование объединений

**Решение:** При объединении forward+reply логировать на уровне info (chat_id, message_id) для отладки.

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| Буфер: ложное ожидание — пользователь переслал без комментария | Таймаут 3 с, после — обрабатывать как одиночное |
| Буфер: память при всплеске | Ограничить размер буфера (например, 10 записей на chat), TTL |
| reply_to_message на пересланное из канала — другая структура | Проверять forward_origin, извлекать caption/text по аналогии с message |
| Два update с одинаковым reply_to_message (редко) | Обрабатывать только первый reply, остальные игнорировать или дедуплицировать по message_id |
