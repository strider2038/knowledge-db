## Context

Нормализация узла выполняется асинхронно и может занимать заметное время. Пользователю нужна наблюдаемость процесса: что агент делает сейчас, есть ли ошибки и на каком шаге (normalize/sync) они возникли.

## Goals / Non-Goals

**Goals:**
- Показать пользователю живой лог операции нормализации.
- Сохранить текущую асинхронную job-модель (start + polling status).
- Добавить инкрементальный API логов без websocket/SSE как MVP.
- Встроить отображение в отдельную нижнюю collapsible панель страницы узла.

**Non-Goals:**
- Постоянное хранение логов в БД.
- Глобальный лог-агрегатор по всем операциям.
- Полноценный терминальный UI с интерактивным вводом.

## Decisions

1. Буфер логов в памяти на каждую операцию
- Храним список строк лога с монотонным offset и метаданными `stream` (`stdout`, `stderr`, `system`) и `timestamp`.
- Причина: просто и достаточно для UI-наблюдаемости.

2. Инкрементальный REST endpoint логов
- Добавляем `GET /api/node-normalization/{id}/logs?after=<offset>`.
- Endpoint возвращает `next_offset` и только новые записи.
- Причина: минимальная сложность по сравнению с SSE/WebSocket.

3. Захват stdout/stderr через `os/exec`
- Используем `StdoutPipe` и `StderrPipe`, читаем построчно в горутинах.
- Причина: реальный прогресс доступен только через вывод дочернего процесса.

4. UI-панель внизу страницы узла
- Фиксированная нижняя панель с режимами collapsed/expanded.
- При активной операции лог и статус обновляются polling-ом.

## Risks / Trade-offs

- [Риск] Много строк лога в памяти → ограничить буфер (ring buffer, например 1000 записей).
- [Риск] Потеря логов после рестарта сервера → допустимо для MVP (in-memory).
- [Trade-off] polling вместо stream push → проще реализация, чуть выше задержка обновления.
