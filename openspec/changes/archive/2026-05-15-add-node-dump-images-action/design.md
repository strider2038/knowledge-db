## Context

В проекте уже реализована асинхронная операция нормализации узла с наблюдаемым логом и post-step `sync`. Для `dump images` нужно обеспечить ту же операционную модель в UI и API, чтобы пользователь видел выполнение и ошибки без ручной диагностики.

Изменение кросс-модульное: web UI, REST API и серверная оркестрация фоновой операции по path узла.

## Goals / Non-Goals

**Goals:**
- Добавить в UI кнопку «Выгрузить изображения» в панели действий статьи.
- Сделать запуск асинхронным с status polling и запретом повторного старта во время выполнения.
- Использовать такой же формат логов, как у нормализации: `stdout/stderr/system`, `offset`, timestamp.
- Автоматически запускать `sync` после успешного `dump images`.

**Non-Goals:**
- Массовый запуск `dump images` для всей базы.
- Полный рефакторинг общей системы фоновых задач.
- Изменение самой доменной логики извлечения изображений вне нужного orchestration-контракта.

## Decisions

1. Единая модель операции с нормализацией
- Решение: `dump images` использует такой же жизненный цикл операции (running/success/error + стадии) и такой же протокол чтения логов.
- Почему: повторное использование уже понятного UX и контрактов, снижение когнитивной нагрузки.
- Альтернатива: отдельный упрощённый endpoint без логов. Отклонено из-за потери наблюдаемости.

2. Асинхронный запуск + отдельный endpoint логов
- Решение: запуск операции возвращает operation id; UI опрашивает статус и логи инкрементально через `after` offset.
- Почему: операция может быть длительной и требовать потокового фидбэка.
- Альтернатива: долгий синхронный HTTP-запрос. Отклонено из-за таймаутов и плохого UX.

3. post-step sync только после успешного dump
- Решение: `sync` выполняется только если основной шаг завершился success.
- Почему: исключаем синхронизацию частичных/ошибочных изменений.
- Альтернатива: всегда делать `sync`. Отклонено как риск фиксации невалидного состояния.

## Risks / Trade-offs

- [Риск] Конкурентный запуск нескольких dump для одного path → Митигация: блокировка повторного старта для активной операции этого path.
- [Риск] Успешный dump и неуспешный sync → Митигация: отдельный статус шага sync и явная ошибка в результате операции.
- [Trade-off] Дополнительные endpoint-ы и UI-состояния усложняют код → компенсируется предсказуемостью и диагностируемостью операции.
