## ADDED Requirements

### Requirement: Единый гибридный retrieval pipeline

Система ДОЛЖНА (SHALL) предоставлять единый retrieval pipeline для поиска и RAG-чата. Pipeline MUST принимать текстовый запрос, фильтры по типу/теме/ручной проверке, лимиты результатов и режим использования (`search` или `chat`). Pipeline SHALL объединять кандидатов из keyword/FTS поиска, точных совпадений и semantic vector search в единый ранжированный список.

#### Scenario: Гибридный поиск объединяет источники кандидатов

- **WHEN** пользователь ищет фразу, которая одновременно совпадает с keyword одной ноды и семантически близка к chunk другой ноды
- **THEN** pipeline возвращает обе ноды в едином списке результатов с указанием причин совпадения

#### Scenario: Фильтры применяются до финальной выдачи

- **WHEN** pipeline вызывается с фильтром `type=article`
- **THEN** итоговая выдача содержит только article-ноды и их фрагменты

### Requirement: Fusion и ранжирование

Система ДОЛЖНА (SHALL) объединять разнородные результаты ранжирования через устойчивый fusion-алгоритм. Fusion MUST учитывать rank из каждого источника кандидатов и SHOULD быть реализован без предположения, что cosine similarity и keyword/FTS score находятся в одной шкале. Точные совпадения по `title`, `path`, `aliases` и `keywords` MUST получать повышенный вклад в итоговый rank.

#### Scenario: Точное совпадение ключевика поднимает результат

- **WHEN** запрос точно совпадает с keyword ноды, но её vector score ниже соседних результатов
- **THEN** нода остаётся в верхней части выдачи благодаря exact/keyword boost

#### Scenario: Semantic-only результат не теряется

- **WHEN** запрос не содержит точных слов из ноды, но semantic vector search находит близкую ноду
- **THEN** нода присутствует в выдаче с причиной совпадения `vector`

### Requirement: Структура результата гибридного поиска

Система ДОЛЖНА (SHALL) возвращать результаты гибридного поиска как карточки нод с объяснимой релевантностью. Каждый результат MUST содержать `path`, `title`, `type`, `annotation`, `keywords`, `score`, `rank`, `match_reasons` и MAY содержать `source_url`. Результат для article-ноды SHOULD включать массив `fragments` с `heading`, `snippet`, `score` и типом совпадения.

#### Scenario: Результат содержит причины совпадения

- **WHEN** нода найдена по keyword и vector search
- **THEN** результат содержит `match_reasons`, отражающие оба источника совпадения

#### Scenario: Статья содержит релевантный фрагмент

- **WHEN** найден chunk внутри article-ноды
- **THEN** результат article-ноды содержит fragment со snippet и heading найденного chunk

### Requirement: Порог релевантности для chat mode

Система ДОЛЖНА (SHALL) поддерживать relevance cutoff для режима `chat`. В режиме `chat` pipeline MUST исключать vector-only результаты ниже минимального порога релевантности, чтобы RAG-чат не отвечал на нерелевантном контексте. Exact/keyword совпадения MAY проходить cutoff по отдельному правилу, даже если vector score отсутствует.

#### Scenario: Нерелевантный вопрос не получает случайный top-K

- **WHEN** пользователь задаёт вопрос, не связанный с базой знаний, и pipeline находит только слабые vector-only совпадения
- **THEN** pipeline возвращает пустой или недостаточный контекст для чата

#### Scenario: Точный термин проходит cutoff

- **WHEN** пользователь ищет редкое название библиотеки, точно присутствующее в keywords
- **THEN** результат проходит cutoff даже при низкой semantic близости
