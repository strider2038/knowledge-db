## Context

Текущее состояние:
- Форма «Добавить» в web: textarea + кнопка «Добавить», вызов `ingestText(text)`.
- API POST /api/ingest принимает `{ text, source_url?, source_author? }`.
- IngestRequest: Text, SourceURL, SourceAuthor. ProcessInput — те же поля плюс ExistingThemes, ExistingKeywords.
- LLM-оркестратор сам определяет тип (article/link/note) по тексту. Подсказки типа нет.
- При отправке: `loading` блокирует textarea и кнопку, текст кнопки «Отправка...», но нет анимации и явного индикатора прогресса.

Ограничения: оффлайн-first, git-first; LLM может работать 10–60+ секунд; UI должен оставаться отзывчивым.

## Goals / Non-Goals

**Goals:**
- Пользователь может указать тип контента (авто/статья/ссылка/заметка) — подсказка передаётся в оркестратор.
- При долгой обработке: анимация загрузки, блокировка повторной отправки.
- Минимальные изменения в API и pipeline.

**Non-Goals:**
- Streaming прогресса (этапы LLM/fetch/git) — не в scope.
- Отмена запроса в процессе — не требуется на первом этапе.
- Изменение логики Telegram-бота — type_hint остаётся опциональным, бот не передаёт его.

## Decisions

### 1. Передача type_hint через цепочку

**Решение:** Добавить опциональное поле `TypeHint string` в IngestRequest и ProcessInput. Значения: `""` (auto), `"article"`, `"link"`, `"note"`. API принимает `type_hint` (snake_case). При пустом или `"auto"` оркестратор ведёт себя как сейчас.

**Альтернативы:** Отдельный query-параметр — менее RESTful для POST с телом. Enum в отдельном типе — избыточно для 4 значений.

### 2. Учёт type_hint в оркестраторе

**Решение:** При `TypeHint != "" && TypeHint != "auto"` добавлять в system prompt (или в начало user message) явную инструкцию: «Пользователь указал тип: <type>. Используй именно этот тип при вызове create_node.» Это гарантирует, что LLM выберет указанный тип.

**Альтернативы:** Модифицировать только system prompt — достаточно. Передавать в отдельном «мета-сообщении» — усложняет формат. Жёстко переопределять результат create_node на бекенде — нарушает семантику LLM, может конфликтовать с fetch_url_content/fetch_url_meta.

### 3. UX при долгой обработке

**Решение:**
- Состояние `loading`: блокировать textarea и кнопку, показывать спиннер рядом с кнопкой или вместо текста кнопки.
- Текст кнопки: «Добавить» → «Обработка...» (или «Отправка...» — оставить текущий).
- Использовать существующий компонент Spinner/Loader из UI-библиотеки (если есть) или простой CSS-анимацию.
- Не разблокировать форму до завершения запроса (успех или ошибка).

**Альтернативы:** Skeleton вместо формы — слишком радикально. Прогресс-бар — нет данных о прогрессе с бекенда. Только disable кнопки без анимации — хуже восприятие.

### 4. Валидация type_hint

**Решение:** API принимает `type_hint` как строку. Допустимые значения: `"auto"`, `"article"`, `"link"`, `"note"`. При неизвестном значении — игнорировать (трактовать как auto). Не возвращать 400 — сохраняем обратную совместимость.

### 5. Подсказка при выборе «статья»/«ссылка»

**Решение:** При выборе типа «статья» или «ссылка» под полем переключателя MUST отображаться текст: «Вставьте URL в текст». Это снижает риск некорректного результата, когда пользователь забывает добавить ссылку.

### 6. Сигнализация успеха и ошибки

**Решение:** Использовать **вариант B** — inline Alert/Banner под формой.

- **Успех:** блок с зелёным фоном, иконка ✓, текст «Добавлено» и ссылка «Перейти к узлу» (если API вернул path). Ссылка ведёт на `/node/{path}`.
- **Ошибка:** блок с красным/деструктивным фоном, иконка ✗, текст ошибки из API.
- Сообщение сбрасывается при следующей отправке (как сейчас).

**Альтернативы:**
- **A. Текущий подход** (простой текст): уже есть. Минус — легко пропустить, нет ссылки на узел.
- **B. Inline Alert/Banner** ✓ — заметно, без новых зависимостей, можно добавить ссылку на узел.
- **C. Toast** (sonner, react-hot-toast): всплывает в углу, автозакрытие. Плюс — не занимает место. Минус — новая зависимость, toast может исчезнуть до того, как пользователь прочитает.
- **D. Редирект на узел при успехе:** сразу переход на `/node/{path}`. Плюс — мгновенный feedback. Минус — пользователь теряет контекст формы, не может быстро добавить ещё.

## Risks / Trade-offs

- **[Risk]** Пользователь выберет «статья», но вставит заметку без URL → LLM может создать некорректный узел.  
  **Mitigation:** В инструкции указать: при type_hint=article/link ожидается URL в тексте; при отсутствии — LLM может вернуть ошибку или интерпретировать по контексту. Документировать в UI подсказкой.

- **[Trade-off]** Нет streaming — пользователь ждёт без обратной связи.  
  **Mitigation:** Явный спиннер и блокировка снижают неопределённость. Streaming — отдельная задача.

## Migration Plan

- Изменения обратно совместимы: `type_hint` опционален, при отсутствии поведение как сейчас.
- Деплой: сборка web, перезапуск kb-server. Rollback — откат бинарника и статики.
