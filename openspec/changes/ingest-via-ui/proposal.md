## Why

Сейчас ingest доступен через Telegram-бот и API, но веб-интерфейс имеет минимальную форму: одно текстовое поле и кнопка. Пользователь не может указать тип контента (статья, ссылка, заметка), а LLM определяет его автоматически. При длительной обработке (LLM, fetch URL, git commit) нет явной индикации прогресса — пользователь не понимает, идёт ли процесс или произошла ошибка.

Цель: сделать ingest через UI полноценным и удобным — с возможностью подсказки типа и понятным UX при долгой обработке.

## What Changes

- **Переключатель типа контента** в форме «Добавить»: «авто» (по умолчанию), «статья», «ссылка», «заметка». При выборе не-«авто» значение передаётся на бекенд, оркестратор использует его как подсказку при выборе типа узла.
- **Расширение API** POST /api/ingest: опциональное поле `type_hint` (auto | article | link | note).
- **Расширение ingestion pipeline**: IngestRequest и ProcessInput получают опциональное поле TypeHint; оркестратор при наличии подсказки учитывает её в инструкциях для LLM.
- **UX при отправке**: анимация загрузки (спиннер/индикатор), блокировка кнопки «Добавить» и поля ввода на время обработки, чтобы исключить повторную отправку.

## Capabilities

### New Capabilities

- `ingest-type-hint`: поддержка подсказки типа контента (type_hint) в API, pipeline и оркестраторе — от UI до LLM.

### Modified Capabilities

- `webapp`: расширение раздела «Добавить» — переключатель типа, улучшенный UX при долгой обработке (анимация, блокировка формы).
- `rest-api`: расширение эндпоинта POST /api/ingest — опциональное поле `type_hint`.
- `ingestion-pipeline`: IngestRequest и ProcessInput с опциональным TypeHint; оркестратор передаёт подсказку в LLM.

## Impact

- **internal/ingestion**: IngestRequest, pipeline (buildProcessInput), stub
- **internal/ingestion/llm**: ProcessInput, orchestrator, prompt (учёт type_hint)
- **internal/api**: handler Ingest — парсинг type_hint
- **web/src**: AddPage, api.ts — переключатель типа, отправка type_hint, UX при загрузке
