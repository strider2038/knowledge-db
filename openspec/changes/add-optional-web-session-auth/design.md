## Context

Сейчас `kb-server` обслуживает web UI, REST API и `/api/mcp` без аутентификации. Для локального режима это удобно, но при публикации в вебе создаёт риск несанкционированного доступа к данным и операциям записи.

Из proposal следует требование: авторизация должна быть **опциональной** и включаться только при наличии `KB_LOGIN` и `KB_PASSWORD` в окружении, без ухудшения текущего поведения по умолчанию.

Ограничения:
- Не усложнять локальный/dev сценарий, когда auth выключена.
- Сохранить текущий стек (Go stdlib + существующая архитектура `internal/bootstrap` и `internal/api`).
- Обеспечить разумную безопасность для публичного веб-доступа без избыточной инфраструктуры.

## Goals / Non-Goals

**Goals:**
- Ввести опциональную сессионную авторизацию для web UI и HTTP API.
- Добавить login/logout/session-check API и cookie-сессию с серверной валидацией.
- При включённой авторизации защищать API, включая `/api/mcp`.
- Добавить в web UI login flow (экран входа, проверка сессии, редиректы).
- Сохранить полную обратную совместимость при unset `KB_LOGIN`/`KB_PASSWORD`.

**Non-Goals:**
- Мульти-пользовательская модель, роли и ACL.
- OAuth/OIDC и внешние identity providers.
- Персистентное/распределённое хранилище сессий (Redis, DB) в первой итерации.
- Полноценный IAM для Telegram-бота и CLI.

## Decisions

### 1) Режим включения авторизации через env-флаги

**Решение:** auth-режим считается включённым только при одновременном наличии непустых `KB_LOGIN` и `KB_PASSWORD`.  
**Почему:** минимальный вход в эксплуатацию и прозрачный UX для self-hosting; отсутствие переменных сохраняет текущий open-режим.

**Альтернатива:** отдельный флаг `KB_AUTH_ENABLED=true`.  
**Почему не выбрано:** выше риск неконсистентной конфигурации (флаг включён, но креды не заданы).

### 2) Серверные cookie-сессии вместо Basic/Bearer

**Решение:** использовать stateful server-side сессии с cookie (`HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`, TTL). Для TTL добавить настройку `KB_SESSION_TTL` с дефолтом `8h`.  
**Почему:** подходит для browser login form, не хранит секреты в JS, поддерживает logout/revoke и не требует от фронта ручной работы с токенами.

**Альтернатива A:** HTTP Basic.  
**Почему не выбрано:** неудобный UX и слабая пригодность для публичного веб-продукта.

**Альтернатива B:** JWT/Bearer в `localStorage`.  
**Почему не выбрано:** выше XSS-риск, более сложная ротация/отзыв.

### 3) In-memory session store на первом этапе

**Решение:** хранить сессии в памяти процесса (`map + mutex`) с `sessionID` как случайным криптостойким идентификатором и сроком жизни.  
**Почему:** просто внедряется в текущую архитектуру, без внешних зависимостей.

**Альтернатива:** Redis/DB.  
**Почему не выбрано сейчас:** лишняя операционная сложность для первой версии.

### 4) Единый auth middleware на HTTP-уровне

**Решение:** добавить middleware в цепочку сервера, который:
- пропускает open endpoints (`/api/auth/login`, `/api/auth/session`, `/api/auth/logout`);
- для защищённых маршрутов проверяет валидность сессии;
- при невалидной/отсутствующей сессии возвращает `401`.

При включённой авторизации middleware применяется к:
- `/api/*`, включая `/api/mcp` и доступ к данным через `/api/assets/*`;
- не применяется к раздаче SPA-статики (`/`, `/index.html`, assets web bundle).

**Почему:** данные и операции защищаются на backend-уровне, при этом публичность клиентского HTML/JS не считается риском утечки знаний.

**Альтернатива:** точечные проверки в каждом handler.  
**Почему не выбрано:** высокий риск пропусков и дублирования логики.

### 5) Минимальный контракт auth API

**Решение:** ввести три endpoint:
- `POST /api/auth/login` — проверка логина/пароля, выдача cookie-сессии;
- `POST /api/auth/logout` — инвалидирует текущую сессию и очищает cookie;
- `GET /api/auth/session` — возвращает статус сессии (`authenticated: true/false`).

**Почему:** достаточно для web guard и прозрачного client flow.

### 6) Frontend guard и dedicated login page

**Решение:** добавить страницу `/login`, инициализационный session-check и редирект на `/login` при `401` в защищённом режиме. После успешного login — возврат на исходный маршрут.

**Почему:** предсказуемый UX и минимальные изменения структуры web-приложения.

### 7) Базовые меры защиты от brute force и CSRF

**Решение:**
- ограничение частоты login-попыток (простое rate limiting по IP/окну времени);
- для state-changing запросов в auth-режиме — проверка `Origin` (и/или `Referer`) на ожидаемый host;
- сравнение credentials через constant-time операции.

**Почему:** заметно повышает безопасность при умеренной сложности.

### 8) Исключения для health/readiness из auth middleware

**Решение:** предусмотреть allowlist для health/readiness endpoint (например, `/healthz`, `/readyz`) и не требовать сессию на этих маршрутах при включённой авторизации.

**Почему:** инфраструктурные проверки оркестраторов и балансировщиков должны работать независимо от пользовательской сессии.

## Risks / Trade-offs

- **[Потеря сессий после рестарта процесса]** → В первой версии принимается как компромисс; документируется для операторов.
- **[Один инстанс, нет shared-session между репликами]** → На первом этапе поддерживается single-instance deployment; для HA отдельный follow-up с Redis.
- **[Жёсткая привязка к cookie-модели в браузере]** → Для машинных клиентов можно оставить отдельный future-path (service tokens), но не в scope текущего change.
- **[Ошибки прокси/TLS-конфигурации могут ослабить защиту cookie]** → Добавить эксплуатационные рекомендации (TLS обязателен для внешнего доступа, корректные `X-Forwarded-*` настройки при reverse proxy).

## Migration Plan

1. Добавить новые env-поля в конфиг и включение auth-режима по `KB_LOGIN`/`KB_PASSWORD` + `KB_SESSION_TTL` (по умолчанию `8h`).
2. Реализовать backend auth-компоненты: session store, auth middleware, auth handlers.
3. Интегрировать middleware в bootstrap-цепочку и защитить API-эндпоинты (`/api/*`, включая `/api/mcp` и `/api/assets/*`), оставив SPA-статику открытой.
4. Добавить web login flow (страница `/login`, guard, logout, обработка 401 от API).
5. Обновить документацию (`README`, `.env.example`) с примерами запуска в open и auth режимах.
6. Добавить тесты backend/web для сценариев включённой/выключенной авторизации.
7. Rollback: unset `KB_LOGIN`/`KB_PASSWORD` и перезапуск сервера возвращает open-режим.
