## Why

Проект планируется к публикации в вебе, где текущий полностью открытый доступ к UI и API создаёт риск несанкционированного чтения и изменения данных. Нужна простая и достаточно безопасная опциональная авторизация, которая включается только при заданных `KB_LOGIN` и `KB_PASSWORD`, без усложнения локального сценария.

## What Changes

- Добавить опциональный режим авторизации для `kb-server`, активируемый только при наличии `KB_LOGIN` и `KB_PASSWORD` в окружении.
- Добавить HTTP API для сессионной авторизации (login/logout/session-check) на основе cookie-сессий.
- Ограничить доступ к защищённым API-эндпоинтам при включённой авторизации: неавторизованные запросы получают `401`.
- Добавить поддержку login flow в web UI: страница входа, проверка сессии при старте, редирект на вход при отсутствии сессии.
- Сохранить обратную совместимость: при отсутствии `KB_LOGIN`/`KB_PASSWORD` приложение работает как сейчас, без auth-барьеров.

## Capabilities

### New Capabilities

- `web-session-auth`: Опциональная авторизация через login form и серверные cookie-сессии для защиты web UI и HTTP API.

### Modified Capabilities

- `rest-api`: Изменяются требования к доступу к эндпоинтам (режим защищённого доступа при включённой авторизации) и добавляются auth-эндпоинты.
- `webapp`: Изменяются требования к поведению UI при отсутствии/наличии сессии (login page, guard, logout).
- `mcp-server`: Изменяются требования к доступу к `/api/mcp` при включённой авторизации (доступ только при валидной сессии).

## Impact

- Backend: `internal/bootstrap/config`, `internal/bootstrap/bootstrap`, `internal/api/*` (роутинг, middleware, auth handlers, cookie/session logic).
- Frontend: `web/src/App.tsx`, `web/src/services/api.ts`, новые/изменённые страницы и компоненты для login/logout и session-check.
- Конфигурация: новые env-переменные `KB_LOGIN`, `KB_PASSWORD` (опционально, включают auth-режим).
- Тесты: новые API-тесты и web-тесты для сценариев входа/выхода/доступа без сессии.
