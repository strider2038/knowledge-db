## Why

Сейчас чат-бот обрабатывает сообщения изолированно и не хранит контекст диалога, из-за чего пользователь теряет непрерывность обсуждения и не может вернуться к прошлым чатам. Нужен лёгкий временный механизм чатов с UX, близким к ChatGPT/OpenWebUI, без усложнения постоянным хранением полной истории.

## What Changes

- Добавить хранение сессий чатов в SQLite с метаданными (id, title, timestamps, состояние).
- Добавить временное хранение контекста диалога в рамках сессии с настраиваемым лимитом.
- Добавить автоматическую суммаризацию старых сообщений при превышении лимита, чтобы сохранять контекст без переполнения prompt.
- Добавить API для списка чатов, просмотра чата, продолжения существующего чата, создания, удаления и переименования чата.
- Добавить автогенерацию названия чата и возможность ручного переименования.
- Обновить web UI до модели «список чатов + активный чат» в стиле привычных LLM-чатов.

## Capabilities

### New Capabilities
- `chat-session-memory`: Управление жизненным циклом временных чатов в SQLite, ограниченный контекст, суммаризация и управление заголовками.

### Modified Capabilities
- `rag-chat`: Изменение поведения чата: ответы должны учитывать историю активной сессии, включая summary-сегменты.
- `rest-api`: Добавление/изменение endpoint'ов для CRUD-операций по чат-сессиям, включая удаление и ручное переименование.
- `webapp`: Изменение UI-потока на многосессионный чат с боковой панелью истории и продолжением диалога.

## Impact

- Backend: `cmd/kb-server`, `internal/api`, `internal/ingestion` (если summary использует существующий pipeline), возможно новый пакет `internal/chat`, `internal/storage/sqlite`.
- Frontend: `web/src` (состояние чатов, роутинг/панели, список и активный диалог).
- API: новые контракты для сессий и сообщений, изменения в существующем chat endpoint.
- Хранилище: SQLite как временное хранилище чатов и служебных summary.
- Тесты: API-тесты для новых endpoint'ов и интеграционные сценарии на лимиты/суммаризацию.
