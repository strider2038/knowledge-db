# Design: Автоматический перевод статей на русский

## Context

Текущий ingestion pipeline: IngestText/IngestURL → LLM (fetch + create_node) → saveNode → git commit. Контент для article берётся из fetch_url_content (кешируется в памяти). Store.CreateNode создаёт плоский файл `{themePath}/{slug}.md`. Оригинал сохраняется первым; перевод — пост-шаг после успешного сохранения.

Ограничения: оффлайн-first (без внешних API перевода), git-friendly формат, совместимость с Obsidian.

## Goals / Non-Goals

**Goals:**
- Автоматический перевод article на русский при сохранении, если контент преимущественно на английском
- Хранение в `{slug}.ru.md` с frontmatter и wikilinks для Obsidian
- Эвристика определения языка без внешних сервисов
- Перевод через тот же LLM, что и ingestion (оффлайн-first при использовании локальной модели)

**Non-Goals:**
- Перевод link и note
- Обновление перевода при изменении оригинала
- Поддержка других целевых языков (пока только ru)
- Внешние API перевода (DeepL, Google)

## Decisions

### 1. Эвристика определения языка

**Решение:** Комбинированная эвристика:
1. Удалить из текста code blocks (содержимое между ` ``` ` и ` ``` `)
2. Подсчитать буквы: кириллица (U+0400–U+04FF) vs латиница (a-z, A-Z)
3. Порог: `cyrillic_ratio = cyrillic / (cyrillic + latin)`; если `cyrillic_ratio < 0.25` → считаем текст на английском
4. Минимальная длина: не переводить, если после удаления code blocks осталось < 200 символов (слишком мало для надёжной оценки)

**Альтернативы:** Word-based (словари the/is vs и/в) — сложнее, больше false positives на смешанном контенте. Только первый абзац — может ошибаться при заголовках на латинице.

### 2. Размещение логики перевода

**Решение:** Отдельный пакет `internal/ingestion/translation` с:
- `Detector` — функция `NeedsTranslation(content string) bool` (эвристика)
- `Translator` — интерфейс `Translate(ctx, content string) (string, error)`; реализация через LLM (отдельный Chat/Completion вызов, без function calling)
- Интеграция в pipeline: после `saveNode` в `PipelineIngester.saveNode` вызывать `maybeTranslateAndSave` при `result.Type == "article"` и `NeedsTranslation(result.Content)`

**Альтернативы:** Встроить в orchestrator как второй tool — увеличивает сложность и риск усечения длинного контента. Отдельный runnable-воркер — избыточно для синхронного flow.

### 3. API Store для перевода

**Решение:** Расширить `Store` методом `CreateTranslationFile(ctx, basePath, themePath, slug, lang string, frontmatter map[string]any, content string) error`. Создаёт файл `{themePath}/{slug}.{lang}.md`. Не использует resolveSlug — файл перевода всегда с суффиксом `.ru`, коллизий нет.

**Альтернативы:** Отдельный `TranslationStore` — дублирование логики записи. Параметр в CreateNode (`TranslationContent`) — смешивает ответственности.

### 4. Вызов LLM для перевода

**Решение:** Responses API (тот же, что и orchestrator), но без tools. Один вызов: Input — user message с контентом для перевода, Instructions — «Переведи следующий markdown-текст на русский. Сохрани структуру: заголовки, списки, code blocks оставь без изменений. Верни только перевод, без пояснений.» Ответ — переведённый текст из output items.

Использовать тот же `openai.Client` и модель, что и для orchestrator. Новый метод в `internal/ingestion/llm` (например `TranslateToRussian`) или в пакете `translation` с зависимостью от LLM-клиента.

**Почему Responses API, а не Chat Completions:** Responses API — более новый подход (2025), Chat Completions — legacy. Проект уже использует Responses API для orchestrator; единый протокол упрощает код и конфигурацию. Для простого single-turn запроса tools не нужны — достаточно Input + Instructions.

**Альтернативы:** Chat Completions — устаревший путь, другой код и клиент. Отдельная модель для перевода — усложняет конфиг; одна модель проще.

### 5. Frontmatter и wikilinks

**Решение:**
- В файле перевода: `translation_of: {slug}`, `lang: ru`, остальные поля дублируются из оригинала (keywords, annotation, title, aliases, created, updated, type, source_url, source_date, source_author)
- В конце тела перевода добавить: `\n\n[[{slug}|Original]]`
- В оригинал: после создания перевода обновить frontmatter — добавить `translations: [{slug}.ru]`; в конце тела добавить `\n\n[[{slug}.ru|Русский перевод]]`

**Почему обновлять оригинал:** Двусторонняя связь улучшает навигацию в Obsidian — из оригинала видно ссылку на перевод, из перевода — на оригинал. Graph view и backlinks работают в обе стороны. Обновление выполняется в том же flow (сразу после записи перевода): прочитать оригинальный файл, добавить поле и wikilink, перезаписать.

**Итог:** В переводе — `translation_of`, `lang`, wikilink на оригинал. В оригинале — `translations`, wikilink на перевод.

### 6. Конфигурация

**Решение:** Env `KB_AUTO_TRANSLATE` (bool, default `true`). При `true` — выполнять пост-шаг перевода. При `false` — отключить. Передавать в PipelineIngester при создании.

**Альтернативы:** Настройка в UI/API — требует персистентного хранилища. Флаг в IngestRequest — гибко, но усложняет API.

### 7. Git commit перевода

**Решение:** После создания `{slug}.ru.md` и обновления оригинала вызывать `GitCommitter.CommitNode` для обоих файлов (или два коммита: оригинал при saveNode, перевод+обновлённый оригинал при переводе — в зависимости от порядка операций). Сообщение: `add: {themePath}/{slug}.ru (translation)`.

### 8. Чанкинг для длинных статей

**Решение:** Включить в первый релиз. Концепция:

1. **Порог:** Если контент после удаления code blocks > 6000 символов (~1500 токенов) — разбивать на чанки.
2. **Разбиение:** Делить по границам абзацев (двойной `\n\n`). Цель чанка — ~4000 символов (~1000 токенов), не резать посередине предложения. Если абзац сам по себе длиннее порога — резать по `\n`.
3. **Перевод:** Каждый чанк переводить отдельным вызовом LLM. Промпт для чанков 2+ добавить контекст: «Это продолжение перевода. Предыдущая часть заканчивалась на: "...". Переведи следующий фрагмент, сохраняя связность.»
4. **Склейка:** Конкатенировать переводы чанков. Проверить, что на стыках нет дублирования (если LLM повторил конец предыдущего — обрезать).
5. **Code blocks:** Не переводить — при разбиении вырезать их во временный буфер, вставлять обратно после склейки по позициям.

**Альтернативы:** Overlap между чанками — усложняет склейку. Фиксированный размер в символах без учёта абзацев — может резать предложения.

### 9. Валидация переводов в kb-cli

**Решение:** Расширить `kb-cli validate` проверкой файлов переводов (`*.ru.md`):

1. **Распознавание:** Файлы вида `{slug}.ru.md` в той же директории, что и `{slug}.md`, считаются переводами.
2. **Frontmatter перевода:** Обязательные поля — `translation_of`, `lang`. `translation_of` MUST совпадать со slug оригинала (без `.ru`).
3. **Ссылки (wikilinks):** Извлечь все `[[target]]` и `[[target|label]]` из тела перевода. Для каждого target проверить:
   - Если target = slug оригинала → должен существовать `{themePath}/{slug}.md`
   - Если target = slug другого узла → должен существовать `{themePath}/{target}.md` или в родительской теме
   - Если target содержит `.ru` (ссылка на другой перевод) → проверить существование `{target}.md`
4. **Оригинал:** Если у узла есть файл перевода, в frontmatter оригинала должно быть поле `translations`, содержащее slug перевода (напр. `{slug}.ru`).
5. **Ошибки:** Нарушения — в общий отчёт validate как ValidationError с путём к файлу перевода.

**Реализация:** Расширить `Store.Validate` или добавить отдельный проход в cmd/kb-cli для файлов `*.ru.md`; парсер wikilinks — regex или простой state machine.

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| Длинный контент усекается LLM | Чанкинг по абзацам (~4k символов), склейка с проверкой дубликатов на стыках |
| Эвристика даёт false positive (русский текст помечен как английский) | Порог 0.25 консервативен; при сомнении не переводим. Можно добавить лог для анализа |
| Эвристика даёт false negative (английский не переведён) | Порог можно понизить до 0.2 в настройках; мониторинг через пользовательский фидбек |
| Дополнительная задержка ingestion | Перевод асинхронно не делаем в MVP; пользователь ждёт. При необходимости — фоновый воркер позже |
| Стоимость токенов LLM | Один дополнительный вызов на статью; при локальной модели — только время |

## Migration Plan

- Изменения обратно совместимы: новые файлы `.ru.md`; оригиналы обновляются только при создании перевода
- Rollback: отключить `KB_AUTO_TRANSLATE`; существующие переводы остаются в базе
- Миграция существующих статей не требуется
