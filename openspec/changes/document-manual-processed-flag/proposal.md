## Why

При массовом добавлении заметок (фоновый ingest, импорт) не всегда есть время сразу проверить корректность сортировки и обработки. Спустя время сложно отличить уже проверенные записи от непроверенных. Нужна явная отметка «проверено вручную» на уровне узла и возможность отфильтровать такие записи в списке.

## What Changes

- Опциональное поле метаданных узла (frontmatter / модель API): флаг «проверено вручную» (булево), по умолчанию при отсутствии — «не проверено».
- REST API: поле в ответах узла; возможность изменить флаг при обновлении узла; query-параметр фильтрации списка `GET /api/nodes` (например: все / только проверенные / только непроверенные).
- Веб UI: переключатель или чекбокс на странице/форме документа; в разделе «Обзор» — фильтр по состоянию проверки, согласованный с API.
- Валидация базы: новое поле опционально, не ломает существующие узлы.

### Modified Capabilities

- `knowledge-storage`: расширить формат frontmatter опциональным булевым полем «ручная проверка обработана»; описать семантику отсутствия значения.
- `rest-api`: отдавать поле в метаданных узла и списке; поддержать обновление; добавить фильтр списка узлов по этому полю.
- `webapp`: отображение и редактирование флага на экране узла; передача фильтра при запросе списка из UI.
- `overview-browse`: фильтр в таблице обзора (в дополнение к типам и поиску), сценарии взаимодействия с API.

## Impact

- Пакеты `internal/kb` (модель узла, валидация, чтение/запись frontmatter), handlers списка и PATCH/PUT узла в `internal/api`.
- Тесты API и при необходимости unit-тесты хранилища.
- `web/`: страница узла, страница обзора, типы API-клиента, query-параметры списка.
- Существующие записи без поля остаются валидными; поведение по умолчанию — «не отмечено как проверенное».
