## Context

Записи живут в markdown с YAML frontmatter; список узлов и карточка узла идут через REST и React-обзор. Сейчас нет явного признака «пользователь вручную проверил сортировку и обработку». Предложение: одно опциональное булево поле в метаданных узла и фильтрация на `GET /api/nodes`.

## Goals / Non-Goals

**Goals:**

- Опциональное поле `manual_processed` (boolean) в frontmatter и в JSON узла: `true` — отмечено пользователем как проверенное; отсутствие или `false` — не отмечено.
- Список узлов: query-параметр с тремя состояниями — не передавать (все узлы), `manual_processed=true` (только отмеченные), `manual_processed=false` (только неотмеченные).
- Обновление флага через тот же механизм, которым веб уже сохраняет метаданные узла (без отдельного «микро-API» только для галочки, если в коде не окажется иного удобного места).
- Веб: чекбокс «Проверено вручную» на странице узла с немедленным или явным сохранением; в обзоре — сегментированный фильтр или select (все / проверено / не проверено), значение в URL query вместе с path, type, q, sort, order, page.

**Non-Goals:**

- Автоматическая простановка флага при ingest или импорте Telegram.
- Отдельный журнал аудита «кто и когда отметил».
- Фильтрация в MCP/Telegram (можно добавить позже).

## Decisions

1. **Имя поля `manual_processed`** — короткое, машиночитаемое, без пробелов; в UI подпись по-русски («Проверено вручную» / «Ручная проверка»). Альтернатива `reviewed` отклонена из-за двусмысленности с редакционной вычиткой.
2. **Семантика по умолчанию** — отсутствие ключа в YAML эквивалентно `false` для фильтра «не проверено» и для отображения чекбокса снятым.
3. **Фильтр списка** — один query-параметр с булевым значением, а не отдельные эндпоинты; совместимо с пагинацией и остальными фильтрами.
4. **Обзор и URL** — добавить `manual_processed` в перечень параметров состояния обзора (как `path`, `type`, …), включая сохранение в `returnTo` при переходе на узел, чтобы возврат «Назад» сохранял фильтр.

## Risks / Trade-offs

- **[Risk] Путаница с «обработано пайплайном»** → в UI явная формулировка «проверено вручную», не «обработано».
- **[Risk] Старые клиенты API** → поле опционально в ответе; клиенты без поддержки игнорируют.
- **[Trade-off] Только bool** — нет даты проверки; при необходимости расширения позже можно добавить опциональное поле времени отдельным change.

## Migration Plan

Новые поля опциональны: существующие файлы без изменений. После деплоя пользователь постепенно отмечает узлы. Откат — удалить поле из кода; данные в git останутся как необрабатываемые ключи YAML (безопасно).

## Примечание для реализации

Здесь не задаётся отдельный вопрос автору change. Имеется в виду только то, что **конкретный путь в коде** (какой handler и тело запроса уже обновляют frontmatter) выбирает тот, кто пишет реализацию: нужно добавить поле `manual_processed` в ту же модель и тот же save-flow, что и для остальных редактируемых метаданных узла. Это не требует вашего ответа заранее.
