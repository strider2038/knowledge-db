## Why

Сейчас web-доступ в закрытом режиме опирается на пару логин/пароль в переменных окружения. Для личной базы знаний удобнее и безопаснее вход через Google-аккаунт с явным списком разрешённых адресов (allowlist), без хранения общего пароля в конфиге. Это снижает риск утечки пароля и упрощает идентификацию пользователя.

## What Changes

- Подключение OAuth 2.0 Authorization Code flow к Google: эндпоинты старта и callback на бэкенде, обмен `code` на токены, получение email пользователя.
- Проверка email пользователя по allowlist из переменной окружения (список разрешённых адресов); при несовпадении — отказ в входе без создания сессии.
- Сохранение существующей модели серверных cookie-сессий (`kb_session`) после успешной проверки: OAuth только устанавливает сессию, защищённые маршруты остаются как сейчас.
- Web UI: кнопка «Войти через Google» и обработка редиректа после callback (в дополнение или вместо формы логин/пароль в зависимости от конфигурации — см. design).
- Документирование переменных окружения для Google OAuth, redirect URI и allowlist.

### Modified Capabilities

- `web-session-auth`: расширение условий включения авторизации и сценариев входа (режим Google OAuth + allowlist email, взаимосвязь с существующим логином/паролем).
- `rest-api`: новые/изменённые маршруты для OAuth (start, callback) и требования к `GET /api/auth/session` при OAuth-режиме.
- `webapp`: экран входа с сценарием «Войти через Google» и редиректы после OAuth.

## Impact

- **serverapp (Go):** `internal/api` (новые handlers), `internal/bootstrap/config` (новые env), возможное расширение `AuthHandler`, зависимость на HTTP-клиент для token endpoint Google.
- **webapp (React):** `LoginPage`, `AuthContext`/`api.ts` — ссылка на старт OAuth, обработка возврата с кодом/ошибкой.
- **Документация:** `.env.example`, README при необходимости.
- **Внешние:** консоль Google Cloud (OAuth client, redirect URI).
