## Контекст

В knowledge-db уже есть сессионная авторизация: пара `KB_LOGIN` / `KB_PASSWORD`, cookie `kb_session`, маршруты `POST /api/auth/login`, `GET /api/auth/session`, `POST /api/auth/logout` (см. `internal/api/auth_handlers.go`, `web-session-auth`). В веб-интерфейсе — форма логин/пароль на `LoginPage`. Нужно спроектировать вход через Google с ограничением по списку email в окружении, не нарушая существующую модель сессий после успешной аутентификации.

## Цели и вне scope

**Цели:**

- Поток authorization code с Google (стандартные `accounts.google.com` + token endpoint) и получение **подтверждённого** email (поля `email` и `email_verified=true`).
- Allowlist: серверная сессия выдаётся только адресам из `KB_AUTH_ALLOWED_EMAILS`; остальные получают отказ (без сессии, без утечки в UI деталей, кроме общего «доступ запрещён» или аналогичного сообщения).
- Защита callback от CSRF через криптостойкий `state` (и сверку redirect URL с заранее заданным в env).
- Предсказуемая матрица env: задокументированные переменные для OAuth, сессий и CORS.
- OpenSpec-артефакты (delta-спеки) для последующей реализации и тестов.

**Вне scope:**

- Мульти-тенантность, роли, профили пользователей в БД.
- OpenID и провайдеры, отличные от Google.
- Service account или доступ к API Google от имени сервера, кроме обмена `code` на токены для входа.
- Смена механизма хранения сессий (как сейчас — in-memory в `session.Store` для браузерных сессий).

## Решения

1. **Режимы авторизации: пароль и Google — взаимоисключающие.**  
   - **Обоснование:** два способа входа на одной инсталляции усложняют продукт и тесты; личный инстанс обычно выбирает один механизм.  
   - **Правило:** при полной настройке Google OAuth (см. env) пароль **не** используется, даже если заданы `KB_*` — **недопустимый старт** (лог + выход), чтобы владелец явно убрал лишние переменные.  
   - **Альтернатива:** «сначала пароль, если нет Google» — отклонена из-за риска оставить старый пароль активным.

2. **Список email: одна переменная `KB_AUTH_ALLOWED_EMAILS`.**  
   - **Формат:** разделитель — запятая (и при необходимости пробелы после запятой); сравнение **без учёта регистра**; **не** допускаются `*` / регулярные выражения в первой версии (только полные адреса).  
   - **Обоснование:** простая проверка в .env, без внешних БД.  
   - **Альтернатива:** путь к файлу со списком — отложена.

3. **Callback URL: явный `KB_GOOGLE_OAUTH_REDIRECT_URL`.**  
   - **Обоснование:** значение **должно в точности** совпадать с зарегистрированным в Google Cloud Console; явный URL снижает ошибки при подстановке базового адреса.  
   - **Связь с существующим:** `KB_PUBLIC_WEB_BASE_URL` — для ссылок из Telegram и др.; редирект в SPA после callback может опираться на него, но **redirect URI для Google** — только `KB_GOOGLE_OAUTH_REDIRECT_URL`.

4. **Параметр `state` в OAuth:** HMAC/подпись или сопоставление `state` в store с `KB_OAUTH_STATE_SECRET` (секрет из env; рекомендуемо ≥ 32 байта, base64/hex) либо отдельный ключ.  
   - **Обоснование:** стандартная защита от подделки callback.  
   - **Альтернатива:** только карта `state` в памяти — слабее при перезапуске процесса; предпочтительна stateless-подпись.

5. **Сессия после Google:** существующий `session.Store.Create`, те же атрибуты cookie, тот же `GET /api/auth/session`.  
   - **Обоснование:** без смены middleware `auth.Middleware` и ответа 401 на `/api/*`.

6. **UI:** кнопка «Войти через Google» ведёт на `GET /api/auth/google` (редирект 302 на Google). После успешного callback (`GET /api/auth/google/callback`) сервер перенаправляет в SPA на `KB_PUBLIC_WEB_BASE_URL` (и при необходимости хранит маршрут `redirect` в `state`) с установленной cookie.  
   - **Обоснование:** не делаем CORS preflight к Google с фронта, секреты остаются на сервере.

## Переменные окружения (сводка)

| Переменная | Назначение |
|------------|------------|
| `KB_GOOGLE_OAUTH_CLIENT_ID` | Client ID из Google Cloud Console. |
| `KB_GOOGLE_OAUTH_CLIENT_SECRET` | Client secret. |
| `KB_GOOGLE_OAUTH_REDIRECT_URL` | Полный URL callback (как в консоли Google), например `https://host/api/auth/google/callback`. |
| `KB_AUTH_ALLOWED_EMAILS` | Список разрешённых email через запятую. Обязателен в режиме Google. |
| `KB_OAUTH_STATE_SECRET` | Секрет для подписи/шифрования `state` (и при появлении — PKCE). |
| `KB_SESSION_TTL` | Уже есть — время жизни сессии после OAuth и в парольном режиме. |
| `ALLOWED_CORS_ORIGIN` | Уже есть — origin SPA для `login` / `logout` / `session`. |
| `KB_PUBLIC_WEB_BASE_URL` | Публичный базовый URL веба — для финального редиректа браузера после успешного callback. |

Парольный режим по-прежнему: `KB_LOGIN`, `KB_PASSWORD` (и не задан вместе с полным набором Google OAuth; при конфликте — ошибка старта).

## Риски и компромиссы

- **Секреты в env** → **Смягчение:** не логировать client secret; в проде — secret manager.  
- **In-memory сессии** (как сейчас) — сброс сессий при рестарте → существующий компромисс, для OAuth не усугубляем.  
- **Только allowlist по email** — смена email в Google-аккаунте меняет доступ — владелец обновляет список.  
- **Блокировка при ошибочном .env** — взаимоисключение пароля и Google: **смягчение** — явное сообщение в логе при недопустимой комбинации.

## План внедрения

1. Создать OAuth client (Web) в Google Cloud, в «Authorized redirect URIs» указать `KB_GOOGLE_OAUTH_REDIRECT_URL`.  
2. Задать `KB_AUTH_ALLOWED_EMAILS` и секреты на сервере; убрать `KB_LOGIN` / `KB_PASSWORD` при переходе с пароля.  
3. Развернуть новую версию бинарника и веб-статики, проверить `GET /api/auth/session` и «счастливый» путь.  
4. **Откат:** откат релиза и возврат к парольному набору env (без переменных Google).

## Открытые вопросы

- Точные пути эндпоинтов OAuth зафиксированы в delta-спеке `rest-api`; при реализации соблюдать их для согласованности с API-тестами.  
- Нужны ли `offline` / refresh у Google в будущем — **нет** в текущем scope (сессия только в cookie на сервере).
