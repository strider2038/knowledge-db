## Context

Страница узла (`NodePage`) отображает `annotation` (описание источника) и `content` (тело) в режиме только чтения. Редактируются лишь отдельные поля frontmatter (`title`, `keywords`, `labels`). Ingestion и agent-edit могут перезаписывать тело узла — личные заметки не должны попадать в `.md` файл и не должны теряться при refresh.

Принципы проекта: git-first, offline-first. Пользователь хочет версионировать аннотации в git. Поиск и RAG по аннотациям не нужны; UI — только страница узла. Заметки общие для оригинала и переводов. Формат текста заметки — plain text.

## Goals / Non-Goals

**Goals:**

- Sidecar `{baseSlug}/annotations.yaml` в git рядом с `{slug}.md`.
- CRUD общих и привязанных аннотаций через REST API.
- Панель заметок на странице узла (десктоп — справа; мобила — bottom sheet).
- Привязка к фрагменту через `text_quote` (exact + prefix/suffix + опционально `heading_id`).
- Якорь сохраняет `content_path` файла, в котором создан (оригинал или перевод); resolve ищет цитату в соответствующем теле.
- Перенос и удаление sidecar вместе с узлом (move/delete).
- Исключение аннотаций из embedding/search index.

**Non-Goals:**

- Markdown-разметка в теле заметки.
- Поиск по аннотациям (hybrid/FTS/overview filter).
- RAG/чат-контекст из аннотаций.
- Telegram-бот, MCP, CLI.
- Индикаторы аннотаций на «Обзоре» и в результатах поиска.
- Редактирование `annotation` или `content` узла в UI.
- Wikilinks, треды, совместное редактирование.

## Decisions

### 1. Хранение: `{slug}/annotations.yaml`

**Выбор:** один YAML-файл в директории вложений узла (`theme/slug/annotations.yaml`).

**Альтернативы:**
- `.local/` (gitignored) — отклонено: пользователь хочет git.
- Frontmatter-поле — ломается при refresh, смешивается с контрактом узла.
- SQLite — не git-first.
- Отдельный `.md` на заметку в `notes/` — неудобно для line-anchor metadata и CRUD.

Схема файла:

```yaml
version: 1
notes:
  - id: "<uuid-v7>"
    created: "<iso8601>"
    updated: "<iso8601>"
    body: "plain text"
    anchor: null   # общая заметка
  - id: "<uuid-v7>"
    created: "<iso8601>"
    updated: "<iso8601>"
    body: "plain text"
    anchor:
      type: text_quote
      content_path: "theme/slug"   # или theme/slug.ru
      exact: "выделенный текст"
      prefix: "до "
      suffix: " после"
      heading_id: "section-slug"   # опционально
```

### 2. Привязка к узлу, не к файлу перевода

**Выбор:** файл `annotations.yaml` всегда у **базового** пути (`theme/slug/`), API принимает `basePath` без суффикса языка. На странице перевода показывается та же лента.

**Альтернатива:** отдельные аннотации на перевод — отклонено по запросу пользователя.

### 3. Якорь с `content_path` (вариант B)

**Выбор:** при создании привязанной аннотации сохранять `content_path` текущего открытого файла. Resolve выполняется в теле этого файла. На странице другого языка заметка видна в списке, но подсветка цитаты доступна только если resolve успешен в **текущем** теле (иначе — статус `resolved: false` без ошибки удаления).

**Альтернатива:** якорь только в оригинале — хуже при чтении переводов.

### 4. Алгоритм resolve якоря

**Выбор:** поиск подстроки `exact` в plain-text представлении тела (markdown без разметки или с нормализованными пробелами). При нескольких совпадениях — уточнение по `prefix`+`suffix`; при неоднозначности — первое совпадение + флаг `ambiguous` в ответе API (опционально для UI).

Устаревший якорь: заметка сохраняется, `resolved: false`, UI показывает сохранённую цитату и кнопку «Перепривязать» (повторное выделение → PATCH anchor).

### 5. API отдельно от PATCH метаданных

**Выбор:** dedicated routes `/api/nodes/{basePath}/annotations[/{id}]`, не расширять `PATCH /api/nodes`.

**Причина:** другая модель данных, другой файл, частые автосохранения; не раздувать frontmatter PATCH.

### 6. UI layout

**Выбор:**
- Десктоп (lg+): правая колонка ~280px «Заметки», контент сужается; оглавление слева без изменений.
- Мобила: кнопка «Заметки (N)» → `Sheet`; маркеры ● в тексте остаются.
- Создание: `+ Заметка` (общая) и floating toolbar при выделении в блоке «Содержание».
- Редактирование: `<textarea>`, debounced autosave (~500ms).
- Сортировка в панели: сначала привязанные (по позиции в тексте), затем общие (по `updated` desc).

### 7. Индекс и RAG

**Выбор:** не читать `annotations.yaml` в index sync; не добавлять в `buildNodeEmbeddingText`. Изменение только аннотаций не меняет `content_hash`.

### 8. Move и delete

**Выбор:** при `POST .../move` переносить `{slug}/annotations.yaml` вместе с `{slug}/` (images и прочие вложения). При `DELETE` узла удалять sidecar. При отсутствии файла — пустой список (не ошибка).

### 9. Лимиты

- Максимум **200** аннотаций на узел.
- Длина `body` — **4000** символов.
- Длина `exact` — **500** символов; `prefix`/`suffix` — по **80** символов.

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| Якорь ломается после refresh/agent-edit | text_quote + orphan UI; перепривязка |
| Конфликты git при одновременном редактировании sidecar | обычный merge YAML; файл маленький |
| Выделение в сложном markdown (код, таблицы) | MVP: привязка к выделенному plain text; не поддерживать multi-block |
| Производительность resolve на больших статьях | кэш resolve на клиенте по `(content_hash, notes)`; серверный resolve только в GET |
| Путаница annotation vs заметки | разные заголовки UI: «Аннотация» vs «Мои заметки» |

## Migration Plan

Обратная совместимость: узлы без `annotations.yaml` работают как сегодня (пустая панель). Деплой: server + embedded web. Переиндексация не требуется. Откат: удалить поддержку API/UI; файлы `annotations.yaml` останутся в репозитории без вреда.
