# chat-session-memory Specification

## Purpose

Персистентные чат-сессии во временном SQLite-хранилище: история диалога, лимит контекста, служебная суммаризация без отображения в UI. См. реализацию в `internal/chat`.

## Requirements

### Requirement: Персистентные чат-сессии

Система ДОЛЖНА (SHALL) создавать и хранить чат-сессии в SQLite как временные рабочие сущности, чтобы пользователь мог открыть прошлый чат и продолжить его без потери контекста в рамках жизненного цикла приложения.

#### Scenario: Создание новой сессии

- **WHEN** пользователь начинает новый чат
- **THEN** система MUST создать новую сессию с уникальным идентификатором, временем создания и пустой историей сообщений

#### Scenario: Продолжение существующей сессии

- **WHEN** пользователь открывает ранее созданный чат
- **THEN** система SHALL загрузить историю сообщений этой сессии и использовать её для последующих ответов

### Requirement: Автоматическое название чата

Система SHALL автоматически определять название чата на основе первых сообщений пользователя и/или итоговой summary, а также MUST поддерживать ручное переименование чата.

#### Scenario: Название после первых сообщений

- **WHEN** в новой сессии накоплены первые пользовательские реплики
- **THEN** система MUST присвоить чату человеко-читаемое название автоматически

#### Scenario: Ручное переименование чата

- **WHEN** пользователь задаёт новое название существующему чату
- **THEN** система MUST сохранить новое название и отразить его в списке чатов

### Requirement: Лимит контекстного окна и суммаризация

Система MUST ограничивать размер контекста, отправляемого в LLM, и при превышении лимита SHALL суммаризировать старые сообщения в компактный блок, сохраняя смысл беседы.

#### Scenario: Превышение лимита

- **WHEN** история сообщений превышает настроенный лимит
- **THEN** система SHALL заменить часть самых старых сообщений summary-блоком и сохранить новые сообщения в полном виде

#### Scenario: Использование summary в последующих ответах

- **WHEN** формируется новый prompt для ответа ассистента
- **THEN** система MUST включать актуальный summary-блок вместе с недавними сообщениями активной сессии

### Requirement: Служебный характер summary

Система MUST использовать summary-сообщения только как внутренний механизм управления контекстом и SHALL не отображать их пользователю в UI первой версии.

#### Scenario: Загрузка истории для UI

- **WHEN** клиент запрашивает историю сообщений для отображения
- **THEN** система MUST вернуть пользовательские/ассистентские сообщения без показа служебных summary-блоков
