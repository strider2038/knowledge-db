# node-cursor-agent-edit Specification

## Purpose

Серверный сценарий произвольного редактирования узла базы знаний через Cursor Agent по пользовательской инструкции, сбор логов и post-sync.

## Requirements

### Requirement: Редактирование узла по инструкции через Cursor Agent

Система SHALL предоставлять серверный сценарий редактирования конкретного узла базы знаний через Cursor Agent. Клиент MUST передавать текстовую инструкцию (`instruction`); сервер MUST сформировать промт с ограничениями (редактировать только файл узла, не менять path) и контекстом узла (metadata, annotation, content). Операция MUST быть привязана к одному path узла и MUST возвращать диагностируемый статус выполнения.

#### Scenario: Успешный запуск редактирования

- **WHEN** клиент отправляет непустую `instruction` для существующего узла и `cursor-agent` доступен в PATH
- **THEN** сервер создаёт операцию со статусом `running`, стадией `edit` и возвращает идентификатор операции

#### Scenario: Пустая инструкция

- **WHEN** клиент отправляет пустую или состоящую только из пробелов `instruction`
- **THEN** сервер возвращает ошибку валидации (4xx) и MUST NOT запускать Cursor Agent

#### Scenario: Узел не найден

- **WHEN** клиент запрашивает редактирование для несуществующего path
- **THEN** сервер возвращает ошибку not found и MUST NOT запускать Cursor Agent

#### Scenario: Cursor Agent недоступен

- **WHEN** `cursor-agent` отсутствует в PATH сервера
- **THEN** сервер возвращает диагностируемую ошибку конфигурации/runtime и MUST NOT создавать операцию

### Requirement: Сбор логов операции agent-edit

Система SHALL собирать лог выполнения операции из `stdout` и `stderr` процесса Cursor Agent (включая разбор `stream-json`, если применимо), а также системные события стадий (`edit`, `sync`, `done`/`error`). Каждая запись MUST иметь монотонный offset, временную метку и тип потока.

#### Scenario: Логи во время выполнения

- **WHEN** операция agent-edit в статусе `running`
- **THEN** записи вывода агента и системные события доступны через API логов с инкрементальным чтением

#### Scenario: Системные записи стадий

- **WHEN** операция переходит между стадиями `edit`, `sync` и финальным состоянием
- **THEN** в лог добавляется запись типа `system` с описанием перехода

### Requirement: Автоматический sync после успешного редактирования

После успешного завершения шага `edit` система SHALL автоматически выполнять `sync` (если git не отключён) и MUST отражать итог обоих шагов в результате операции (`edit_ok`, `sync_done`).

#### Scenario: Edit и sync успешны

- **WHEN** Cursor Agent завершил редактирование без ошибки и `sync` выполнился успешно
- **THEN** итоговый статус операции `success`, стадия `done`, флаги отражают успех обоих шагов

#### Scenario: Edit успешен, sync с ошибкой

- **WHEN** редактирование завершилось успешно, но `sync` вернул ошибку
- **THEN** операция возвращает ошибку на стадии `sync` с диагностикой, при этом факт успешного локального edit сохраняется в метаданных операции

### Requirement: Блокировка параллельных операций на одном узле

Система MUST NOT запускать вторую операцию agent-edit для того же path, пока предыдущая операция agent-edit для этого path в статусе `running`. При повторном запросе сервер SHALL вернуть идентификатор уже выполняющейся операции или диагностируемую ошибку конфликта (согласованно с поведением нормализации).

#### Scenario: Повторный запуск во время running

- **WHEN** для path уже есть running-операция agent-edit
- **THEN** сервер не создаёт вторую операцию и возвращает информацию о существующей
