# rag-chat Specification

## Purpose

RAG-чатбот по базе знаний с SSE, гибридным retrieval и серверной памятью активной чат-сессии (`session_id`), режимами `chat_memory` / `rag_kb` / `hybrid` и защитой от пустого RAG-контекста.

## Requirements

### Requirement: Endpoint чатбота

API ДОЛЖЕН (SHALL) предоставлять endpoint `POST /api/chat` для RAG-чатбота с поддержкой чат-сессий. Запрос MUST содержать `message` и идентификатор активной сессии `session_id`. Создание новой сессии выполняется отдельным endpoint `POST /api/chats`. Запрос MAY содержать список `source_paths` для ограничения ответа выбранными источниками из поиска. Ответ MUST быть streaming (Server-Sent Events) с источниками, найденным контекстом (при наличии) и токенами ответа LLM. Генерация ответа SHOULD использовать OpenAI-compatible Chat Completions streaming, чтобы работать с локальными провайдерами вроде LM Studio. Endpoint MUST возвращать 503 при `KB_EMBEDDING_ENABLED=false`.

#### Scenario: Успешный запрос к чатботу в существующей сессии

- **WHEN** `POST /api/chat` отправлен с `session_id` и `{ "message": "Какие паттерны DI в Go?" }`
- **THEN** выполняется retrieval по базе, учитывается контекст сессии, и LLM генерирует ответ как SSE stream

#### Scenario: Запрос с выбранными источниками

- **WHEN** `POST /api/chat` содержит `source_paths`
- **THEN** retrieval и контекст ответа ограничиваются указанными нодами и их фрагментами

#### Scenario: Пустое сообщение

- **WHEN** `POST /api/chat` с пустым или отсутствующим `message`
- **THEN** возвращается 400 с описанием ошибки валидации

#### Scenario: Не передан идентификатор сессии

- **WHEN** `POST /api/chat` отправлен без `session_id`
- **THEN** возвращается 400 с описанием ошибки валидации

#### Scenario: Embeddings отключены

- **WHEN** `POST /api/chat` при `KB_EMBEDDING_ENABLED=false`
- **THEN** возвращается 503

#### Scenario: Streaming совместим с LM Studio

- **WHEN** chat provider настроен на OpenAI-compatible `/v1` endpoint локальной модели
- **THEN** backend использует chat completions streaming и stream-ит delta content как SSE token events

#### Scenario: SSE не буферизуется gzip middleware

- **WHEN** браузер или клиент отправляет `Accept-Encoding: gzip`
- **THEN** `/api/chat` response не сжимается gzip и содержит headers, разрешающие немедленную доставку SSE chunks

#### Scenario: Превышение лимита истории сессии

- **WHEN** контекст активной сессии превышает лимит prompt budget
- **THEN** система MUST использовать summary-сегменты и последние сообщения, сохраняя непрерывность диалога без превышения лимита

### Requirement: Контекстная сборка для RAG

Система ДОЛЖНА (SHALL) собирать контекст для LLM из результатов гибридного retrieval pipeline. Контекст SHALL формироваться из ранжированных нод и фрагментов, найденных через exact/keyword/FTS/vector совпадения. Общий размер контекста MUST ограничиваться (не более ~4000 токенов). Если найдены и ноды, и чанки одной и той же статьи, статья MUST быть представлена наиболее релевантными чанками и краткими метаданными ноды.

#### Scenario: Найдены релевантные ссылки и статьи

- **WHEN** hybrid retrieval нашёл 3 link-ноды по keywords и 2 chunk'а из статьи по semantic близости
- **THEN** контекст содержит annotations ссылок и фрагменты статьи, общим размером не более 4000 токенов

#### Scenario: Ничего релевантного не найдено

- **WHEN** hybrid retrieval не нашёл результатов выше chat cutoff
- **THEN** LLM получает пустой контекст или сервер stream-ит ответ о том, что информация в базе не найдена

#### Scenario: Приоритет точного совпадения

- **WHEN** запрос содержит точный термин из keywords
- **THEN** соответствующая нода включается в контекст даже при отсутствии высокого vector score

### Requirement: LLM для генерации ответов

Система ДОЛЖНА (SHALL) использовать LLM (OpenAI-совместимое API) для генерации ответов чатбота. Система MUST использовать тот же API-конфиг что и embedding provider (KB_EMBEDDING_API_KEY, KB_EMBEDDING_API_URL), но с chat-моделью из `KB_CHAT_MODEL` (default: `gpt-4o` при пустом значении). При необходимости можно задать отдельные `KB_CHAT_API_URL` и `KB_CHAT_API_KEY` для использования другого API endpoint'а для чата. System prompt SHALL инструктировать LLM отвечать на основе предоставленного контекста базы знаний.

#### Scenario: Генерация ответа с контекстом

- **WHEN** собран контекст из 5 релевантных фрагментов
- **THEN** LLM получает system prompt + контекст + вопрос пользователя и генерирует ответ по данным базы

#### Scenario: Streaming ответ

- **WHEN** LLM начинает генерировать токены
- **THEN** каждый токен отправляется клиенту как SSE event `data: {"token": "..."}\n\n`, stream завершается событием `data: [DONE]\n\n`

### Requirement: Источники в ответе

Ответ чатбота ДОЛЖЕН (SHALL) включать список источников — нод базы знаний, выбранных retrieval pipeline для формирования ответа. Каждый источник MUST содержать: path, title, type и MAY содержать fragments с heading/snippet. Источники SHALL отправляться как SSE event `data: {"sources": [...]}\n\n` перед началом токенов ответа. UI MAY отображать их как “найденные источники”, если LLM не выдаёт inline citations.

#### Scenario: Ответ с источниками

- **WHEN** чатбот формирует ответ на основе 3 нод
- **THEN** перед токенами ответа отправляется SSE event с sources: [{path, title, type, fragments}, ...]

#### Scenario: Ответ без релевантных нод

- **WHEN** hybrid retrieval не нашёл релевантных результатов
- **THEN** sources отправляется как пустой массив

#### Scenario: Источник с фрагментом статьи

- **WHEN** источником ответа является chunk статьи
- **THEN** source содержит fragment с heading и snippet, чтобы пользователь мог понять основание ответа

### Requirement: Серверная память диалога

Чат ДОЛЖЕН (SHALL) обрабатывать `POST /api/chat` с серверной памятью активной сессии. Backend MUST строить prompt из текущего `message`, выбранных `source_paths`, найденного RAG-контекста и истории активной сессии (включая summary-сегменты при необходимости).

#### Scenario: Следующий вопрос получает историю активной сессии

- **WHEN** пользователь отправляет несколько сообщений подряд в одном чате
- **THEN** backend использует предыдущие user/assistant сообщения активной сессии как conversation history

### Requirement: Маршрутизация запросов чата по режимам

Чат ДОЛЖЕН (SHALL) определять режим обработки сообщения перед retrieval: `chat_memory` для вопросов о текущем диалоге, `rag_kb` для явных вопросов по базе знаний и `hybrid` для обычных уточняющих сообщений. В режиме `chat_memory` backend MUST не выполнять поиск по базе знаний и MUST отвечать только на основе истории активной сессии.

#### Scenario: Резюме текущего диалога не ищет по базе знаний

- **WHEN** пользователь спрашивает "Сделай краткое резюме чата"
- **THEN** система MUST использовать режим `chat_memory`, вернуть пустой список источников и сформировать ответ из conversation history

#### Scenario: Вопрос по базе знаний использует RAG

- **WHEN** пользователь спрашивает "Что есть в базе про RAG?"
- **THEN** система SHALL использовать режим `rag_kb` и выполнить retrieval по базе знаний

#### Scenario: Уточнение без явного указания базы использует гибридный режим

- **WHEN** пользователь спрашивает "Расскажи подробнее"
- **THEN** система SHALL использовать режим `hybrid`, учитывая историю активной сессии и доступный RAG-контекст при наличии

### Requirement: Компактный retrieval-запрос для чата

Чат ДОЛЖЕН (SHALL) отделять исходное сообщение пользователя от поискового запроса к базе знаний. Backend MUST очищать retrieval-запрос от служебных слов и фраз вроде "что есть", "в базе знаний", "какие статьи", сохраняя доменные термины, акронимы и продуктовые названия.

#### Scenario: Инвентарный вопрос про тему ищется по доменному термину

- **WHEN** пользователь спрашивает "Что есть про RAG в базе знаний?"
- **THEN** retrieval MUST выполняться по компактному запросу `RAG`, а исходный вопрос MUST оставаться вопросом для LLM-ответа

#### Scenario: Смешанный русско-английский запрос сохраняет технические термины

- **WHEN** пользователь спрашивает "Найди в базе dependency injection Go"
- **THEN** retrieval MUST сохранять поисковые термины `dependency injection Go` без служебной фразы "Найди в базе"

### Requirement: Фильтрация источников чата

Чат ДОЛЖЕН (SHALL) фильтровать источники перед передачей в prompt так, чтобы снижать шум от слабых vector-only совпадений, но не отбрасывать релевантные lexical/exact/keyword результаты только из-за низкого общего hybrid score. Фильтрация MUST учитывать overlap доменных терминов запроса с title, path, annotation, keywords и найденными фрагментами.

#### Scenario: Тема с несколькими keyword-источниками сохраняет список источников

- **WHEN** retrieval для вопроса "Что есть про RAG в базе знаний?" возвращает несколько keyword/exact источников с overlap по `RAG`
- **THEN** система MUST сохранить эти источники для ответа, включая более низко оценённые lexical результаты, если они попадают в лимит источников

#### Scenario: Нерелевантный vector-only источник отбрасывается

- **WHEN** retrieval возвращает vector-only источник без overlap с доменными терминами запроса
- **THEN** система MUST исключить такой источник из RAG-контекста перед генерацией ответа

#### Scenario: Обрезка хвоста по score gap не удаляет первые релевантные источники

- **WHEN** после нескольких релевантных источников появляется заметный перепад score
- **THEN** система SHALL обрезать хвост результатов только после сохранения минимального набора релевантных источников

### Requirement: Безопасный fallback при отсутствии контекста

Чат ДОЛЖЕН (SHALL) явно сообщать пользователю, когда в режиме `rag_kb` после retrieval и фильтрации не осталось релевантного контекста базы знаний. В этом случае backend MUST не просить LLM придумывать ответ по пустому контексту базы.

#### Scenario: В базе не найден релевантный контекст для RAG-запроса

- **WHEN** пользователь задаёт явный вопрос по базе знаний, но retrieval не возвращает релевантных источников
- **THEN** система MUST вернуть ответ "В базе знаний не найдено релевантной информации по запросу." и сохранить его в истории чата

#### Scenario: Гибридный режим без контекста сообщает о недостатке данных

- **WHEN** пользователь задаёт гибридный вопрос и после фильтрации нет RAG-контекста
- **THEN** система SHALL вернуть диагностируемое сообщение о недостатке данных в базе знаний без генерации неподтверждённого ответа
