## Purpose

Telegram-бот для приёма сообщений и URL, вызова ingestion pipeline. Работает в том же процессе, что и kb-server. Принимает сообщения только от авторизованного пользователя.

## Requirements

### Requirement: Конфигурация через env

Бот ДОЛЖЕН (SHALL) читать TELEGRAM_TOKEN, KB_DATA_PATH и TELEGRAM_OWNER_ID из переменных окружения. TELEGRAM_OWNER_ID MUST быть задан при наличии TELEGRAM_TOKEN.

#### Сценарий: Запуск без токена

- **WHEN** kb-server запущен без TELEGRAM_TOKEN
- **THEN** бот не стартует, логирует предупреждение

#### Сценарий: Запуск без TELEGRAM_OWNER_ID

- **WHEN** kb-server запущен с TELEGRAM_TOKEN, но без TELEGRAM_OWNER_ID
- **THEN** сервер возвращает ошибку конфигурации и не стартует

### Requirement: Приём сообщений

Бот MUST принимать текстовые сообщения только от авторизованного пользователя и передавать их в ingestion pipeline через IngestText. Бот не выполняет предварительный анализ контента — LLM-оркестратор внутри pipeline определяет тип контента (текст, URL на статью, URL на ресурс) и выбирает действия.

#### Сценарий: Получение текста

- **WHEN** пользователь отправляет боту текст
- **THEN** бот передаёт текст в IngestText и отправляет подтверждение с результатом

#### Сценарий: Получение URL

- **WHEN** пользователь отправляет боту сообщение с URL
- **THEN** бот передаёт всё сообщение в IngestText (LLM внутри pipeline определит, что делать с URL)

#### Сценарий: Получение текста с инструкциями

- **WHEN** пользователь отправляет "https://habr.com/... сохрани в go/concurrency"
- **THEN** бот передаёт всё сообщение в IngestText (LLM учтёт инструкции)

#### Сценарий: Пересланное медиа с подписью

- **WHEN** пользователь пересылает боту сообщение с медиа (фото, видео, документ) и подписью
- **THEN** бот передаёт текст подписи (caption) в IngestText и отправляет подтверждение

### Requirement: Немедленное подтверждение приёма

Бот ДОЛЖЕН (SHALL) сразу после получения сообщения от авторизованного пользователя отправить подтверждение в духе «Принял, обрабатываю...» до вызова ingestion pipeline. Это даёт пользователю мгновенную обратную связь, пока pipeline обрабатывает контент.

#### Сценарий: Подтверждение до обработки

- **WHEN** пользователь отправляет боту сообщение
- **THEN** бот сначала отправляет подтверждение («Принял, обрабатываю...» или аналогичное), затем передаёт сообщение в IngestText и по завершении отправляет итоговый ответ с результатом

### Requirement: Авторизация по Telegram user ID

Бот ДОЛЖЕН (SHALL) проверять `from.id` входящего сообщения против `TELEGRAM_OWNER_ID`. Сообщения от неавторизованных пользователей MUST игнорироваться.

#### Сценарий: Сообщение от владельца

- **WHEN** бот получает сообщение с `from.id` равным TELEGRAM_OWNER_ID
- **THEN** сообщение обрабатывается и передаётся в ingestion pipeline

#### Сценарий: Сообщение от постороннего

- **WHEN** бот получает сообщение с `from.id` не равным TELEGRAM_OWNER_ID
- **THEN** сообщение игнорируется, логируется warning

### Requirement: Ответ пользователю

Бот ДОЛЖЕН (SHALL) отправлять пользователю два сообщения: немедленное подтверждение приёма (см. Requirement: Немедленное подтверждение приёма) и итоговый ответ после завершения ingestion. Итоговый ответ MUST содержать: путь к созданному узлу (тема/подтема/slug), тип контента (article/link/note) и ключевые слова.

#### Сценарий: Успешный ingestion

- **WHEN** ingestion pipeline успешно создал узел
- **THEN** бот отправляет сообщение с путём узла, типом и keywords

#### Сценарий: Ошибка ingestion

- **WHEN** ingestion pipeline вернул ошибку
- **THEN** бот отправляет сообщение об ошибке пользователю

### Requirement: Long polling

Бот ДОЛЖЕН (SHALL) использовать long polling для получения обновлений (без публичного URL).

#### Сценарий: Работа без webhook

- **WHEN** бот запущен
- **THEN** он получает обновления через long polling, без необходимости настраивать webhook
