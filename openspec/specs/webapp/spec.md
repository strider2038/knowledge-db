## Purpose

Веб-интерфейс для просмотра, поиска, создания и редактирования записей базы знаний. React + Vite, статика встраивается в бинарник kb-server.
## Requirements
### Requirement: Navbar с меню

UI ДОЛЖЕН (SHALL) содержать верхнюю панель навигации (Navbar) с пунктами меню: «Добавить», «Обзор». После успешного ответа `GET /api/git/status` Navbar MUST показывать справа (перед logout и theme toggle) элемент сохранения в git: при **включённом** git и **наличии** незакоммиченных изменений — активную кнопку **«Сохранить (N)»**; при включённом git и **отсутствии** изменений — неактивную кнопку **«Сохранить»** с подсказкой по наведению; при **отключённом** git (ответ 503) — всегда неактивную кнопку **«⚠️ Сохранить»** с подсказкой по наведению (причина: git отключён на сервере, например KB_GIT_DISABLED). UI MUST периодически (каждые 30 секунд) проверять git-статус. При нажатии активной кнопки выполняется коммит+пуш с LLM-генерацией сообщения. Результат (успех или ошибка) MUST показываться **всплывающим уведомлением снизу экрана** (toast), а не строкой рядом с кнопкой.

#### Сценарий: Переключение между разделами

- **WHEN** пользователь выбирает пункт «Добавить» или «Обзор»
- **THEN** отображается соответствующий раздел

#### Сценарий: Кнопка «Сохранить» активна

- **WHEN** GET /api/git/status возвращает has_changes: true (git не отключён)
- **THEN** в Navbar отображается активная кнопка «Сохранить» с количеством изменённых файлов

#### Сценарий: Кнопка «Сохранить» неактивна без изменений

- **WHEN** GET /api/git/status возвращает has_changes: false и git не отключён
- **THEN** в Navbar отображается неактивная кнопка «Сохранить» с подсказкой по наведению

#### Сценарий: Git отключён на сервере

- **WHEN** GET /api/git/status возвращает 503
- **THEN** в Navbar отображается неактивная кнопка «⚠️ Сохранить» с подсказкой по наведению о причине

#### Сценарий: Успешное сохранение через Navbar

- **WHEN** пользователь нажимает активную «Сохранить»
- **THEN** вызывается POST /api/git/commit, при успехе показывается toast снизу с commit message, после обновления статуса кнопка становится неактивной или скрывается с активной зоны (нет незакоммиченных изменений)

### Requirement: Раздел «Добавить»

При выборе «Добавить» MUST отображаться страница с табами: «Вручную» и «Импорт из Telegram».

**Вкладка «Вручную»:** форма с переключателем типа контента (авто, статья, ссылка, заметка), большое текстовое поле (textarea) и кнопка «Добавить». При выборе «статья» или «ссылка» MUST отображаться подсказка «Вставьте URL в текст». Форма предназначена для ввода текста нового знания и отправки в ingestion pipeline. При выборе типа, отличного от «авто», значение MUST передаваться в API как type_hint.

Во время обработки (loading) форма MUST блокировать повторную отправку: textarea и кнопка disabled, MUST отображаться анимация загрузки (спиннер или индикатор прогресса). Кнопка MUST показывать состояние обработки (напр. «Обработка...» или спиннер).

При успехе MUST отображаться блок (Alert/Banner) с сообщением «Добавлено» и ссылкой «Перейти к узлу» (если API вернул path). При ошибке MUST отображаться блок с текстом ошибки. Сообщение сбрасывается при следующей отправке.

**Вкладка «Импорт из Telegram»:** MUST отображаться возможность загрузить JSON-файл экспорта чата. После загрузки MUST отображаться индикатор прогресса (N/M, принято X, отклонено Y), карточка текущей записи (текст, source_author), переключатель типа контента, кнопки «Отклонить» и «Принять». При «Принять» MUST вызываться API accept с выбранным type_hint; при «Отклонить» — API reject. При наличии активной сессии (session_id в state или из предыдущей загрузки) MUST автоматически загружаться состояние сессии (GET session) и отображаться текущая запись. Должна быть возможность «Начать заново» для очистки сессии и загрузки нового файла.

**Завершение импорта:** когда обработана последняя запись сессии (ответ accept или reject без следующей записи, `next_item` отсутствует), UI MUST отображать промежуточное состояние завершения до возврата к выбору файла: явное сообщение, что импорт завершён; итоговые числа — всего в выгрузке, принято, отклонено; если последнее действие было «Принять», MUST отображаться ссылка на последний созданный узел; MUST быть действие для начала нового импорта (например кнопка «Импортировать другой файл»), по которому снова показывается форма загрузки JSON. Немедленный переход только к форме выбора файла без такого экрана MUST NOT выполняться (чтобы не создавать впечатление, что результаты импорта потеряны).

#### Сценарий: Добавление знания

- **WHEN** пользователь на вкладке «Вручную» вводит текст и нажимает «Добавить»
- **THEN** текст отправляется в API (ingestion), отображается результат

#### Сценарий: Добавление с выбором типа

- **WHEN** пользователь на вкладке «Вручную» выбирает тип «статья», вводит текст и нажимает «Добавить»
- **THEN** текст и type_hint=article отправляются в API

#### Сценарий: Пустой ввод

- **WHEN** пользователь нажимает «Добавить» без ввода текста
- **THEN** кнопка неактивна или показывается валидация

#### Сценарий: Блокировка при обработке

- **WHEN** запрос на добавление выполняется (loading)
- **THEN** textarea и кнопка disabled, отображается анимация загрузки, повторная отправка невозможна

#### Сценарий: Подсказка при выборе статьи или ссылки

- **WHEN** пользователь выбирает тип «статья» или «ссылка»
- **THEN** под переключателем отображается текст «Вставьте URL в текст»

#### Сценарий: Успешное добавление

- **WHEN** запрос завершился успешно
- **THEN** отображается блок с сообщением «Добавлено» и ссылкой «Перейти к узлу» (если API вернул path)

#### Сценарий: Ошибка при добавлении

- **WHEN** запрос завершился с ошибкой
- **THEN** отображается блок с текстом ошибки из API

#### Сценарий: Загрузка JSON для импорта

- **WHEN** пользователь на вкладке «Импорт из Telegram» выбирает JSON-файл и загружает
- **THEN** создаётся сессия, отображается первая запись и прогресс

#### Сценарий: Принять запись при импорте

- **WHEN** пользователь нажимает «Принять» для текущей записи импорта
- **THEN** запись передаётся в ingestion, отображается следующая запись

#### Сценарий: Отклонить запись при импорте

- **WHEN** пользователь нажимает «Отклонить»
- **THEN** запись пропускается, отображается следующая

#### Сценарий: Восстановление сессии импорта

- **WHEN** пользователь открывает вкладку «Импорт из Telegram» и есть активная сессия (session_id)
- **THEN** загружается состояние сессии, отображается текущая запись и прогресс

#### Сценарий: Завершение импорта (все записи обработаны)

- **WHEN** пользователь нажимает «Принять» или «Отклонить» для последней записи сессии (в ответе API нет следующей записи)
- **THEN** отображается экран завершения с сообщением о завершении импорта, итоговой статистикой (всего, принято, отклонено) и возможностью перейти к последнему добавленному узлу, если последнее действие было принятие; до загрузки нового файла пользователь не видит только пустую форму выбора JSON без этого экрана

### Requirement: Раздел «Обзор»

При выборе «Обзор» MUST отображаться страница обзора базы: слева — иерархическое дерево тем (фильтр по поддереву), справа — таблица узлов с колонками Название, Тип, Дата, Ссылка. Дерево поддерживает «Вся база»; таблица — фильтры по типам (article/link/note), поиск по тексту, фильтр по ручной проверке (все / проверено / не проверено), сортировку, пагинацию. Детали фильтра по ручной проверке — в спецификации overview-browse. При переходе на страницу узла ссылка MUST передавать state с returnTo (текущий pathname + search, включая manual_processed при наличии) для сохранения контекста обзора.

#### Сценарий: Навигация по дереву

- **WHEN** пользователь раскрывает/выбирает тему в дереве
- **THEN** таблица справа обновляется и показывает узлы выбранного поддерева (рекурсивно)

#### Сценарий: Отображение списка узлов

- **WHEN** выбрана тема, подтема или «Вся база»
- **THEN** в таблице отображаются узлы с title, type, path, датой, ссылкой (для link/article)

#### Сценарий: Переход на страницу узла с сохранением контекста

- **WHEN** пользователь кликает по узлу в таблице
- **THEN** переход на страницу узла с state.returnTo, содержащим URL обзора с текущими фильтрами (path, type, q, sort, order, page, manual_processed)

### Requirement: Сборка через Vite

Веб-приложение ДОЛЖНО (SHALL) собираться через Vite. Исходники в web/, результат сборки — web/dist.

#### Сценарий: Сборка

- **WHEN** выполняется npm run build в web/
- **THEN** создаётся web/dist с статическими файлами

### Requirement: Обращение к API

UI MUST обращаться к API по URL, задаваемому через переменную VITE_API_URL (по умолчанию localhost).

#### Сценарий: Запрос данных

- **WHEN** пользователь открывает страницу
- **THEN** UI запрашивает данные у API по сконфигурированному URL

### Requirement: Просмотр узла

UI ДОЛЖЕН (SHALL) позволять просматривать содержимое узла. Страница узла MUST содержать: кнопку «Назад»; breadcrumbs по path; панель действий (Action Bar) после breadcrumbs; заголовок из metadata.title; header с type badge, created, updated, keywords; строку атрибуции источника; блок аннотации и блок содержания с рендерингом markdown.

Панель действий (Action Bar) MUST объединять все управляющие действия: toggle «Проверено», кнопку «Перевести» (для article без перевода), разделитель, «Переместить», «Удалить». Кнопка «Удалить» MUST открывать модалку подтверждения (AlertDialog). Кнопка «Переместить» MUST открывать модалку перемещения с текстовым полем полного пути и деревом тем.

#### Сценарий: Открытие узла

- **WHEN** пользователь выбирает узел из таблицы
- **THEN** отображаются breadcrumbs, панель действий, заголовок, header с метаданными, аннотация и содержание

#### Сценарий: Панель действий для article

- **WHEN** открыт узел типа article без переводов
- **THEN** панель содержит: toggle «Проверено», «Перевести», разделитель, «Переместить», «Удалить»

#### Сценарий: Модалка подтверждения удаления

- **WHEN** пользователь нажимает «Удалить»
- **THEN** открывается AlertDialog с заголовком узла, путём, кнопками «Отмена» и «Удалить» (destructive)

#### Сценарий: Успешное удаление

- **WHEN** пользователь подтверждает удаление и API возвращает успех
- **THEN** переход на страницу обзора

#### Сценарий: Модалка перемещения с полным путём

- **WHEN** пользователь нажимает «Переместить»
- **THEN** открывается Dialog с текстовым полем «Новый путь» (предзаполнено текущим путём, редактируемое), деревом тем и кнопкой «Переместить»

#### Сценарий: Перемещение с созданием новых тем

- **WHEN** пользователь вводит путь с несуществующими темами и нажимает «Переместить»
- **THEN** API создаёт промежуточные директории, при успехе — переход на новую страницу узла

#### Сценарий: Выбор темы из дерева

- **WHEN** пользователь кликает тему в дереве
- **THEN** текстовое поле «Новый путь» обновляется: `{выбранная_тема}/{текущий_slug}`

### Requirement: Экран входа и проверка сессии

Web UI ДОЛЖЕН (SHALL) поддерживать страницу входа и проверку статуса сессии через `GET /api/auth/session` при запуске в режиме включённой авторизации. При отсутствии валидной сессии UI MUST направлять пользователя на страницу входа.

#### Сценарий: Пользователь без сессии открывает защищённый раздел

- **WHEN** пользователь открывает раздел, требующий авторизацию, и `GET /api/auth/session` возвращает неаутентифицированный статус
- **THEN** UI перенаправляет пользователя на страницу входа

#### Сценарий: Пользователь с активной сессией открывает приложение

- **WHEN** пользователь открывает приложение и `GET /api/auth/session` подтверждает валидную сессию
- **THEN** UI отображает запрошенный раздел без редиректа на login

### Requirement: Кнопка и переход к Google OAuth

При **Google-режиме** (сервер настроен на OAuth) веб-интерфейс SHALL на странице входа отображать действие «Войти через Google», ведущее на `GET /api/auth/google` (полный переход в браузере: тот же origin/порт, что и API, либо иной явно согласованный URL из конфигурации), без передачи секретов на клиент. В **парольном** режиме эта кнопка SHALL NOT отображаться (либо SHALL быть отключена с пояснением — на усмотрение UI; по умолчанию скрыть).

#### Сценарий: Показ кнопки

- **WHEN** `GET /api/auth/session` возвращает `auth_mode: "google"` (и `auth_enabled: true`)
- **THEN** UI отображает «Войти через Google»

#### Сценарий: Навигация на старт OAuth

- **WHEN** пользователь нажимает «Войти через Google»
- **THEN** UI SHALL инициировать переход к `GET /api/auth/google` (например, `location.href` или ссылка)

### Requirement: Поведение login/logout flow

Web UI MUST обрабатывать сценарии входа в соответствии с режимом: в **парольном** — отправка учётных данных на `POST /api/auth/login`, после успешного ответа — обновление сессии и переход на `redirect` или `/`. В **Google-режиме** сессия SHALL устанавливается сервером в ходе редиректов OAuth; Web UI SHALL после возврата в SPA (на `KB_PUBLIC_WEB_BASE_URL` или экв.) вызвать `GET /api/auth/session` и при `authenticated` перенаправить на сохранённый `redirect` или на `/`. UI MUST поддерживать выход через `POST /api/auth/logout` с последующим переходом на login.

#### Сценарий: Успешный парольный вход

- **WHEN** парольный режим и пользователь вводит корректные login/password и отправляет форму
- **THEN** UI вызывает `POST /api/auth/login` и перенаправляет на исходный или маршрут по умолчанию

#### Сценарий: Ошибка парольного входа

- **WHEN** парольный режим и учётные данные неверны
- **THEN** UI отображает сообщение об ошибке и остаётся на login

#### Сценарий: Успешный возврат из Google

- **WHEN** Google-режим, браузер возвращается на публичный базовый URL веба (см. `KB_PUBLIC_WEB_BASE_URL`) после callback с выставленной `kb_session`
- **THEN** UI SHALL вызвать `GET /api/auth/session`, при `authenticated` SHALL перенаправить согласно `redirect` (из query или sessionStorage до OAuth) или на `/`

#### Сценарий: Отказ allowlist или ошибка OAuth

- **WHEN** Google-режим, callback отказал (ошибка в query или пустой вход)
- **THEN** UI SHALL показать понятное сообщение, не аутентифицировать, оставить на login или с приглашением повторить

#### Сценарий: Выход

- **WHEN** пользователь инициирует logout
- **THEN** UI вызывает `POST /api/auth/logout` и переводит на страницу входа
