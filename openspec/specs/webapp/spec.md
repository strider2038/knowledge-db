## Purpose

Веб-интерфейс для просмотра, поиска, создания и редактирования записей базы знаний. React + Vite, статика встраивается в бинарник `kb`.
## Requirements
### Requirement: Navbar с меню

UI ДОЛЖЕН (SHALL) содержать верхнюю панель навигации (Navbar) с пунктами меню: «Добавить», «Обзор». После успешного ответа `GET /api/git/status` Navbar MUST показывать справа (перед logout и theme toggle) элементы работы с git: кнопку **«Обновить»** (активна при включённом git, вызывает `POST /api/git/sync`) и блок сохранения: при **включённом** git и **наличии** незакоммиченных изменений — активную кнопку **«Сохранить (N)»**; при включённом git и **отсутствии** изменений — неактивную кнопку **«Сохранить»** с подсказкой по наведению; при **отключённом** git (ответ 503) — неактивные **«Обновить»** и **«⚠️ Сохранить»** с подсказками по наведению (причина: git отключён на сервере, например KB_GIT_DISABLED). UI MUST периодически (каждые 30 секунд) проверять git-статус. При нажатии «Обновить» выполняется синхронизация с удалённым репозиторием; при нажатии активной «Сохранить» — коммит+пуш с LLM-генерацией сообщения. Результат (успех или ошибка) MUST показываться **всплывающим уведомлением снизу экрана** (toast), а не строкой рядом с кнопкой.

#### Сценарий: Переключение между разделами

- **WHEN** пользователь выбирает пункт «Добавить» или «Обзор»
- **THEN** отображается соответствующий раздел

#### Сценарий: Кнопка «Сохранить» активна

- **WHEN** GET /api/git/status возвращает has_changes: true (git не отключён)
- **THEN** в Navbar отображается активная кнопка «Сохранить» с количеством изменённых файлов

#### Сценарий: Кнопка «Сохранить» неактивна без изменений

- **WHEN** GET /api/git/status возвращает has_changes: false и git не отключён
- **THEN** в Navbar отображается неактивная кнопка «Сохранить» с подсказкой по наведению

#### Сценарий: Git отключён на сервере

- **WHEN** GET /api/git/status возвращает 503
- **THEN** в Navbar отображается неактивная кнопка «⚠️ Сохранить» с подсказкой по наведению о причине

#### Сценарий: Успешное сохранение через Navbar

- **WHEN** пользователь нажимает активную «Сохранить»
- **THEN** вызывается POST /api/git/commit, при успехе показывается toast снизу с commit message, после обновления статуса кнопка становится неактивной или скрывается с активной зоны (нет незакоммиченных изменений)

#### Сценарий: Обновление из удалённого репозитория через Navbar

- **WHEN** git включён и пользователь нажимает «Обновить»
- **THEN** вызывается POST /api/git/sync, при успехе показывается toast с результатом, git-статус в Navbar обновляется, и открытые страницы «Обзор» и узла MUST перезагрузить данные с API без полной перезагрузки браузера

#### Сценарий: Git отключён — «Обновить» недоступна

- **WHEN** GET /api/git/status сообщает об отключённом git (503 / git_disabled)
- **THEN** кнопка «Обновить» отображается неактивной с подсказкой о причине (как у «Сохранить»)

### Requirement: Раздел «Добавить»

При выборе «Добавить» MUST отображаться страница с табами: «Вручную» и «Импорт из Telegram».

**Вкладка «Вручную»:** форма с переключателем режима сохранения (авто, как есть, полная статья, выжимка, закладка) как primary control для обработки тела, переключателем типа контента (авто, статья, ссылка, заметка) как secondary storage hint, большое текстовое поле (textarea) и кнопка «Добавить». При выборе «статья» или «ссылка» и режиме, отличном от «как есть», MUST отображаться подсказка «Вставьте URL в текст». При выборе «как есть» + «статья» MUST отображаться пояснение, что вставленный текст сохранится без скачивания URL. Форма предназначена для ввода текста нового знания и отправки в ingestion pipeline. При выборе типа, отличного от «авто», значение MUST передаваться в API как type_hint. При выборе режима, отличного от «авто», значение MUST передаваться в API как content_mode.

Во время обработки (loading) форма MUST блокировать повторную отправку: textarea и кнопка disabled, MUST отображаться анимация загрузки (спиннер или индикатор прогресса). Кнопка MUST показывать состояние обработки (напр. «Обработка...» или спиннер).

При успехе MUST отображаться блок (Alert/Banner) с сообщением «Добавлено» и ссылкой «Перейти к узлу» (если API вернул path). При ошибке MUST отображаться блок с текстом ошибки. Сообщение сбрасывается при следующей отправке.

**Вкладка «Импорт из Telegram»:** MUST отображаться возможность загрузить JSON-файл экспорта чата. После загрузки MUST отображаться индикатор прогресса (N/M, принято X, отклонено Y), карточка текущей записи (текст, source_author), переключатель режима сохранения (те же опции, что на вкладке «Вручную»), переключатель типа контента, кнопки «Отклонить» и «Принять». При «Принять» MUST вызываться API accept с выбранным type_hint и content_mode; при «Отклонить» — API reject. При наличии активной сессии (session_id в state или из предыдущей загрузки) MUST автоматически загружаться состояние сессии (GET session) и отображаться текущая запись. Должна быть возможность «Начать заново» для очистки сессии и загрузки нового файла.

**Завершение импорта:** когда обработана последняя запись сессии (ответ accept или reject без следующей записи, `next_item` отсутствует), UI MUST отображать промежуточное состояние завершения до возврата к выбору файла: явное сообщение, что импорт завершён; итоговые числа — всего в выгрузке, принято, отклонено; если последнее действие было «Принять», MUST отображаться ссылка на последний созданный узел; MUST быть действие для начала нового импорта (например кнопка «Импортировать другой файл»), по которому снова показывается форма загрузки JSON. Немедленный переход только к форме выбора файла без такого экрана MUST NOT выполняться (чтобы не создавать впечатление, что результаты импорта потеряны).

#### Сценарий: Добавление знания

- **WHEN** пользователь на вкладке «Вручную» вводит текст и нажимает «Добавить»
- **THEN** текст отправляется в API (ingestion), отображается результат

#### Сценарий: Добавление с выбором типа

- **WHEN** пользователь на вкладке «Вручную» выбирает тип «статья», вводит текст и нажимает «Добавить»
- **THEN** текст и type_hint=article отправляются в API

#### Сценарий: Пустой ввод

- **WHEN** пользователь нажимает «Добавить» без ввода текста
- **THEN** кнопка неактивна или показывается валидация

#### Сценарий: Блокировка при обработке

- **WHEN** запрос на добавление выполняется (loading)
- **THEN** textarea и кнопка disabled, отображается анимация загрузки, повторная отправка невозможна

#### Сценарий: Подсказка при выборе статьи или ссылки

- **WHEN** пользователь выбирает тип «статья» или «ссылка»
- **THEN** под переключателем отображается текст «Вставьте URL в текст»

#### Сценарий: Успешное добавление

- **WHEN** запрос завершился успешно
- **THEN** отображается блок с сообщением «Добавлено» и ссылкой «Перейти к узлу» (если API вернул path)

#### Сценарий: Ошибка при добавлении

- **WHEN** запрос завершился с ошибкой
- **THEN** отображается блок с текстом ошибки из API

#### Сценарий: Загрузка JSON для импорта

- **WHEN** пользователь на вкладке «Импорт из Telegram» выбирает JSON-файл и загружает
- **THEN** создаётся сессия, отображается первая запись и прогресс

#### Сценарий: Принять запись при импорте

- **WHEN** пользователь нажимает «Принять» для текущей записи импорта
- **THEN** запись передаётся в ingestion, отображается следующая запись

#### Сценарий: Отклонить запись при импорте

- **WHEN** пользователь нажимает «Отклонить»
- **THEN** запись пропускается, отображается следующая

#### Сценарий: Восстановление сессии импорта

- **WHEN** пользователь открывает вкладку «Импорт из Telegram» и есть активная сессия (session_id)
- **THEN** загружается состояние сессии, отображается текущая запись и прогресс

#### Сценарий: Завершение импорта (все записи обработаны)

- **WHEN** пользователь нажимает «Принять» или «Отклонить» для последней записи сессии (в ответе API нет следующей записи)
- **THEN** отображается экран завершения с сообщением о завершении импорта, итоговой статистикой (всего, принято, отклонено) и возможностью перейти к последнему добавленному узлу, если последнее действие было принятие; до загрузки нового файла пользователь не видит только пустую форму выбора JSON без этого экрана

### Requirement: Раздел «Обзор»

При выборе «Обзор» MUST отображаться страница обзора базы: слева — иерархическое дерево тем (фильтр по поддереву), справа — таблица узлов с колонками Название, Тип, Дата, Метки, Ссылка. Дерево поддерживает «Вся база»; таблица — фильтры по типам (article/link/note), поиск по тексту, фильтр по ручной проверке и личным меткам, сортировку, пагинацию. Детали панели фильтров (toolbar на десктопе, Sheet на мобильных) — в спецификации overview-browse. При переходе на страницу узла ссылка MUST передавать state с returnTo (текущий pathname + search, включая manual_processed и labels при наличии) для сохранения контекста обзора.

#### Сценарий: Навигация по дереву

- **WHEN** пользователь раскрывает/выбирает тему в дереве
- **THEN** таблица справа обновляется и показывает узлы выбранного поддерева (рекурсивно)

#### Сценарий: Отображение списка узлов

- **WHEN** выбрана тема, подтема или «Вся база»
- **THEN** в таблице отображаются узлы с title, type, path, датой, ссылкой (для link/article)

#### Сценарий: Переход на страницу узла с сохранением контекста

- **WHEN** пользователь кликает по узлу в таблице
- **THEN** переход на страницу узла с state.returnTo, содержащим URL обзора с текущими фильтрами (path, type, q, sort, order, page, manual_processed)

### Requirement: Сборка через Vite

Веб-приложение ДОЛЖНО (SHALL) собираться через Vite. Исходники в web/, результат сборки — web/dist.

#### Сценарий: Сборка

- **WHEN** выполняется npm run build в web/
- **THEN** создаётся web/dist с статическими файлами

### Requirement: Обращение к API

UI MUST обращаться к API по URL, задаваемому через переменную VITE_API_URL (по умолчанию localhost).

#### Сценарий: Запрос данных

- **WHEN** пользователь открывает страницу
- **THEN** UI запрашивает данные у API по сконфигурированному URL

### Requirement: Просмотр узла

UI ДОЛЖЕН (SHALL) позволять просматривать содержимое узла. Страница узла MUST содержать: кнопку «Назад»; breadcrumbs по path; панель действий (Action Bar) после breadcrumbs; заголовок из metadata.title; header с type badge, created, updated, keywords; строку атрибуции источника; блок аннотации и блок содержания с рендерингом markdown.

Панель действий (Action Bar) MUST объединять все управляющие действия: toggle «Проверено», кнопку «Перевести» (для article без перевода), разделитель, «Переместить», «Удалить». Кнопка «Удалить» MUST открывать модалку подтверждения (AlertDialog). Кнопка «Переместить» MUST открывать модалку перемещения с текстовым полем полного пути и деревом тем.

#### Сценарий: Открытие узла

- **WHEN** пользователь выбирает узел из таблицы
- **THEN** отображаются breadcrumbs, панель действий, заголовок, header с метаданными, аннотация и содержание

#### Сценарий: Панель действий для article

- **WHEN** открыт узел типа article без переводов
- **THEN** панель содержит: toggle «Проверено», «Перевести», разделитель, «Переместить», «Удалить»

#### Сценарий: Модалка подтверждения удаления

- **WHEN** пользователь нажимает «Удалить»
- **THEN** открывается AlertDialog с заголовком узла, путём, кнопками «Отмена» и «Удалить» (destructive)

#### Сценарий: Успешное удаление

- **WHEN** пользователь подтверждает удаление и API возвращает успех
- **THEN** переход на страницу обзора

#### Сценарий: Модалка перемещения с полным путём

- **WHEN** пользователь нажимает «Переместить»
- **THEN** открывается Dialog с текстовым полем «Новый путь» (предзаполнено текущим путём, редактируемое), деревом тем и кнопкой «Переместить»

#### Сценарий: Перемещение с созданием новых тем

- **WHEN** пользователь вводит путь с несуществующими темами и нажимает «Переместить»
- **THEN** API создаёт промежуточные директории, при успехе — переход на новую страницу узла

#### Сценарий: Выбор темы из дерева

- **WHEN** пользователь кликает тему в дереве
- **THEN** текстовое поле «Новый путь» обновляется: `{выбранная_тема}/{текущий_slug}`

### Requirement: Ручное редактирование заголовка и ключевых слов на странице узла

UI SHALL предоставлять на странице узла ручное редактирование `title` и `keywords` через модальные окна. Рядом с заголовком и рядом с блоком ключевых слов MUST отображаться иконка редактирования (карандаш). Нажатие по иконке MUST открывать соответствующую модалку и позволять сохранить изменения через `PATCH /api/nodes/{path...}` без перезагрузки страницы.

#### Сценарий: Редактирование заголовка

- **WHEN** пользователь нажимает карандаш у заголовка, вводит новое значение и нажимает «Сохранить»
- **THEN** UI отправляет PATCH с полем `title`, обновляет заголовок на текущей странице и закрывает модалку

#### Сценарий: Ошибка сохранения заголовка

- **WHEN** PATCH для заголовка возвращает ошибку
- **THEN** UI показывает диагностируемое сообщение и оставляет модалку открытой

### Requirement: UX редактирования тегов с подсказками

Модалка ключевых слов MUST использовать token-input UX: существующие keywords отображаются как chips с удалением, новые keywords можно добавлять через кнопку и с клавиатуры (`Enter` или `,`), удаление последнего тега через `Backspace` при пустом input. UI MUST показывать suggestions на основе уже существующих keywords в базе и позволять добавлять их кликом.

#### Сценарий: Добавление тега из suggestions

- **WHEN** пользователь открывает модалку ключевых слов и выбирает подсказку
- **THEN** выбранный keyword добавляется в список тегов, дубль не создаётся

#### Сценарий: Сохранение ключевых слов

- **WHEN** пользователь сохраняет модалку keywords
- **THEN** UI отправляет PATCH с массивом `keywords`, обновляет метаданные на странице узла и закрывает модалку

#### Сценарий: Пустой список ключевых слов

- **WHEN** у узла нет keywords
- **THEN** header страницы показывает состояние «без ключевых слов», но карандаш редактирования остаётся доступным

### Requirement: Ручное редактирование labels на странице узла

UI SHALL предоставлять на странице узла редактирование `labels` отдельно от `keywords`. Рядом с блоком меток MUST отображаться иконка редактирования. Нажатие MUST открывать модалку с token-input UX (аналогично keywords): chips с удалением, добавление через Enter/запятую, подсказки из GET /api/label-suggestions. Сохранение MUST отправлять PATCH с полем `labels` и обновлять страницу без перезагрузки.

#### Сценарий: Добавление метки

- **WHEN** пользователь добавляет метку «favorite» и сохраняет
- **THEN** UI отправляет PATCH с labels, отображает чип «favorite» с цветом по hash-палитре

#### Сценарий: Подсказка из существующих меток

- **WHEN** пользователь открывает модалку labels и выбирает подсказку из API
- **THEN** метка добавляется без дубликата (с учётом регистра на сервере)

#### Сценарий: Очистка всех меток

- **WHEN** пользователь удаляет все chips и сохраняет
- **THEN** UI отправляет PATCH с `labels: []`, блок меток показывает пустое состояние

#### Сценарий: Ошибка сохранения

- **WHEN** PATCH для labels возвращает ошибку валидации
- **THEN** UI показывает сообщение и оставляет модалку открытой

### Requirement: Стили чипов меток

Web UI MUST использовать общий модуль стилей меток (например `label-styles.ts`): фиксированная палитра, выбор класса по hash строки метки. Стили меток MUST NOT переиспользовать палитру типов контента (`type-styles`).

#### Сценарий: Стабильный цвет метки

- **WHEN** одна и та же метка отображается на странице узла и в таблице обзора
- **THEN** цвет чипа совпадает для этой строки метки

### Requirement: Различие keywords и labels в интерфейсе

Подписи и подсказки в UI MUST явно разделять «Ключевые слова» (семантика, поиск, RAG) и «Метки» (личная разметка, фильтр обзора).

#### Сценарий: Страница узла

- **WHEN** пользователь просматривает узел с keywords и labels
- **THEN** отображаются два отдельных блока с разными заголовками и визуальными стилями чипов

### Requirement: Унифицированный UI для job-операций

Web UI ДОЛЖЕН (SHALL) использовать единый клиент фоновых jobs для долгих операций узла через generic API:
- запуск `POST /api/jobs`
- поллинг статуса `GET /api/jobs/{id}`
- инкрементальные логи `GET /api/jobs/{id}/logs?after=...`

Для операций нормализации, выгрузки изображений, обновления описания из источника и async-перевода UI MUST использовать общий UX-паттерн: единая панель логов, отображение stage/status, финальный success/error.
Панель логов SHOULD поддерживать компактный и raw-режимы просмотра, автоскролл и завершение polling после terminal-статуса.

#### Сценарий: Единый запуск долгой операции

- **WHEN** пользователь запускает любую поддержанную долгую операцию из Action Bar
- **THEN** UI вызывает `POST /api/jobs`, получает `job_id` и отображает общий прогресс/логи в единой панели

#### Сценарий: Live-логи refresh-description

- **WHEN** пользователь нажимает «Обновить описание из источника»
- **THEN** UI показывает live-логи job с этапами и финальный статус аналогично нормализации

#### Сценарий: Ошибка job

- **WHEN** job завершается со статусом `error`
- **THEN** UI отображает сообщение об ошибке и не помечает операцию как успешную

### Requirement: Переходная совместимость frontend-клиента

На переходном этапе UI MAY поддерживать legacy-формы ответов async endpoint-ов как fallback, но основной контракт ДОЛЖЕН (SHALL) быть построен на `/api/jobs`.

#### Сценарий: Fallback на legacy-форму

- **WHEN** backend возвращает legacy-ответ для старого endpoint-а
- **THEN** UI корректно обрабатывает его без регрессии пользовательского сценария

### Requirement: Экран входа и проверка сессии

Web UI ДОЛЖЕН (SHALL) поддерживать страницу входа и проверку статуса сессии через `GET /api/auth/session` при запуске в режиме включённой авторизации. При отсутствии валидной сессии UI MUST направлять пользователя на страницу входа.

#### Сценарий: Пользователь без сессии открывает защищённый раздел

- **WHEN** пользователь открывает раздел, требующий авторизацию, и `GET /api/auth/session` возвращает неаутентифицированный статус
- **THEN** UI перенаправляет пользователя на страницу входа

#### Сценарий: Пользователь с активной сессией открывает приложение

- **WHEN** пользователь открывает приложение и `GET /api/auth/session` подтверждает валидную сессию
- **THEN** UI отображает запрошенный раздел без редиректа на login

### Requirement: Кнопка и переход к Yandex OAuth

Если `yandex` ∈ `auth_methods`, страница входа SHALL показывать «Войти через Yandex» с иконкой Yandex (inline SVG / локальный asset), переход на `GET /api/auth/yandex` без секретов на клиенте. Если `yandex` ∉ `auth_methods`, кнопка SHALL NOT отображаться.

#### Сценарий: Показ кнопки Yandex

- **WHEN** `auth_methods` содержит `yandex`
- **THEN** кнопка с иконкой и текстом отображается

#### Сценарий: Навигация на старт Yandex OAuth

- **WHEN** пользователь нажимает кнопку Yandex
- **THEN** полный переход на `GET /api/auth/yandex` (с сохранением redirect в sessionStorage, как у Google)

### Requirement: Иконки провайдеров на OAuth-кнопках

Кнопки «Войти через Google» и «Войти через Yandex» SHALL содержать узнаваемую иконку слева от текста. Ресурсы MUST быть локальными (inline SVG), без загрузки с внешних CDN.

#### Сценарий: Визуальное различие провайдеров

- **WHEN** отображаются обе OAuth-кнопки
- **THEN** у каждой своя иконка; подписи однозначно называют провайдера

### Requirement: Комбинированная страница входа

Web UI SHALL на `/login` отображать **все** настроенные способы: OAuth-кнопки для каждого элемента `{ google, yandex }` ∩ `auth_methods`; форму логин/пароль, если `password` ∈ `auth_methods`. При пароле и хотя бы одном OAuth SHOULD быть визуальный разделитель («или»). Порядок: OAuth-кнопки (Google, затем Yandex), разделитель, форма пароля.

#### Сценарий: Пароль и два OAuth

- **WHEN** `auth_methods` = `password`, `google`, `yandex`
- **THEN** UI показывает две OAuth-кнопки с иконками, разделитель, форму пароля

#### Сценарий: Только OAuth

- **WHEN** `auth_methods` без `password`
- **THEN** только OAuth-кнопки, форма скрыта

#### Сценарий: Только пароль

- **WHEN** `auth_methods` = `password`
- **THEN** только форма, OAuth-кнопки скрыты

### Requirement: Кнопка и переход к Google OAuth

Если `google` ∈ `auth_methods`, UI SHALL показывать «Войти через Google» с иконкой Google и переходом на `GET /api/auth/google`. Условие **не** «единственный auth_mode», а наличие `google` в `auth_methods`. При наличии пароля кнопка Google SHALL отображаться **вместе** с формой.

#### Сценарий: Показ кнопки Google

- **WHEN** `auth_methods` содержит `google`
- **THEN** кнопка с иконкой Google

#### Сценарий: Навигация на старт OAuth

- **WHEN** нажатие «Войти через Google»
- **THEN** переход на `GET /api/auth/google`

### Requirement: Поведение login/logout flow

Web UI MUST определять поведение по `auth_methods` (с fallback на `auth_mode` только если `auth_methods` отсутствует — legacy). Парольный вход — при `password` ∈ `auth_methods`. OAuth — сессия через callback; после возврата на `KB_PUBLIC_WEB_BASE_URL` — `GET /api/auth/session`, redirect из sessionStorage/query. Ошибки OAuth: учитывать query `provider` (`google`|`yandex`) для текста; без `provider` — нейтральное сообщение.

#### Сценарий: Успешный парольный вход

- **WHEN** `password` ∈ `auth_methods`, верные credentials
- **THEN** `POST /api/auth/login`, редирект на `redirect` или `/`

#### Сценарий: Ошибка парольного входа

- **WHEN** неверные credentials
- **THEN** сообщение об ошибке, остаёмся на login

#### Сценарий: Успешный возврат из Google

- **WHEN** `google` ∈ `auth_methods`, callback установил `kb_session`
- **THEN** session check, редирект на сохранённый путь

#### Сценарий: Успешный возврат из Yandex

- **WHEN** `yandex` ∈ `auth_methods`, callback установил `kb_session`
- **THEN** session check, редирект на сохранённый путь

#### Сценарий: Отказ allowlist или ошибка OAuth Google

- **WHEN** `error` в query, `provider=google` или не указан при Google-only инстансе
- **THEN** понятное сообщение; не аутентифицировать

#### Сценарий: Отказ allowlist или ошибка OAuth Yandex

- **WHEN** `error` в query, `provider=yandex`
- **THEN** сообщение с упоминанием Yandex при `oauth`/`forbidden`

#### Сценарий: Выход

- **WHEN** logout
- **THEN** `POST /api/auth/logout`, переход на login

### Requirement: Потребление auth_methods в AuthContext

`AuthContext` (и `api.ts`) SHALL экспортировать `authMethods: ('password'|'google'|'yandex')[]` из session API; компоненты login MUST использовать его вместо единственного `authMode === 'google'`.

#### Сценарий: Обновление после session

- **WHEN** `getSession()` вернул `auth_methods`
- **THEN** LoginPage и guards используют актуальный набор способов

### Requirement: Вкладка чатбота

UI ДОЛЖЕН (SHALL) отображать вкладку «Чат» в Navbar при `KB_EMBEDDING_ENABLED=true` и доступности RAG-индекса. Вкладка MUST содержать: ленту сообщений в чат-формате, поле ввода сообщения внизу, streaming-ответ ассистента, stop-кнопку во время генерации, список источников под ответом ассистента (path + title + type, кликабельные), и возможность раскрыть найденные фрагменты контекста при наличии. При `KB_EMBEDDING_ENABLED=false` вкладка MUST быть скрыта. UI MUST определять доступность чата через `GET /api/index/status` — при 503 вкладка скрывается.

#### Scenario: Отправка сообщения чатботу

- **WHEN** пользователь вводит сообщение и нажимает отправку
- **THEN** сообщение отображается как user bubble, отправляется `POST /api/chat` с `session_id` активной сессии, ответ отображается как assistant bubble streaming, источники отображаются под этим ответом

#### Scenario: Чатбот недоступен

- **WHEN** `GET /api/index/status` возвращает 503
- **THEN** вкладка «Чат» не отображается в Navbar

#### Scenario: Отображение источников

- **WHEN** чатбот возвращает ответ с sources
- **THEN** UI отображает кликабельные ссылки на ноды базы знаний с path, title, type и фрагментами, если они есть

#### Scenario: Сброс выбранных источников

- **WHEN** чат открыт из поиска с `source_paths`
- **THEN** UI показывает плашку выбранных источников с кнопкой-крестиком и подсказкой, позволяющей сбросить ограничение и искать по всей базе

#### Scenario: Пустой чат без лишней прокрутки

- **WHEN** вкладка чата открыта и сообщений ещё нет
- **THEN** UI показывает пустое состояние без внутренней полосы прокрутки

#### Scenario: Недостаточно данных в базе

- **WHEN** чат возвращает пустые sources или ответ о недостатке контекста
- **THEN** UI отображает это как нормальное состояние, а не как техническую ошибку

### Requirement: UI с историей чатов

Web-приложение SHALL показывать список существующих чатов и позволять пользователю переключаться между ними без потери состояния активного диалога.

#### Scenario: Переключение между чатами

- **WHEN** пользователь выбирает чат в боковой панели
- **THEN** приложение MUST загрузить и отобразить сообщения выбранного чата как активную сессию

### Requirement: Продолжение и создание чатов в одном потоке

Web-приложение MUST поддерживать создание нового чата и продолжение старого в едином UX-потоке, похожем на привычные LLM chat interfaces.

#### Scenario: Создание нового чата

- **WHEN** пользователь нажимает "Новый чат"
- **THEN** приложение SHALL создать новую сессию через API и перевести пользователя в неё

#### Scenario: Продолжение старого чата

- **WHEN** пользователь отправляет сообщение в ранее открытый чат
- **THEN** приложение MUST отправить сообщение в API выбранной сессии и обновить ленту без сброса контекста

### Requirement: Управление чатами в UI

Web-приложение SHALL поддерживать удаление и ручное переименование чатов из списка истории.

#### Scenario: Удаление чата пользователем

- **WHEN** пользователь подтверждает удаление чата
- **THEN** приложение MUST удалить чат через API и убрать его из списка истории

#### Scenario: Ручное переименование чата пользователем

- **WHEN** пользователь задаёт новое название чата
- **THEN** приложение MUST отправить обновление title через API и отобразить новое имя в списке

### Requirement: Скрытие служебных summary

Web-приложение MUST не показывать служебные summary-сообщения в ленте диалога первой версии.

#### Scenario: Отрисовка истории сообщений

- **WHEN** приложение получает данные чата для отображения
- **THEN** оно SHALL отображать только пользовательские и ассистентские сообщения

### Requirement: Вкладка гибридного поиска

UI ДОЛЖЕН (SHALL) отображать вкладку «Поиск» в Navbar при доступности гибридного индекса. Вкладка MUST содержать поле ввода запроса, фильтры по типу ноды, теме/поддереву и ручной проверке, а также список результатов в виде карточек. Каждая карточка MUST показывать title, type, annotation, path, keywords, score/rank, причины совпадения, source kinds и релевантные фрагменты, если они есть. При недоступном индексе вкладка MUST быть скрыта или показывать понятное состояние недоступности.

#### Scenario: Поиск по базе

- **WHEN** пользователь вводит запрос на вкладке «Поиск»
- **THEN** UI отправляет `POST /api/search` и отображает карточки результатов

#### Scenario: Карточка article с фрагментом

- **WHEN** API возвращает result с fragments
- **THEN** карточка показывает heading/snippet релевантного фрагмента

#### Scenario: Диагностика поиска

- **WHEN** API возвращает search meta и ranked results
- **THEN** UI показывает раскрываемый блок с исходным запросом, query rewrite, keyword index и total results

#### Scenario: Диагностика карточки результата

- **WHEN** карточка результата отображается пользователю
- **THEN** UI показывает score/rank, match reasons, source kinds и score фрагментов

#### Scenario: Сворачивание хвоста после перепада score

- **WHEN** между соседними результатами найден заметный перепад score
- **THEN** UI показывает оставшиеся результаты в компактном раскрываемом виде

#### Scenario: Нет результатов

- **WHEN** API возвращает пустой список results
- **THEN** UI отображает пустое состояние с текстом, что ничего не найдено

#### Scenario: Индекс недоступен

- **WHEN** `GET /api/index/status` возвращает 503
- **THEN** вкладка «Поиск» не отображается в Navbar или показывает состояние недоступности без падения приложения

### Requirement: Переход из поиска в чат

UI ДОЛЖЕН (SHALL) позволять пользователю перейти из результатов поиска в чат с сохранением запроса и выбранных источников. Карточка результата или группа результатов MUST иметь действие “Спросить по этим источникам” или эквивалентное действие, которое открывает вкладку «Чат» с initial message/source hints.

#### Scenario: Спросить по найденным источникам

- **WHEN** пользователь нажимает действие “Спросить по этим источникам” на странице поиска
- **THEN** UI открывает чат с предзаполненным вопросом и передаёт выбранные source_paths в `POST /api/chat` вместе с `session_id` активной сессии

#### Scenario: Открыть найденную ноду

- **WHEN** пользователь нажимает на title карточки результата
- **THEN** UI открывает страницу соответствующей ноды

### Requirement: Content mode selector on Add page

The manual Add (ingest) form SHALL provide a content mode control with options: Auto, Verbatim, Full article fetch, Digest, Link bookmark. Default selection is Auto. Labels and short hints SHALL explain when to use each mode (Russian UI). The content mode control is the primary control for body handling. The existing `type_hint` control is a secondary storage-form hint.

#### Scenario: User selects Verbatim before submit

- **WHEN** user chooses «Как есть» and submits URL with pasted body
- **THEN** request includes `content_mode: "verbatim"`
- **AND** result points to a node whose body is preserved from paste

#### Scenario: Auto mode unchanged UX

- **WHEN** user leaves mode on Auto
- **THEN** request omits `content_mode` or sends `auto` per API contract
- **AND** behavior matches pre-change auto rules after backend implementation

#### Scenario: User selects Verbatim with Article type

- **WHEN** user chooses «Как есть» and secondary type hint «Статья»
- **THEN** UI copy explains that pasted body will be saved as an article without fetching over it
- **AND** request includes `content_mode: "verbatim"` and `type_hint: "article"`

### Requirement: Content mode selector on Telegram import tab

The Telegram import accept UI on the Add page SHALL provide the same content mode options as the manual Add form. Default selection is Auto. Accepting an item SHALL send selected `content_mode` to the import accept API.

#### Scenario: Import item accepted as bookmark

- **WHEN** user chooses «Закладка» for the current Telegram import item and accepts it
- **THEN** request includes `content_mode: "link_bookmark"`
- **AND** result shows the saved node path and keeps processing the next import item
