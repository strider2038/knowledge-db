## Purpose

Веб-интерфейс для просмотра, поиска, создания и редактирования записей базы знаний. React + Vite, статика встраивается в бинарник kb-server.

## Requirements

### Requirement: Navbar с меню

UI ДОЛЖЕН (SHALL) содержать верхнюю панель навигации (Navbar) с пунктами меню: «Добавить», «Обзор».

#### Сценарий: Переключение между разделами

- **WHEN** пользователь выбирает пункт «Добавить» или «Обзор»
- **THEN** отображается соответствующий раздел

### Requirement: Раздел «Добавить»

При выборе «Добавить» MUST отображаться форма: переключатель типа контента (авто, статья, ссылка, заметка), большое текстовое поле (textarea) и кнопка «Добавить». При выборе «статья» или «ссылка» MUST отображаться подсказка «Вставьте URL в текст». Форма предназначена для ввода текста нового знания и отправки в ingestion pipeline. При выборе типа, отличного от «авто», значение MUST передаваться в API как type_hint.

Во время обработки (loading) форма MUST блокировать повторную отправку: textarea и кнопка disabled, MUST отображаться анимация загрузки (спиннер или индикатор прогресса). Кнопка MUST показывать состояние обработки (напр. «Обработка...» или спиннер).

При успехе MUST отображаться блок (Alert/Banner) с сообщением «Добавлено» и ссылкой «Перейти к узлу» (если API вернул path). При ошибке MUST отображаться блок с текстом ошибки. Сообщение сбрасывается при следующей отправке.

#### Сценарий: Добавление знания

- **WHEN** пользователь вводит текст и нажимает «Добавить»
- **THEN** текст отправляется в API (ingestion), отображается результат

#### Сценарий: Добавление с выбором типа

- **WHEN** пользователь выбирает тип «статья», вводит текст и нажимает «Добавить»
- **THEN** текст и type_hint=article отправляются в API

#### Сценарий: Пустой ввод

- **WHEN** пользователь нажимает «Добавить» без ввода текста
- **THEN** кнопка неактивна или показывается валидация

#### Сценарий: Блокировка при обработке

- **WHEN** запрос на добавление выполняется (loading)
- **THEN** textarea и кнопка disabled, отображается анимация загрузки, повторная отправка невозможна

#### Сценарий: Подсказка при выборе статьи или ссылки

- **WHEN** пользователь выбирает тип «статья» или «ссылка»
- **THEN** под переключателем отображается текст «Вставьте URL в текст»

#### Сценарий: Успешное добавление

- **WHEN** запрос завершился успешно
- **THEN** отображается блок с сообщением «Добавлено» и ссылкой «Перейти к узлу» (если API вернул path)

#### Сценарий: Ошибка при добавлении

- **WHEN** запрос завершился с ошибкой
- **THEN** отображается блок с текстом ошибки из API

### Requirement: Раздел «Обзор»

При выборе «Обзор» MUST отображаться страница обзора базы: слева — иерархическое дерево тем (фильтр по поддереву), справа — таблица узлов с колонками Название, Тип, Дата, Ссылка. Дерево поддерживает «Вся база»; таблица — фильтры по типам (article/link/note), поиск по тексту, сортировку, пагинацию. Детали — в спецификации overview-browse. При переходе на страницу узла ссылка MUST передавать state с returnTo (текущий pathname + search) для сохранения контекста обзора.

#### Сценарий: Навигация по дереву

- **WHEN** пользователь раскрывает/выбирает тему в дереве
- **THEN** таблица справа обновляется и показывает узлы выбранного поддерева (рекурсивно)

#### Сценарий: Отображение списка узлов

- **WHEN** выбрана тема, подтема или «Вся база»
- **THEN** в таблице отображаются узлы с title, type, path, датой, ссылкой (для link/article)

#### Сценарий: Переход на страницу узла с сохранением контекста

- **WHEN** пользователь кликает по узлу в таблице
- **THEN** переход на страницу узла с state.returnTo, содержащим URL обзора с текущими фильтрами (path, type, q, sort, order, page)

### Requirement: Сборка через Vite

Веб-приложение ДОЛЖНО (SHALL) собираться через Vite. Исходники в web/, результат сборки — web/dist.

#### Сценарий: Сборка

- **WHEN** выполняется npm run build в web/
- **THEN** создаётся web/dist с статическими файлами

### Requirement: Обращение к API

UI MUST обращаться к API по URL, задаваемому через переменную VITE_API_URL (по умолчанию localhost).

#### Сценарий: Запрос данных

- **WHEN** пользователь открывает страницу
- **THEN** UI запрашивает данные у API по сконфигурированному URL

### Requirement: Просмотр узла

UI ДОЛЖЕН (SHALL) позволять просматривать содержимое узла. Страница узла MUST содержать: кнопку «Назад» (возврат на обзор с сохранением фильтров при переходе из обзора, иначе на главную); breadcrumbs по path (сегменты — ссылки на обзор с path-фильтром); заголовок из metadata.title с fallback на slug из path; header с type badge, created, updated, keywords (чипсы); при наличии source_url, source_author или source_date — отдельную строку атрибуции источника (ссылка на source_url, автор, дата источника) для всех типов узлов (article, link, note); отдельный блок аннотации и блок содержания с рендерингом markdown (таблицы, подсветка кода, диаграммы Mermaid). Метаданные НЕ отображаются отдельным блоком. При наличии переводов MUST отображаться выбор языка (оригинал / переводы).

**Особенности типа link:** Узлы типа link (ссылка-закладка) НЕ имеют тела контента — блок «Содержание» MUST НЕ отображаться. Перед блоком аннотации MUST отображаться явная кликабельная ссылка на source_url (если задана): карточка/кнопка с иконкой и URL, target="_blank".

#### Сценарий: Открытие узла

- **WHEN** пользователь выбирает узел из таблицы
- **THEN** отображаются breadcrumbs, заголовок (title), header с метаданными, аннотация и содержание с рендерингом markdown

#### Сценарий: Выбор языка

- **WHEN** у узла есть переводы (translations)
- **THEN** отображается переключатель для выбора оригинала или перевода

#### Сценарий: Кнопка «Назад» при переходе из обзора

- **WHEN** пользователь перешёл на страницу узла из обзора и нажимает «Назад»
- **THEN** возврат на страницу обзора с теми же query-параметрами (path, type, q, sort, order, page)

#### Сценарий: Кнопка «Назад» при прямом заходе

- **WHEN** пользователь открыл страницу узла напрямую (закладка, ссылка) и нажимает «Назад»
- **THEN** переход на главную (/)

#### Сценарий: Breadcrumbs

- **WHEN** отображается страница узла с path «programming/scaling/load-balancing»
- **THEN** отображаются breadcrumbs: Обзор > programming > scaling > load-balancing, где каждый сегмент — ссылка на обзор с ?path=накопленный_путь

#### Сценарий: Markdown в аннотации и содержании

- **WHEN** аннотация или содержание содержат markdown (заголовки, списки, таблицы, код)
- **THEN** контент рендерится как markdown с поддержкой таблиц (GFM), подсветки синтаксиса кода и диаграмм Mermaid (блоки \`\`\`mermaid)

#### Сценарий: Mermaid-диаграммы в содержании

- **WHEN** аннотация или содержание содержат блок кода с языком mermaid (fenced code block \`\`\`mermaid)
- **THEN** блок рендерится как интерактивная диаграмма (flowchart, sequenceDiagram и др.)

#### Сценарий: Узел типа link

- **WHEN** пользователь открывает узел типа link с source_url
- **THEN** перед аннотацией отображается явная кликабельная ссылка на source_url; блок «Содержание» НЕ отображается

#### Сценарий: Атрибуция источника

- **WHEN** у узла заданы source_url, source_author и/или source_date (любой тип: article, link, note)
- **THEN** под строкой с type badge и датами отображается отдельная строка: кликабельная ссылка на source_url (если есть), автор (если есть), дата источника (если есть)
