## Purpose

Консольная утилита `kb` для валидации структуры базы, инициализации новой базы (gitignore, установка skills) и офлайн-операций с embedding-index.
## Requirements
### Requirement: Подкоманда validate

CLI ДОЛЖЕН (SHALL) предоставлять подкоманду `validate` для проверки структуры базы.

#### Сценарий: Валидация указанного пути

- **WHEN** `kb validate --path /path/to/base`
- **THEN** проверяется структура и выводится отчёт об ошибках или успехе

#### Сценарий: Валидация текущей директории

- **WHEN** `kb validate` (без `--path`) в директории базы
- **THEN** проверяется текущая директория

### Requirement: Подкоманда init

CLI ДОЛЖЕН (SHALL) предоставлять подкоманду `init` для инициализации новой базы знаний.

#### Сценарий: Init в указанной директории

- **WHEN** `kb init --path /path/to/base`
- **THEN** создаётся `.gitignore` с правилами `**/.local/`, `**/.local/**`, копируются agent skills в `~/.cursor/skills/` с подстановкой пути

#### Сценарий: Init в текущей директории

- **WHEN** `kb init` (без `--path`)
- **THEN** инициализация выполняется в текущей директории

### Requirement: Идемпотентность init

`init` MUST быть идемпотентным: повторный вызов не должен ломать существующую конфигурацию.

#### Сценарий: Повторный init

- **WHEN** `kb init` вызывается повторно в той же директории
- **THEN** `.gitignore` и skills обновляются, без потери данных

### Requirement: Валидация файлов переводов

`kb validate` ДОЛЖЕН (SHALL) проверять файлы переводов (`*.ru.md`). Файл считается переводом, если в той же директории существует `{slug}.md`, где `{slug}` — часть имени до `.ru.md`. Для каждого файла перевода система MUST проверять: frontmatter содержит обязательные поля `translation_of` и `lang`; `translation_of` совпадает со slug оригинала; все wikilinks в теле (`[[target]]`, `[[target|label]]`) ведут на существующие узлы базы. Если у узла есть файл перевода, в frontmatter оригинала MUST быть поле `translations`, содержащее slug перевода. Нарушения выводятся в общий отчёт `validate`.

#### Сценарий: Валидный файл перевода

- **WHEN** файл `slug.ru.md` имеет `translation_of: slug`, `lang: ru`, оригинал `slug.md` существует, wikilinks ведут на существующие узлы, в оригинале есть `translations: [slug.ru]`
- **THEN** валидация проходит без ошибок

#### Сценарий: Отсутствует translation_of

- **WHEN** в frontmatter перевода отсутствует поле `translation_of`
- **THEN** `validate` сообщает об ошибке

#### Сценарий: Wikilink на несуществующий узел

- **WHEN** в теле перевода есть `[[missing-node]]` и узла `missing-node` нет в базе
- **THEN** `validate` сообщает об ошибке

#### Сценарий: Оригинал без translations

- **WHEN** существует `slug.ru.md`, но в frontmatter `slug.md` отсутствует поле `translations`
- **THEN** `validate` сообщает об ошибке

### Requirement: Подкоманда dump-images

CLI ДОЛЖЕН (SHALL) предоставлять подкоманду `dump-images` для скачивания удалённых изображений из markdown-статьи и замены ссылок на локальные пути. Подробная спецификация — в `kb-cli-dump-images`.

#### Сценарий: Наличие подкоманды

- **WHEN** пользователь вызывает `kb dump-images --help`
- **THEN** отображается справка по флагам `--path`, `--file`, `--dry-run`

### Requirement: Подкоманда expand-urls

CLI ДОЛЖЕН (SHALL) предоставлять подкоманду `expand-urls` для раскрытия редиректных ссылок и удаления UTM/трекинговых параметров в markdown-файлах базы. Подробная спецификация — в `kb-cli-expand-urls`.

#### Сценарий: Наличие подкоманды

- **WHEN** пользователь вызывает `kb expand-urls --help`
- **THEN** отображается справка по флагам `--path`, `--file`, `--dry-run`

### Requirement: Подкоманда migrate-node-ids

CLI MUST предоставлять подкоманду `migrate-node-ids` для одноразового присвоения поля `id` (UUID v7) всем узлам и файлам переводов в указанной базе (`--path` или `KB_DATA_PATH`), у которых поле `id` отсутствует. Подкоманда MUST поддерживать флаг `--dry-run` (только отчёт без записи). Подкоманда MUST выводить количество обработанных файлов и список конфликтов (дубликаты id или source_url если проверяются).

#### Сценарий: Dry-run

- **WHEN** выполняется `kb migrate-node-ids --path /data/kb --dry-run`
- **THEN** файлы не изменяются, в stdout перечислены пути файлов без id

#### Сценарий: Применение миграции

- **WHEN** выполняется `kb migrate-node-ids --path /data/kb`
- **THEN** каждый узел без id получает уникальный UUID v7 в frontmatter, updated сохраняется или обновляется по политике Store

#### Сценарий: Повторный запуск идемпотентен

- **WHEN** `migrate-node-ids` выполняется повторно на базе, где у всех узлов уже есть id
- **THEN** ни один файл не изменяется, отчёт показывает 0 изменений

### Requirement: Подкоманда rebuild-index

CLI ДОЛЖЕН (SHALL) предоставлять подкоманду `rebuild-index` для полной перестройки embedding-index (`.kb/index.db`) без запущенного `kb serve`. Подкоманда MUST использовать ту же логику, что `SyncWorker.ManualRebuild`: очистка индекса и полная переиндексация всех нод из FS. Подкоманда MUST читать конфигурацию из `.env` и переменных окружения (как `kb serve`). Подкоманда MUST принимать `--path` (или `KB_DATA_PATH`, иначе `./data`) — корень базы знаний. Подкоманда MUST требовать `KB_EMBEDDING_ENABLED=true` с заданными `KB_EMBEDDING_API_URL` и `KB_EMBEDDING_API_KEY`. Подкоманда MUST выполняться синхронно до завершения переиндексации и выводить итоговые метрики (total_nodes, total_chunks, embedding_model, keyword_index, status).

#### Сценарий: Успешная перестройка индекса

- **WHEN** выполняется `kb rebuild-index --path /data/kb` при настроенных embedding-переменных
- **THEN** индекс очищается, все ноды с валидным `id` переиндексируются, команда завершается с кодом 0 и печатает метрики индекса

#### Сценарий: Embedding не включён

- **WHEN** выполняется `kb rebuild-index` при `KB_EMBEDDING_ENABLED=false`
- **THEN** команда завершается с ошибкой и сообщением о необходимости включить embedding API

#### Сценарий: Наличие подкоманды в help

- **WHEN** пользователь вызывает `kb rebuild-index --help`
- **THEN** отображается справка с флагом `--path` / `-p`
